**Parent Outcome:** [RHAISTRAT-1345](https://redhat.atlassian.net/browse/RHAISTRAT-1345) (Agent Memory)

## Summary

Regulated customers cannot adopt agent memory until every memory record is isolated to its intended scope, screened for sensitive data at write time, and attributable to who wrote it and when. This RFE delivers that minimum governance bar alongside the first memory Dev Preview, so the very first thing customers touch is governable, not a compliance liability to retrofit.

## Problem Statement

Nothing in current agent-memory approaches (file-based memories, open-source memory frameworks) prevents PII, credentials, or classified data from entering shared memory. Memories written by one team's agents can be visible to another team's. There is no record of who wrote what, when, into which scope. From the user's perspective:

- A compliance officer cannot answer "what did this agent memorize about this customer last week?"
- A platform admin cannot promise that team A's agents will never surface team B's data.
- An AI engineer has no guardrail when an agent tries to memorize a credit-card number or PHI from a conversation.

For healthcare (PHI/HIPAA), financial services, and federal customers this is disqualifying today, before any memory feature can even be piloted.

## Affected Customers

- **SAS**: blocked from moving agents to production without a full auditable trace of every data access.
- **NTT Data**: Zero Trust requirement that agents follow the same data-access policies as employees.
- **OCBC Bank**: multi-tenant requirement that agents from different teams must not see each other's data.
- **BNP Paribas**: strict compliance audit requirements in disconnected deployments.
- **ITZBund** (German federal IT): data sovereignty and auditability; all AI activity must stay inside the governance perimeter.
- Healthcare accounts raised by field engineering: PHI exposure through shared agent memory is the named blocker to multi-agent healthcare workflows.

## Business Justification

- Six enterprise verticals surveyed (healthcare, financial services, ITOps, legal, manufacturing, defense) independently converge on the same requirements: scope isolation, write-path governance, and auditability. The mechanism is common; only the policy differs. This is the enterprise linchpin of the memory investment.
- EU AI Act obligations for general-purpose AI are enforced from August 2026; regardless of the pending high-risk timeline revision, the named accounts above impose the requirement contractually today. These are blocked-deal governance gaps, not future risks.
- Governance depth is Red Hat's differentiation in the memory market: hyperscaler and startup memory offerings are exactly weakest here.

## User Scenarios

1. As a compliance officer, I review which memories an agent wrote last week, on whose behalf, into which scope, and export that record for an audit.
2. As a platform admin at a bank, I verify that agents in one project cannot read memories written in another project on the same cluster.
3. As an AI engineer, when my agent attempts to memorize sensitive data (a card number, a patient identifier), the write is intercepted according to policy and I can see why.

## Acceptance Criteria

- [ ] Every memory record carries a scope, and reads return only records the caller's scope permits.
- [ ] Teams/projects on a shared cluster cannot read each other's memories; isolation is demonstrable to a customer security reviewer.
- [ ] Memory writes are screened for PII, secrets, and other sensitive-data patterns, with policy determining whether a flagged write is blocked, quarantined, or logged.
- [ ] Administrators can review and remove flagged, quarantined, or otherwise sensitive memories.
- [ ] Every memory write is recorded (who, what, when, which scope) in a log administrators can review.
- [ ] The write log can be exported by authorized users for compliance review.

## Success Criteria

- A regulated pilot customer's internal security review passes with memory enabled.
- Zero cross-scope reads under isolation testing.
- 100% of memory writes appear in the write log.

## Scope

### In Scope
- Record-level scope isolation and enforcement on reads.
- Sensitive-data screening on the write path with configurable policy outcome.
- A basic, reviewable, exportable write-event log.
- Admin review/removal of flagged memories.

### Out of Scope
- The full append-only audit trail of reads and writes with erasure primitives (GA-gating capability, separate later RFE).
- Contradiction detection and provenance metadata (Tech Preview phase).
- Adversarial memory-injection defense (separate later RFE; screening here targets sensitive-data leakage, not attack patterns).
- The full multi-tier scope hierarchy (user/project/role/org); this RFE requires record-level scoping and project isolation, with the richer tier model following at Tech Preview.

## Open Questions

- Default screening policies per vertical (block vs quarantine) and who ships them.
- The internal governance prototype under evaluation is contingent on a written IP/license transfer before any productization decision.
