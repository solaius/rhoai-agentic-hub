- **Closes:** enhancements backlog #2 (fully). Also delivers the Jira

## Problem

Jira is the delivery source of truth but has no intake pipeline into the
hub. `views/jira-map.md` exists but is derived only from hand-filed `ref-`
entries (one row today). `hub.intake` files Jira/RFE links URL-only,
explicitly deferring field ingestion to this backlog item; `hub.research`
reserves a `jira-gap` lens gated on it. pm-toolkit has a proven async Jira
client, but that repo is read-only source material — daily work happens
here.

## Decisions made during brainstorming

1. **The client lives in this repo.** pm-toolkit's
   `scripts/clients/jira.py` is ported near-verbatim into
   `scripts/hublib/jira.py` — no sibling-repo import. `httpx>=0.28` joins
   `scripts/requirements.txt` (a deliberate dependency-posture change;
   doctor section 1 learns to check it).
2. **Client vs MCP:** scripts cannot call MCP tools, so everything
   deterministic (sweep, sync diff, snapshot, view, doctor probe) is
   scripted through the client. The Atlassian MCP stays the tool for
   ad-hoc conversational Jira work; the two coexist.
3. **Read-only skills, full client.** The port keeps the write methods
   (comment/transition/update/create — future #30 and write-back items),
   but `hub.jira-sweep` / `hub.jira-sync` never write to Jira.
4. **Two-tier sweep output:** gated `ref-` entries only for strategic
   issues (`ref_types`, default `[Outcome, Feature]`); the full result set
   goes into a per-feature machine-written snapshot.
5. **Snapshot is tracked and public, curated fields only** — `key, type,
   status, resolution, fix_versions, updated, summary`. Summaries are
   admitted by the **unauthenticated-probe rule**: the sweep re-fetches
   each issue with no token; HTTP 200 (world-readable in Jira itself) →
   summary included, anything else → `summary: null`. Fail-closed in the
   disclosure direction; no model judgment involved.
6. **JQL is the single stored scope.** Feature↔Jira mapping is messy
   (e.g., a feature may map to a shared component plus labels), so no
   components/labels shorthand schema — one `jql` string per feature in
   `features/features.yaml`, derived conversationally at first sweep.
7. **jtbd backlinks:** jtbd entries get an optional `jira: [KEY, …]` list
   (the indexer already renders a `jira` field on jtbd rows); sync watches
   those keys.

## Architecture

New files except two surgical touches (`indexer.py`, `schema.py`) and one
optional yaml key — deliberately conflict-free with the concurrent
enhancement-batch work:

| path | holds |
|---|---|
| `scripts/hublib/jira.py` | ported async `JiraClient` (httpx; Cloud basic auth via `JIRA_SERVER`/`JIRA_USER`/`JIRA_TOKEN` + `client_from_env()`; 429 retry, token pagination, ADF→text; write methods ported but unused) |
| `scripts/hublib/jiramap.py` | snapshot build/load/validate + sync diff logic |
| `scripts/hub_jira.py` | thin CLI: `--check` (connectivity probe) · `--sweep <feature> [--jql …]` · `--sync [<feature>]` |
| `.claude/skills/hub.jira-sweep/SKILL.md` | the sweep procedure (gated) |
| `.claude/skills/hub.jira-sync/SKILL.md` | the refresh procedure (gated) |
| `features/<id>/work/jira-snapshot.yaml` | per-feature machine-written snapshot (tracked) |

### Scope config (`features/features.yaml`)

```yaml
- id: mcp-registry
  title: MCP Registry
  description: …
  jira:
    jql: 'project = RHOAIENG AND component = "AI Hub" AND labels in (mcp, mcp-registry)'
    ref_types: [Outcome, Feature]   # strategic tier → ref- proposals (default)
```

`schema.py` only reads `id` from feature entries today; the `jira:` block
is optional and tolerated. Stored scope is what makes sync repeatable
without re-asking.

### Snapshot format

```yaml
# generated by hub.jira-sweep (scripts/hub_jira.py) — do not hand-edit
feature: mcp-registry
jql: '…'
swept: 2026-07-09
issues:
  - key: RHOAIENG-1234
    type: Feature
    status: In Progress
    resolution: null
    fix_versions: [RHOAI 3.5]
    updated: 2026-07-08
    summary: "…"   # null when the unauthenticated probe could not read it
```

Sorted by key, fixed field order — byte-stable for identical input, so
diffs are reviewable. `jiramap.validate()` (called from `lint_repo`)
checks the marker comment is present and the shape is well-formed;
hand-editing is prohibited by convention like other generated files.

### View

`views/jira-map.md` becomes two-part, generated from tracked files only
(deterministic; CI `--check` needs no credentials):

- per-feature sections from snapshots — `key · type · status ·
  fix_versions · summary-or-redacted`, linked to the ref- entry when one
  exists;
- the existing `resource:`-derived rows for refs not covered by any
  snapshot (today's behavior preserved — RHAISTRAT-1345 keeps working).

### Data flow

`features.yaml jira:` scope → CLI fetch (+probe) → CLI emits final
snapshot YAML + strategic-tier ref- candidates → skill gates (batch
table, per-line approve/edit/reject) → agent writes approved files →
`hub_index.py` regenerates the view → `hub_lint.py` (disclosure lint sees
the snapshot too) → commit.

## Skill flows

### hub.jira-sweep `<feature>`

1. **Scope discovery** (first run / no `jira:` block): user gives hints
   (project, component, label guesses); the skill runs candidate JQLs via
   the CLI, shows result counts + sample rows, iterates until the scope
   looks right; the approved JQL becomes a gated `features.yaml` edit.
2. **Fetch:** stored JQL → snapshot build (probe rule applied) +
   `ref_types` issues picked as strategic-tier candidates.
3. **Scope sanity:** result count wildly off vs. the last snapshot (0
   rows, big swings) → flagged, never silently written — labels drift.
4. **Gate** (one batch table): the snapshot file (redacted-summary list
   called out), one line per proposed strategic `ref-` (description
   drafted from the issue), the `features.yaml` edit if scope changed.
5. **On OK:** write files → `python scripts/hub_index.py` →
   `python scripts/hub_lint.py` (0 errors) → commit.

### hub.jira-sync `[<feature>]` (default: all features with a `jira:` block)

1. CLI re-runs each stored JQL **plus** every Jira key referenced by
   tracked entries (`ref-` `resource:` URLs, jtbd `jira:` lists) even when
   outside the JQL — refs never silently rot.
2. Emits a pure diff report: status changes, resolutions (done /
   won't-fix), fixVersion moves, new issues entering scope, tracked issues
   that vanished.
3. Gate proposes the consequences: snapshot refresh, per-ref updates where
   the diff matters (closed Outcome → status note / `review_after` bump),
   jtbd status nudges (all implementing Jiras resolved → propose
   `delivered`), new strategic-tier arrivals as ref- candidates.
4. Nothing auto-applies; empty diff = "nothing changed", zero ceremony.

Two features' JQLs may catch the same issue; it then appears in both
snapshots — consistent with the `features:` cross-ref philosophy.

## Error handling

- 401/403 → CLI prints remediation (where to regenerate the token, which
  env var), never the token value; skills point at `hub.doctor`.
- 429/5xx → client retry with backoff; still failing → sweep aborts
  **before** the gate. A partial snapshot is never proposed —
  all-or-nothing per feature.
- Probe failures (network blip vs 401) → treated as "not public": false
  negatives redact a summary; false positives impossible.
- Sync with no snapshot yet → "run hub.jira-sweep first", not a stack
  trace.
- MCP has no role in the pipeline, so MCP availability never blocks it.

## Testing

All offline, CI-safe:

- `test_jira.py` — client against `httpx.MockTransport`: auth header
  shape, pagination, 429 retry, `adf_to_text`, env mapping.
- `test_jiramap.py` — snapshot build byte-stable, diff logic (changed /
  new / vanished), validator (marker present, malformed rows).
- `test_indexer.py` — jira-map view: snapshot + ref merge, redacted rows,
  no-snapshot fallback.

Live acceptance (owner-in-the-loop, not subagent tasks): first sweep on a
real feature including scope discovery; a sync run days later; the doctor
probe on this machine.
