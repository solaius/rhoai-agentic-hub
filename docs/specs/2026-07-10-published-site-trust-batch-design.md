# Published-site trust batch (#34, #8, #4) - design spec

- **Date:** 2026-07-10 · **Owner:** Peter Double · **Status:** approved design,
  pre-implementation
- **Closes:** enhancements backlog #34 (disclosure heuristic frontmatter gap),
  #8 (pages-site landing page upgrade), #4 (`hub.refresh-site`).
- **Theme:** all three items harden or upgrade the published/public surface:
  the disclosure net that guards it, the front door people see, and the
  update path that keeps the two migrated hub sites from rotting.

## Problem

Three related gaps in the public surface:

1. **#34:** the CI-visible generic disclosure heuristic (`RESTRICTED_HINTS`)
   scans knowledge-entry **bodies only** (`lint_entry`, `schema.py`);
   frontmatter (`description:` etc.) is never scanned, and the generated
   `views/*.md` plus index files that propagate entry descriptions are not
   scanned by either net. The local restricted-patterns pass covers
   frontmatter (it reads raw lines) but not views.
2. **#8:** the pages-site landing page is a flat `<ul>` generated by an
   f-string in `publisher.py`. The manifest already carries title +
   description per artifact; nothing groups by area or signals freshness.
   This is the highest-visibility page the hub produces.
3. **#4:** the RHCL hub (23 pages, `features/mcp-gateway/enablement/rhcl-hub/`,
   all pages `data-verified="2026-06-21"`) and the Management hub (22 pages,
   `features/mcp-ecosystem/enablement/management-hub/`, `2026-06-22`) are the
   live copies and have **no update path**: the old repo's `update-*-hub`
   skills target retired copies. Critically, those old skills had **no
   disclosure rules at all**; they actively swept real customer names in.
   The anonymization now live on the pages was applied once at migration
   (owner rulings 1b/2b) and no skill enforces it going forward.

## Decisions made during brainstorming

1. **Refresh source config is fully tracked** (owner call): per-site YAML at
   `features/<feature>/work/refresh-<slug>.yaml`. Exposes opaque GDoc IDs and
   internal Slack channel names in a world-readable repo; accepted. The new
   file joins the restricted-patterns scan surface as mitigation.
2. **Landing page gets the full-branded front-door treatment** (owner call):
   hero header, Red Hat-style design tokens echoing the hub sites'
   `styles.css`, grouped cards. Still one self-contained generated HTML file:
   inline CSS, no external assets, font stack
   `"Red Hat Display", "Red Hat Text", system-ui`.
3. **Acceptance includes real refresh runs of BOTH hub sites** (owner call):
   RHCL first, then Management, each through the gate and ending in a green
   publish run.
4. **#34 scans raw file text**, not a re-serialized frontmatter dict: YAML
   re-serialization can rewrap lines and silently break multi-word patterns
   (e.g. `signed…agreement`). One warning per file, as today; the
   `restricted` path-parts skip is kept.
5. **One generic refresh skill**, not per-site skills (the old repo's shape):
   config-driven, so a third site is a new YAML, not a new skill.
6. **Badge state lives in the publish snapshot**: `.publish-snapshot.json`
   upgrades from `{dest: source}` to `{dest: {source, hash, published, badge}}`.
   No git archaeology, no external state.
7. **CI stays the link-check authority for refresh runs**: local
   `check_links` on a slug dir false-positives on cross-hub links
   (`../../mcp-gateway/rhcl/` only resolves in the pages layout), so the
   skill verifies the publish run goes green instead.

## Part 1: #34 disclosure net gaps

Changes in `scripts/hublib/schema.py` and `scripts/hublib/disclosure.py`:

1. **Frontmatter coverage:** `lint_entry` runs `RESTRICTED_HINTS` over the
   raw file text (frontmatter + body) instead of the body alone. Same
   warning text, still at most one warning per entry, still skipped when
   `"restricted"` is in the path parts.
2. **Generated-surface coverage:** `disclosure._scan_files` adds the
   generated markdown surfaces: `views/*.md`, `memory/index.md`,
   `features/index.md`, `features/*/index.md`, `narrative/index.md`. They
   get both passes: restricted patterns (errors, local-only file) and the
   generic `RESTRICTED_HINTS` heuristic (warnings, CI-visible). Markdown
   entries keep getting their heuristic from `lint_entry`; the new heuristic
   pass in `disclosure.py` applies only to these generated files (they have
   no entry frontmatter and are not linted as entries).
3. **New-surface coverage:** `features/*/work/refresh-*.yaml` and
   `narrative/work/refresh-*.yaml` (Part 3) join the restricted-patterns
   pass, exactly as `jira-snapshot.yaml` does.
4. **Accepted duplication:** a hit can now warn twice (source entry + the
   view quoting it). Each points at a real public line; fine.

Tests: `test_schema.py` gains a frontmatter-hit case (hint in `description:`
only, body clean, expect warning). `test_disclosure.py` gains: view file
with a pattern hit (error), view file with a heuristic hit (warning),
refresh YAML with a pattern hit (error), generated index files scanned.

## Part 2: #8 full-branded landing page

1. **Template extraction:** the landing page moves to a tracked template at
   `publish/landing-template.html`. `generate_landing()` loads it and
   substitutes three slots: grouped-sections HTML, artifact count, hub sha.
   Growing the f-string was rejected: a branded page in a Python string
   literal is unmaintainable.
2. **Structure:** hero header (site title, subtitle, artifact count, hub
   commit) then one section per area, cards per artifact (linked title,
   description, optional badge).
3. **Grouping:** group key derived from the manifest `source` path:
   `features/<id>/…` maps to the feature `title` in `features/features.yaml`;
   `narrative/…` maps to "Narrative". Group order: known features in
   features.yaml routing-table order, then unknown feature ids (defensive,
   grouped under their raw id, alphabetical), then Narrative last. Cards
   sort by title within a group.
4. **Badges:** snapshot v2 is `{dest: {source, hash, published, badge}}` where
   `hash` is sha256 over content (directory artifacts: deterministic walk of
   relative POSIX paths + file bytes). On `apply()`:
   - dest not in old snapshot: `published = today`, badge **NEW**;
   - hash differs from old: `published = today`, badge **UPDATED <date>**;
   - unchanged: `published` carried forward unchanged.
   The badge KIND ("new"/"updated") is persisted in the snapshot so an
   unchanged artifact keeps its NEW badge for the whole window instead of
   flipping to UPDATED.
   Badges render while `published` is within **14 days** of the run date.
   **v1 migration rule:** a v1 snapshot (string values) carries sources
   forward with unknown hash and **no** `published` date, so the first run
   after this ships shows zero false badges; hashes populate on that run.
5. `generate_landing(plan, hub_sha)` keeps its signature shape but `plan`
   entries gain a `group` field (computed in `build_plan`, which reads
   features.yaml) and a `badge` field (computed in `apply`, which owns the
   snapshot). Escaping discipline unchanged (`html.escape` everywhere user
   data lands).
6. The landing page remains subject to the existing CI link gate.

Tests: grouping + ordering (features.yaml order, Narrative last, unknown id
fallback), badge lifecycle (new, updated, aged-out, carried-forward date),
v1-snapshot migration (no false badges, hashes appear), template render
escaping, dir-hash determinism.

## Part 3: #4 `hub.refresh-site`

### Config: `features/<feature>/work/refresh-<slug>.yaml`

```yaml
site: features/mcp-gateway/enablement/rhcl-hub/
sources:
  gdocs:
    - {id: <doc-id>, title: RHCL Overview Deck}
  github:
    - Kuadrant/mcp-gateway
    - kubernetes-sigs/mcp-lifecycle-operator
  jira:
    keys: [RHAIRFE-1370, RHAIRFE-1457, RHAIRFE-1456, RHAISTRAT-1937]
  slack:
    channels: [mcp-gateway, mcp-gateway-dev]
    window_days: 14
  local:
    - features/mcp-gateway/knowledge/
    - features/mcp-gateway/research/
```

Two configs ship in this batch, seeded from the old `update-gateway-hub` /
`update-ecosystem-hub` source lists (GDocs, GitHub repos, Jira keys or JQL,
Slack channels), with `local:` retargeted to the hub's own knowledge/
research partitions (new vs. the old repo: hub entries are a first-class
source). The Management config's Jira scope keeps the old
`parent = RHAISTRAT-1339` JQL as `jira: {jql: …}`; the schema accepts
`keys`, `jql`, or both. A narrative-owned site's config would live at
`narrative/work/refresh-<slug>.yaml`; none ship now.

A small `scripts/hublib/refresh.py` provides `load_config` + `validate`,
wired into `lint_repo`: `site` must exist and be an
`features/*/enablement/<slug>/` or `narrative/enablement/<slug>/` directory,
`sources` must be non-empty with known source types only, gdocs entries
need `id`, slack needs `channels`. Malformed YAML or violations are errors;
an empty optional block is fine.

### Skill: `.claude/skills/hub.refresh-site/SKILL.md`

Ports the old skills' 6-step shape, hub-gated, plus the disclosure contract
the old skills lacked:

1. **Target:** resolve the site from the invocation ("refresh the rhcl hub");
   if ambiguous, list available `refresh-*.yaml` configs. Read config,
   `artifact.md`, and current pages.
2. **Staleness scan:** parse `data-verified` footers across the site's
   pages, report oldest first.
3. **Sweep (parallel subagents):** GDocs via google-workspace MCP (with
   comments), GitHub via `gh` (commits/releases/README changes since the
   oldest data-verified date), Jira via `hublib/jira.py` / stored snapshots,
   Slack via slack MCP over `window_days`, local hub entries. Any
   unreachable source degrades to a "fetch failures" report section; the
   run continues (Slack token brittleness is expected).
4. **Change report:** categorized New / Changed / Confirmed-current /
   Fetch failures, each finding mapped to the specific page(s) it affects.
5. **Gate:** batch approval: all / by number / skip / show diff. Standard
   inline-gate convention; never auto-commits.
6. **Apply:** surgical in-place HTML edits, never a rebuild. Bump
   `data-verified` to today on every page that was swept-and-updated or
   swept-and-confirmed; unswept pages keep their date. Bump the descriptor
   `timestamp`.
7. **Disclosure gate (the contract):** before commit, `hub_lint.py` must be
   clean over touched files; restricted-pattern errors are hard blockers.
   Skill text embeds the migration scrub rules: never introduce customer or
   partner names (use the anonymized phrasing, e.g. "large enterprise OEM
   customer"); product naming full-then-short ("RHOAI restricted use
   entitlement for OpenShift"); no em dashes in newly written content.
8. **Republish:** commit with a bulleted body and push; `publish.yml`
   republishes and runs the link gate. The skill watches the run to green
   (`gh run watch`) rather than link-checking locally (decision 7).

### Acceptance

Both hubs refreshed through the gate in this batch:

1. RHCL hub run: sweep, gated diffs, apply, green publish run.
2. Management hub run: same, after RHCL proves the skill.

Friction found in run 1 is fixed before run 2. Outcomes recorded in the
plan doc and `memory/log.md`.
