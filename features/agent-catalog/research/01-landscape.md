# Agent catalog landscape — definitions, state of the art, patterns

Research date: 2026-07-16. Sources are a mix of vendor documentation/blogs (marked where the claim is marketing rather than verifiable product fact), press coverage, standards documentation, and one academic survey. RHOAI context used for framing only; nothing here is Red Hat-internal.

## 1. Taxonomy: what "agent catalog" means in 2025–2026

The market has converged on three distinct object types that are frequently conflated under one word. The cleanest published carve-up:

- **Agent catalog** — a curated, governed *directory* focused on discovery and internal oversight. Salesforce's definitional page describes an AI agent catalog as "a centralized directory that organizes, tracks, and governs AI agents across an enterprise environment," with a registry layer tracking each agent's name, owner, purpose, connected systems, and version history ([Salesforce — What Is an AI Agent Catalog?](https://www.salesforce.com/agentforce/ai-agents/ai-agent-catalog/)). Note this is Salesforce defining the term to suit its product, but it matches the pattern across vendors.
- **Agent marketplace / store** — a *procurement and distribution* surface: discover, buy/install, deploy third-party agents, usually with a certification gate and commercial terms ([TrueFoundry — AI Agent Marketplaces](https://www.truefoundry.com/blog/ai-agent-marketplaces); [AWS Marketplace AI Agents and Tools](https://aws.amazon.com/marketplace/solutions/ai-agents-and-tools)).
- **Agent registry** — *runtime infrastructure*: verifiable identity, capability advertisement, and discovery for agents that exist and are (about to be) running. An August 2025 arXiv survey taxonomizes registries into centralized (MCP Registry), enterprise (Microsoft Entra Agent ID), and distributed (A2A Agent Cards, AGNTCY Agent Directory, NANDA AgentFacts) approaches, with the core functions being "trustworthy agent discovery, capability negotiation, and identity assurance" ([arXiv 2508.03095 — Evolution of AI Agent Registry Solutions](https://arxiv.org/abs/2508.03095); [TrueFoundry — What is an AI Agent Registry](https://www.truefoundry.com/blog/ai-agent-registry)).

A fourth object rides underneath all three: the **template / sample / blueprint gallery** — source code you copy, not software you run. Google's Agent Garden, Microsoft's Foundry agent catalog (the code-sample one), NVIDIA Blueprints, and Red Hat's [agentic-starter-kits](https://github.com/red-hat-data-services/agentic-starter-kits) all live here.

**Mapping to RHOAI's three-body model:** RHOAI's Agent Catalog (discovery of curated templates) is a template gallery in this taxonomy; the 3.6 EA1 deploy path moves it toward a deployable store; the MLflow-based Agent Registry (a configured deployed instance = a version) matches the industry registry concept, and MLflow 3's `LoggedModel` entity is explicitly designed for this — "systematic version control for your entire LLM application or AI agent — code, configurations, evaluations, and traces" ([MLflow — Version Tracking for Agents and LLMs](https://mlflow.org/docs/latest/genai/version-tracking/)). The catalog/registry split RHOAI uses also mirrors upstream [kubeflow/hub](https://github.com/kubeflow/hub) itself: Catalog = "a read-only discovery service that aggregates [assets] from external catalog sources," Registry = where teams "register, version, and track" their own artifacts ([Kubeflow Hub overview](https://www.kubeflow.org/docs/components/hub/overview/)).

## 2. State of the art, vendor by vendor

Summary table (verified against sources below; "deploy UX" = what the primary catalog action does):

| Vendor / surface | What ships | Deploy UX | Curation / support |
|---|---|---|---|
| Google Agent Garden | ADK code samples + solutions | One-click deploy to Agent Engine via Agent Starter Pack | Google-curated first-party samples |
| Google Cloud AI Agent Marketplace | Partner-built deployable agents | Marketplace procurement → surfaces in Gemini Enterprise Agent Gallery | Partner listings, Google-reviewed |
| AWS Marketplace "AI Agents and Tools" | SaaS/container/AMI/API agent products | Buy → deploy (AgentCore Runtime path for compatible listings) | Partner listings; MCP/A2A support flagged |
| AWS AgentCore | Runtime + CLI + IaC templates; samples on GitHub | CLI project workflow, CloudFormation/CDK/Terraform templates | First-party samples (awslabs) |

| M365 Copilot Agent Store | Installed, running agents | Install, gated by Microsoft validation + tenant-admin approval | Two-gate: marketplace validation + admin |
| Databricks Agent Bricks / Unity Catalog | Agents as governed Unity Catalog assets | Build/deploy in-platform; agents, MCPs, skills registered in UC | Governance-first, not a storefront |

### Google — the clearest three-surface separation

Google runs **three distinct surfaces** for three audiences, which is the strongest available answer to the template-vs-deployable question:

1. **Agent Garden** (developer surface): "a growing repository of curated agent samples, solutions, and tools" built with ADK; each entry has an overview, architecture insights, capability descriptions, and a GitHub link — and supports **one-click deployment to Vertex AI Agent Engine** via the open-source Agent Starter Pack, landing in a playground UI for testing ([Google Developers Blog — Agent Garden](https://developers.googleblog.com/agent-garden-samples-for-learning-discovering-and-building/)). So even Google's *sample* gallery grew a deploy button — but it deploys to a sandbox/playground, not production. The [Agent Starter Pack](https://googlecloudplatform.github.io/agent-starter-pack/) is the separate template-to-production path: "production-ready templates with built-in CI/CD, evaluation, and observability."
2. **AI Agent Marketplace** (procurement surface): launched at Next '25 as a dedicated Google Cloud Marketplace category for partner-built agents; grown to 70+ partner agents (Accenture, Adobe, Deloitte, UiPath…) with formal [publishing docs for partners](https://docs.cloud.google.com/marketplace/docs/partners/ai-agents) ([Google Cloud blog — AI Agent Marketplace](https://cloud.google.com/blog/topics/partners/google-cloud-ai-agent-marketplace)).
3. **Agent Gallery** (end-user surface): marketplace-procured partner agents surface directly inside the Gemini Enterprise app alongside first-party and custom agents ([Google Cloud blog — partner-built agents in Gemini Enterprise](https://cloud.google.com/blog/products/ai-machine-learning/partner-built-agents-available-in-gemini-enterprise)). Note: Google consolidated branding in 2026 — Vertex AI became the "Gemini Enterprise Agent Platform" ([Google Cloud docs](https://docs.cloud.google.com/gemini-enterprise-agent-platform/overview)); treat exact product names as unstable.

### AWS — marketplace-first, runtime underneath

AWS split the problem into a **runtime** (Bedrock AgentCore, GA October 2025: identity, memory, gateway, observability for any-framework agents — [AWS News Blog](https://aws.amazon.com/blogs/aws/introducing-amazon-bedrock-agentcore-securely-deploy-and-operate-ai-agents-at-any-scale/)) and a **procurement category**: "AI Agents and Tools" launched in AWS Marketplace July 16, 2025 with ~900 listings ([AWS what's-new](https://aws.amazon.com/about-aws/whats-new/2025/07/ai-agents-tools-aws-marketplace/)), reported at 3,400+ partners a year later — "the fastest-growing category in AWS Marketplace history" ([SiliconANGLE](https://siliconangle.com/2026/07/15/aws-product-listing-service-enterprise-agents-ai-awsmarketplaceseries/); growth figures are AWS-sourced, treat as marketing-adjacent). Listings declare deployment options (SaaS, container, AMI, API) and whether they support MCP/A2A; sellers can list agents deployable onto **AgentCore Runtime**, with documented technical requirements ([AWS Marketplace buyer guide — AgentCore integration](https://docs.aws.amazon.com/marketplace/latest/buyerguide/ai-agent-bedrock-agentcore-integration.html)). First-party templates live separately as a GitHub samples repo ([awslabs/agentcore-samples](https://github.com/awslabs/agentcore-samples)) plus IaC templates (CloudFormation/CDK/Terraform — [AWS ML blog](https://aws.amazon.com/blogs/machine-learning/build-ai-agents-with-amazon-bedrock-agentcore-using-aws-cloudformation/)). AWS notably does **not** put a deploy button on its samples — the path is CLI/IaC.

### Microsoft — "catalog" means code samples; "store" means installed agents

Microsoft uses the exact word "agent catalog" for what RHOAI 3.5 is shipping: "a growing library of agent code samples from Microsoft and partners like Auquan, Marquee Insights, MiHCM, Saifr, and Sight Machine, providing developers with reusable building blocks" ([Azure blog — Microsoft Foundry agent stack](https://azure.microsoft.com/en-us/blog/microsoft-foundry-scale-innovation-on-a-modular-interoperable-and-secure-agent-stack/); samples on [GitHub](https://github.com/microsoft-foundry/foundry-samples)). Deployables are handled elsewhere: [Foundry Agent Service](https://learn.microsoft.com/en-us/azure/foundry/agents/overview) runs the agents, a separate 1,400+-entry [tool catalog](https://learn.microsoft.com/en-us/azure/foundry/agents/concepts/tool-catalog) covers tools, and the Build 2026 developer path is "create agents from templates or with GitHub Copilot… deploy to Foundry Agent Service directly from VS Code" ([Foundry blog — Build 2026](https://devblogs.microsoft.com/foundry/agent-service-build2026/)). The end-user surface, the **M365 Copilot Agent Store**, is the strictest deploy gate observed: agents pass Responsible-AI manifest validation, then Microsoft marketplace validation, then **tenant-admin approval with audience scoping** before appearing to users ([Agent Store docs](https://learn.microsoft.com/en-us/microsoft-365/copilot/copilot-agent-store); [admin management docs](https://learn.microsoft.com/en-us/microsoft-365/admin/manage/manage-copilot-agents-integrated-apps?view=o365-worldwide)).

### Databricks — the registry-maximalist position

Databricks has no public agent storefront; its bet is that **agents are governed assets in the same catalog as data**. Unity Catalog now registers "Databricks-hosted models, external model providers, MCP services, agents, and skills… secured and audited using the same governance framework" ([Databricks — Unity Catalog at DAIS 2026](https://www.databricks.com/blog/whats-new-unity-catalog-data-ai-summit-2026)), with Unity AI Gateway extending governance to runtime interactions ([Databricks — Unity AI Gateway](https://www.databricks.com/blog/ai-governance-data-ai-summit-2026-whats-new-unity-ai-gateway)). Agent Bricks is the build/deploy surface, supporting external harnesses (LangGraph, CrewAI, Claude Code SDK, OpenAI Agent SDK) with identity enforced end-to-end via on-behalf-of token passing ([Agent Bricks platform blog](https://www.databricks.com/blog/agent-bricks-governed-enterprise-agent-platform); "100K+ agents built" claims from [DAIS 2026](https://www.databricks.com/blog/agent-bricks-dais-2026) are vendor marketing). Under the hood, MLflow 3's LoggedModel links each agent version to git commit, config, traces, and eval runs ([Databricks docs — version concepts](https://docs.databricks.com/aws/en/mlflow3/genai/prompt-version-mgmt/version-tracking/version-concepts)) — the same foundation as RHOAI's planned Agent Registry.

### IBM — the deployable-catalog archetype

The [watsonx Orchestrate Agent Catalog](https://www.ibm.com/products/watsonx-orchestrate/agent-catalog) is a true deployable store: 150+ prebuilt agents and tools at GA (Think 2026), with partner agents from Box, Mastercard, Oracle, Salesforce, ServiceNow and others ([Enterprise DNA coverage](https://enterprisedna.co/resources/news/ibm-think-2026-watsonx-orchestrate-agent-catalog-enterprise/)). Its differentiator is framework-agnosticism: "any agent, any framework" onboarding via the **Agent Connect** partner program, which standardizes communication with chat-completion-style APIs and MCP, supporting LangChain, LangGraph, CrewAI, Copilot Studio and custom frameworks ([IBM Agent Connect](https://www.ibm.com/products/watsonx-orchestrate/agent-connect); [InfoWorld](https://www.infoworld.com/article/3978695/ibm-updates-watsonx-orchestrate-with-new-agent-building-capabilities.html)). Catalog entries are added/configured into a running Orchestrate environment — the action verb is "add," not "clone."

### NVIDIA — blueprints: the reference-workflow middle ground

[NVIDIA Blueprints](https://build.nvidia.com/blueprints) are the best-defined *template artifact spec* in the market: each blueprint ships "sample applications built with NVIDIA NIM and partner microservices, reference code, customization documentation and a Helm chart for deployment" ([NVIDIA blog — NIM Agent Blueprints](https://blogs.nvidia.com/blog/nim-agent-blueprints/)). Deploy options span one-click **Launchables**, local download, or datacenter/cloud Helm deployment ([NVIDIA blog — Agentic AI Blueprints](https://blogs.nvidia.com/blog/agentic-ai-blueprints/)). Partners (Accenture, Deloitte, SAP, ServiceNow at launch) contribute blueprints, and the commercial support model sits underneath in NVIDIA AI Enterprise rather than per-blueprint ([NVIDIA newsroom](https://nvidianews.nvidia.com/news/nvidia-and-global-partners-launch-nim-agent-blueprints-for-enterprises-to-make-their-own-ai)). This is the closest structural analog to "starter kit with a supported-images runtime underneath."

### Salesforce — componentized marketplace

[AgentExchange](https://www.salesforce.com/news/press-releases/2025/03/04/agentexchange-announcement/) (March 2025, 200+ launch partners) sells **four component types**, not just whole agents: *actions* (integrations), *prompt templates*, *topics* (grouped actions + instructions), and *agent templates* (full solutions combining topics/actions/metadata). Everything passes "rigorous security and customer reviews" before listing, and listings are browsable both on the marketplace and inside Agent Builder ([Salesforce Ben analysis](https://www.salesforceben.com/agentexchange-salesforce-launches-trusted-marketplace-for-agentforce/)). The componentization is notable: "agent template" is an explicit, first-class catalog type distinct from a running agent.

### ServiceNow — store + partner-tier certification

The [ServiceNow Store AI Agent Marketplace](https://store.servicenow.com/store/ai-marketplace) sells installable Now Assist agents (771+ agent records tracked by third-party directory [SNAgents](https://snagents.ai/)). In January 2026 ServiceNow rebuilt its partner program around the store with a tier structure — Registered, Select, Premier, Elite — explicitly positioning the Store as "a global marketplace for partner-built AI agents" ([ServiceNow newsroom](https://newsroom.servicenow.com/press-releases/details/2026/ServiceNow-enhances-global-Partner-Program-to-accelerate-AI-agent-innovation/default.aspx)). Certification lives in the partner tier, not per-listing.

### Hugging Face — the open/community pole

HF treats agents as Hub citizens: agents push to the Hub as Spaces, [Spaces can act as agent tools](https://huggingface.co/docs/hub/en/spaces-agents), and [skills](https://github.com/huggingface/skills) are "self-contained folders that package instructions, scripts, and resources" — with discovery of skills, MCP servers, and Spaces being standardized over MCP resources ([HF agents docs](https://huggingface.co/docs/hub/agents-overview)). Minimal curation, no support tiers — the anti-enterprise reference point.

## 3. Emerging best practices

**Curation models cluster into three tiers, and most vendors run at least two.** (a) *First-party curated* samples/templates (Agent Garden, Microsoft foundry-samples, awslabs, agentic-starter-kits); (b) *partner-certified* listings with an explicit gate — Salesforce security review, Microsoft two-stage validation, ServiceNow partner tiers, IBM Agent Connect onboarding; (c) *community-open* (Hugging Face, ADK community samples). No major vendor mixes community content into its certified deployable surface — the tiers are kept on separate surfaces or separate badges. RHOAI's "Red Hat-built supported images" vs "vendor-maintained validated images" wording maps onto tier (a) vs (b); the industry norm is that the *stricter* word attaches to the party who carries the support obligation.

**Metadata is converging on structured capability manifests, not READMEs.** The A2A **Agent Card** is the emerging deployed-agent standard: a machine-readable JSON manifest requiring `name`, `description`, `version`, `capabilities`, `defaultInputModes/OutputModes`, and `skills` ([A2A protocol — agent discovery](https://a2a-protocol.org/latest/topics/agent-discovery/)); by 2026 A2A sits under the Linux Foundation with 150+ supporting organizations including Microsoft, AWS, Salesforce, and ServiceNow (adoption figures from secondary sources — [arXiv survey](https://arxiv.org/abs/2508.03095) corroborates the direction). For *catalog* (pre-deploy) entries, the de facto card fields across Agent Garden, Blueprints, and AgentExchange are: overview, use case, architecture diagram, framework, required services/models, deploy options, and source link. Template galleries still lean on README + card; deployable stores require structured capability declarations (AWS listings must declare deployment option and MCP/A2A support — [AWS Marketplace docs](https://docs.aws.amazon.com/marketplace/latest/userguide/ai-agents-tools.html)). Best practice: structure the fields you filter on; keep prose for the rest.

**Deploy UX: card → detail page → action verb matched to artifact maturity.** Every surveyed catalog uses browse-cards + detail page; the differentiator is the primary action. Observed verbs: "View on GitHub" (Microsoft agent catalog, RHOAI 3.5), "Deploy" to a *playground* (Agent Garden → Agent Engine playground), "Launch" one-click ephemeral (NVIDIA Launchables), "Install/Add" into a governed environment with admin gate (M365, ServiceNow, IBM), and CLI/IaC-only for production (AWS). Nobody puts a one-click *production* deploy on a code sample — where one-click exists it lands in a sandbox/playground with an explicit separate path to production (Agent Starter Pack CI/CD, AgentCore CLI, VS Code → Agent Service).

**Governance gates sit at deploy time, not catalog time.** The M365 pattern (validate at listing, approve at tenant install, scope audience per agent) and Databricks' runtime policies (allow/deny/require-approval per action — [Unity AI Gateway](https://www.databricks.com/blog/ai-governance-data-ai-summit-2026-whats-new-unity-ai-gateway)) both put the heavyweight control at the point where an agent gains access to an environment. Analyst framing agrees: Forrester positions governance as a control plane that must "sit outside both build and orchestration environments" ([Forrester — Agent Control Planes](https://www.forrester.com/blogs/agent-control-planes-still-need-a-robust-standards-stack/)), and Gartner predicts half of agent deployment failures through 2030 will stem from governance gaps (via [secondary compilation](https://joget.com/ai-agent-adoption-in-2026-what-the-analysts-data-shows/) — not verified against the primary report).

**Template-to-production is a named, separate path.** Google is most explicit: Agent Garden for discovery/learning, Agent Starter Pack ("production-ready templates with built-in CI/CD, evaluation, and observability") for the production journey ([agent-starter-pack](https://github.com/GoogleCloudPlatform/agent-starter-pack)). AWS retired its "Starter Toolkit" in favor of a project-based AgentCore CLI workflow plus IaC templates. The pattern: don't overload the catalog card with the production path; link to it.

## 4. The template-vs-deployable distinction — how vendors resolve it

This is the exact question RHOAI has open (deploy vs register button; starter kits vs quick starts), and the market answer is consistent: **don't blur the surfaces — change the verb, or change the surface.**

1. **Separate surfaces per artifact maturity (Google, Microsoft, AWS).** Code samples live in a "garden"/"catalog" whose action is *view/copy/clone*; deployable products live in a "marketplace"/"store" whose action is *buy/install*; running instances live in a gallery/registry. Google runs all three simultaneously without collision because the names, audiences, and verbs differ.
2. **One surface, graduated verbs (NVIDIA, Agent Garden's evolution).** NVIDIA keeps blueprints as reference workflows but offers escalating actions on the same card: try hosted → launch one-click → download Helm chart for self-managed production. Agent Garden added one-click deploy but pointed it at a *playground*, preserving the "this is a sample" contract. Precedent for RHOAI: a deploy button on a template card is defensible **if** the target is explicitly a dev/sandbox instance, not production.
3. **Deployment creates a different object in a different system (Databricks, MLflow — and RHOAI's stated design).** In the Unity Catalog/MLflow model, the catalog/template is where you start; the moment an agent is configured and logged, it becomes a versioned `LoggedModel` in the registry with its own lifecycle ([MLflow version tracking](https://mlflow.org/docs/latest/genai/version-tracking/)). RHOAI's "configured deployed instance is a version belonging to the Agent Registry" is the same architecture Databricks ships — the strongest external validation of the current RHOAI plan. Under this model the catalog button is naturally **"Deploy"** (an act that *produces* a registry entry), not "Register" — registration is a side effect of deployment, and no surveyed vendor exposes "register" as an end-user catalog verb.
4. **Naming: "starter kit" vs "quick start" is not standardized, but "blueprint"/"template"/"sample" carry consistent expectations.** "Sample" = learning code, no support expectation (Microsoft, ADK). "Template"/"starter pack" = scaffolding you generate a project from, production path attached (Google, Azure VS Code toolkit). "Blueprint" = runnable reference workflow with deployment assets included (NVIDIA, Helm chart in the box). "Quick start" is used mostly for docs-level tutorials, not catalog artifacts (e.g., [Google's ADK quickstarts](https://docs.cloud.google.com/gemini-enterprise-agent-platform/build/runtime/quickstart-adk); Red Hat's own separate [aiquickstarttemplates](https://github.com/redhat-developer/aiquickstarttemplates) repo). If RHOAI's kits ship Helm-chart-like deployment assets, "blueprint" semantics apply; if they're clone-and-modify repos, "starter kit/template" is the honest label — and the 3.5 GitHub-link-out behavior matches Microsoft's agent catalog precedent almost exactly.
5. **Support tiers attach to the runtime, not the template.** NVIDIA supports blueprints via NVIDIA AI Enterprise (the runtime/microservices), not per-blueprint SLAs; AWS Marketplace listings carry seller support; Microsoft samples carry none. RHOAI 3.6's "supported images" framing (support attaches to the Red Hat-maintained harness images an agent deploys onto, while catalog templates stay examples) is aligned with this pattern.

## Candidate knowledge atoms

- **fact**: The industry taxonomy is three-layered — template gallery (copy code), agent marketplace/store (install products), agent registry (versioned running instances with identity) — and no major vendor merges certified-deployable and community-sample content on one surface. Sources: [arXiv 2508.03095](https://arxiv.org/abs/2508.03095), [Salesforce catalog definition](https://www.salesforce.com/agentforce/ai-agents/ai-agent-catalog/).
- **fact**: Microsoft's Foundry "agent catalog" is precisely RHOAI 3.5's shape — curated open-source agent code samples from Microsoft + named partners, consumed via GitHub, no deploy button — validating the 2026-07-09 no-deploy-button decision for a Dev Preview ([Azure blog](https://azure.microsoft.com/en-us/blog/microsoft-foundry-scale-innovation-on-a-modular-interoperable-and-secure-agent-stack/)).
- **fact**: Google Agent Garden's one-click deploy targets the Agent Engine *playground*, not production; the production path is a separate named artifact (Agent Starter Pack with CI/CD). Precedent that a template-catalog deploy button is acceptable if it lands in a sandbox ([Google Developers Blog](https://developers.googleblog.com/agent-garden-samples-for-learning-discovering-and-building/)).
- **fact**: Databricks ships the same catalog→registry architecture RHOAI plans: agents/skills/MCPs registered as governed Unity Catalog assets, with MLflow 3 `LoggedModel` binding each agent version to code, config, traces, and evals ([Unity Catalog DAIS 2026](https://www.databricks.com/blog/whats-new-unity-catalog-data-ai-summit-2026), [MLflow version tracking](https://mlflow.org/docs/latest/genai/version-tracking/)).
- **ref**: A2A Agent Card (Linux Foundation) is the emerging metadata schema for deployed-agent discovery — required fields include name, version, capabilities, skills, I/O modes; worth tracking as a future Agent Registry interchange format ([A2A agent discovery](https://a2a-protocol.org/latest/topics/agent-discovery/)).
- **fact**: NVIDIA Blueprints define the fullest "template artifact spec" in market: sample app + NIM microservices + reference code + customization docs + Helm chart, with support attached to the runtime (NVIDIA AI Enterprise) rather than the blueprint — the closest analog to starter kits over Red Hat-supported harness images ([NVIDIA blog](https://blogs.nvidia.com/blog/nim-agent-blueprints/)).
