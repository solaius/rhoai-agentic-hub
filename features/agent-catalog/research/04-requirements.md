Requirements-lens research for the [Agent Catalog](/features/agent-catalog/index.md): what enterprise customers will actually require of a curated agent storefront that gains a deploy path via [supported harness images](/features/agent-catalog/knowledge/fact-agent-catalog-36-supported-images.md) in 3.6 EA1. Findings separate **hard requirements** (documented external evidence) from **expectations** (reasoned inference). Each section flags which of the hub's open `question-` entries it bears on.

## 1. Disconnected / air-gapped operation

**Hard requirements (evidence).**

- Red Hat already documents a formal disconnected-install contract for RHOAI: mirror all required images to a private registry (Docker V2-2 compatible, e.g. Quay 3.6+) using oc-mirror, with a published "disconnected installer helper" listing every additional image required — including optional-component images ([RHOAI disconnected deployment docs](https://docs.redhat.com/en/documentation/red_hat_openshift_ai_self-managed/2-latest/html/installing_and_uninstalling_openshift_ai_self-managed_in_a_disconnected_environment/deploying-openshift-ai-in-a-disconnected-environment_install); [oc-mirror v2 docs](https://docs.redhat.com/en/documentation/openshift_container_platform/4.21/html/disconnected_environments/about-installing-oc-mirror-v2)). Anything the Agent Catalog ships must slot into this exact mechanism: an enumerable, mirrorable image list. A deploy button whose images aren't on the helper list is, for these customers, a broken feature.
- In air-gapped/classified environments, *nothing may be pulled at runtime*: every image reference must resolve to the local registry, updates arrive on signed media on monthly-to-quarterly cadences, and installation bundles must include all images, weights, and libraries ([defense air-gap practices](https://corvusintell.com/blog/secure-cloud/air-gapped-deployment-defense/); [air-gapped LLM deployment in regulated finance/defense](https://www.truefoundry.com/blog/air-gapped-ai-deploying-enterprise-llms-in-highly-regulated-industries)). This makes "install harness at runtime" categorically non-compliant for this segment — it is not a degraded mode, it is disallowed.
- Self-updating binaries are an active pain point: OpenCode's community has open feature requests for a true air-gapped mode that disables auto-update/version checks and all non-LLM network access ([opencode #16117](https://github.com/anomalyco/opencode/issues/16117), [#20027 how to disable auto-update](https://github.com/anomalyco/opencode/issues/20027)), with `autoupdate: false` config as the current partial answer. A Red Hat supported OpenCode image **must** ship with update/telemetry/version-check calls disabled or removed — patch/fork level work, exactly as flagged in [question-agent-catalog-supported-harness-set](/features/agent-catalog/knowledge/question-agent-catalog-supported-harness-set.md).

**Expectations (inference).**

- The staleness-vs-disconnected tension in [fact-agent-catalog-36-supported-images](/features/agent-catalog/knowledge/fact-agent-catalog-36-supported-images.md) resolves *in favor of disconnected* for the enterprise segment: regulated customers already accept month(s)-old mirrored content as the price of isolation ([signed-media update cadence](https://corvusintell.com/blog/secure-cloud/air-gapped-deployment-defense/)). "Baked-in goes stale in a month" is a demo-audience problem, not an enterprise blocker — the enterprise blocker is the reverse (runtime fetch). Ann Marie Fred's "possibly unsolvable problem" is solvable by picking the customer: self-contained images + a documented refresh cadence.
- Skills-as-markdown mounted from a PVC (the idea already on the table) fits the air-gap model well — content updates without image rebuilds, still no network fetch. Expect customers to ask that skill bundles be mirrorable artifacts too (OCI artifacts would ride the existing oc-mirror flow).
- The 3.5 decision to bake catalog metadata into a YAML source for disconnected support ([fact-agent-catalog-overview](/features/agent-catalog/knowledge/fact-agent-catalog-overview.md)) sets the precedent customers will expect 3.6 to keep: the catalog *browse* experience must also work offline, even though 3.5 cards link out to GitHub (dead links in an enclave — a known cosmetic gap worth documenting).

## 2. Supportability and security posture of shipped images

**Hard requirements (evidence).**

- Red Hat's own Container Health Index grades every catalog image A–F by age of unapplied Critical/Important errata — grade A means no known unapplied Critical/Important fixes; missing a Critical fix for >30 days already drops an image to C ([Container Health Index grades](https://access.redhat.com/articles/2803031)). Customers will hold agent-harness images to the same visible standard as every other image in the Red Hat Ecosystem Catalog; there is no plausible carve-out for "AI images".
- Red Hat's stated policy: Critical and Important CVEs are fixed in all supported product versions, with severity definitions and lifecycle commitments published ([vulnerability management lifecycle](https://access.redhat.com/security/vulnerability-management-lifecyle); [severity ratings](https://access.redhat.com/security/updates/classification)). Whoever owns the supported harness images inherits this treadmill — this is the concrete cost behind [question-agent-catalog-ownership-packaging](/features/agent-catalog/knowledge/question-agent-catalog-ownership-packaging.md), and it is a *product-security policy obligation*, not a nice-to-have.
- Government-adjacent customers operate under FedRAMP-style clocks: high-severity findings remediated within 30 days, only images scanned within a 30-day window deployable to production ([FedRAMP vulnerability scanning requirements](https://www.chainguard.dev/unchained/fedramp-vulnerability-scanning-requirements-explained); [FedRAMP RFC-0012](https://www.fedramp.gov/rfcs/0012/)). An agent image nobody rebuilds for a quarter becomes undeployable in these environments regardless of feature quality.
- Supply-chain artifacts are now procurement table stakes: SBOMs (driven by EO 14028 / OMB M-22-18), signatures, and SLSA provenance attestations verified by admission controllers ([SBOM attestations with Sigstore](https://anchore.com/sbom/creating-sbom-attestations-using-syft-and-sigstore/)). Red Hat's Konflux pipeline already produces signed builds, SLSA provenance, SBOMs, and hermetic builds ([how Red Hat uses software provenance](https://developers.redhat.com/articles/2025/05/15/how-we-use-software-provenance-red-hat)) — and OpenShell DP images are already planned through Konflux ([roadmap](/memory/profiles/roadmap.md)), so the *mechanism* exists; the requirement is that harness images go through it, not a bespoke path. This is direct ammunition for the [AIPCC base-images fit question](/features/agent-catalog/knowledge/question-agent-catalog-aipcc-base-images-fit.md): whatever builds the final images must emit Konflux-grade attestations.
- Precedent for tier communication: RHOAI "validated models" are explicitly *not supported, indemnified, certified or guaranteed* — validated means tested/benchmarked/scanned, with the boundary stated in public docs ([Red Hat AI validated models](https://www.redhat.com/en/products/ai/validated-models)). This external precedent confirms the [supported-not-validated terminology decision](/features/agent-catalog/knowledge/decision-supported-not-validated-images.md): the words already carry defined support-boundary meaning to customers and must be used consistently.

**Expectations (inference).**

- Customers will expect a published *support matrix per image* ("works with vLLM / Responses API / these models") and an explicit statement of what is NOT supported (the embedded frameworks, the upstream harness's own bugs). The MLflow-style due-diligence cost Daniele Zonca flagged ([framework-support boundary](/features/agent-catalog/knowledge/question-agent-catalog-framework-support-boundary.md)) argues for keeping framework-embedding kits out of the supported-image tier entirely and leaving them as link-out templates — a two-tier catalog (supported images vs community templates) mirroring the Health-Index-graded vs ungraded split customers already navigate.
- Expect demand for a stated image-refresh cadence (e.g. rebuild on base-image errata + monthly harness bump), because the Health Index makes staleness *publicly visible*. This converts the "harnesses go stale in a month" tension from an internal worry into an externally graded metric.

## 3. Licensing and procurement of proprietary harnesses

**Hard requirements (evidence).**

- Claude Code is proprietary: the LICENSE is "© Anthropic PBC. All rights reserved," use is governed by Anthropic's Commercial Terms, and modification/redistribution are prohibited — embedding it in another product is generally not permitted ([Claude Code legal and compliance](https://code.claude.com/docs/en/legal-and-compliance); [CLI agent license comparison](https://codenote.net/en/posts/cli-agent-license-commercial-use-comparison/)). Red Hat cannot ship a Claude Code image without an Anthropic agreement — confirming the licensing flag in [question-agent-catalog-supported-harness-set](/features/agent-catalog/knowledge/question-agent-catalog-supported-harness-set.md) as a legal fact, not just a risk.
- By contrast, Codex CLI is Apache-2.0 ([openai/codex LICENSE](https://github.com/openai/codex/blob/main/LICENSE)) and Goose is Apache-2.0, donated to the Linux Foundation's agentic AI foundation ([Block announcement](https://block.xyz/inside/block-open-source-introduces-codename-goose)). Redistribution is legally clean for these — the open question for Codex is *practical* vendor lock-in (model/backend coupling), not license.
- The established industry pattern for exactly this problem is BYOL: the catalog lists the product, the customer brings their own vendor license/credentials, and the marketplace never redistributes the licensed bits ([BYOL in cloud marketplaces](https://www.economize.cloud/glossary/bring-your-own-license); F5, Oracle, IBM all ship marketplace BYOL SKUs).

**Expectations (inference).**

- Enterprises will not be blocked by the *absence* of Claude Code from the supported set — they will be blocked by ambiguity. Three clean postures customers can procure against: (a) Red Hat-redistributed supported image (open-source harnesses only); (b) BYOL Containerfile recipe — Red Hat ships the build instructions, customer's own subscription pulls the proprietary harness at *build* time inside their firewall (also air-gap compatible, since the build happens on their connected staging side); (c) not in catalog. A card that looks deployable but hides licensing friction damages trust in the whole catalog.
- Note the model-key asymmetry: even fully redistributable harnesses still need customer-side model access. The platform's agentic API surface with runtime model-name translation ([fact-agent-deployment-modes](/features/agent-catalog/knowledge/fact-agent-deployment-modes.md)) is the differentiator here — expect customers to require that supported images default to *platform-hosted* models, with external-API credentials as an explicit opt-in (data-egress governance).

## 4. Persona demands (mapped to the locked persona list)

Grounded in the [locked persona set](/features/platform/knowledge/fact-personas.md) plus industry evidence on portals, catalogs, and agent adoption.

**Evidence base.** Portal/catalog trust is fragile: only 3% of engineers fully trust their portal's metadata and 50% doubt its accuracy — when trust breaks, they route around the catalog to humans ([Port 2025 State of Internal Developer Portals](https://www.port.io/state-of-internal-developer-portals)). Golden-path templates are the atomic unit of platform value — pre-approved, opinionated, self-service ([Red Hat on golden paths](https://www.redhat.com/en/topics/platform-engineering/golden-paths)); PagerDuty cut median new-service creation from 75 to 25 hours with template-driven scaffolding ([Spotify Portal case](https://backstage.spotify.com/discover/blog/pagerduty-switches-to-spotify-portal)). On the agent side: 81% of organizations feel pressure to deploy agents before security/governance is ready, ~48% of production agents run effectively unmonitored, and security/risk is the top barrier to scaling ([Gravitee State of AI Agent Security 2026](https://www.gravitee.io/blog/state-of-ai-agent-security-2026-report-when-adoption-outpaces-control); [McKinsey, State of AI trust 2026](https://www.mckinsey.com/capabilities/tech-and-ai/our-insights/tech-forward/state-of-ai-trust-in-2026-shifting-to-the-agentic-era)).

Per persona (expectation/inference, anchored in the above):

- **ai-engineer** — discovery + working code: accurate cards (protocols, models tested-with, resource needs), clone-and-run starter kits, and a path from template to *their* configuration. The Port trust finding is the warning: one stale card (dead GitHub link, wrong model claim) and they bypass the catalog permanently. Bears on the 3.5 link-out design and [question-agent-catalog-quickstarts-overlap](/features/agent-catalog/knowledge/question-agent-catalog-quickstarts-overlap.md).
- **data-scientist** — the "tested with these models" disclosure is the trust signal they read first; they need eval hooks and swap-the-model instructions, not harness internals. Skills shipping as "tested with these models — mileage may vary" is acceptable *if machine-readable and visible on the card*.
- **platform-engineer** — mirrorability (§1), Konflux-grade provenance (§2), and golden-path packaging: the catalog entry should be deployable via the same GitOps flows as everything else, not only the UI button — which is Andrew Ballantyne's point inside [question-agent-catalog-register-vs-deploy](/features/agent-catalog/knowledge/question-agent-catalog-register-vs-deploy.md).
- **agentops-admin** — the persona the Gravitee numbers scream about: inventory of what is deployed from the catalog, from which image/version, with what tools/skills attached; lifecycle ops (update, suspend, decommission). The 3.5 read-only agent-deployments view is the seed; §5 covers the registration linkage.
- **cluster-admin / rhoai-admin** — control over *which* catalog entries are exposed to users (curation/allow-listing per project or tenant), quota/sizing implications of harness pods, and assurance that a deploy button cannot bypass admission policy (image-signature verification via Kyverno/Gatekeeper-style controllers is now standard practice — [Sigstore admission patterns](https://anchore.com/sbom/creating-sbom-attestations-using-syft-and-sigstore/)).
- **business-consumer** — never touches the catalog; their requirement lands on everyone else as *trustworthy defaults*: agents deployed from the catalog behave within governed boundaries (guardrails, identity, disclosure per §5). Per McKinsey, trust in agent behavior — not capability — is what gates enterprise scale-up ([State of AI trust 2026](https://www.mckinsey.com/capabilities/tech-and-ai/our-insights/tech-forward/state-of-ai-trust-in-2026-shifting-to-the-agentic-era)).

## 5. Governance: registration, inventory, and the register-vs-deploy question

**Hard requirements (evidence).**

- NIST AI RMF GOVERN 1.6 calls for an inventory of AI systems, resourced by risk priority ([NIST AI RMF playbook — Govern](https://airc.nist.gov/airmf-resources/playbook/govern/)); the Cloud Security Alliance's agentic profile of the RMF goes further, recommending an *agent accountability register* per deployed agent: business owner, technical owner, delegation lineage ([CSA agentic NIST AI RMF profile](https://labs.cloudsecurityalliance.org/agentic/agentic-nist-ai-rmf-profile-v1/)).
- EU AI Act transparency obligations take effect **2 August 2026** — one release cycle away from 3.6: people must be informed when interacting with an AI system, and high-risk systems must be registered in an EU database before use, with deployers keeping auto-generated logs ≥6 months ([Article 50](https://artificialintelligenceact.eu/article/50/); [Article 26](https://artificialintelligenceact.eu/article/26/)). (Verification note: a May 2026 "AI Omnibus" provisional agreement reportedly extends only the Article 50(2) machine-readable-marking requirement to 2 Dec 2026 for systems already on market; the August applicability date itself stands.)
- US federal agencies must inventory AI use cases at least annually and publish them ([OMB M-24-10](https://www.whitehouse.gov/wp-content/uploads/2024/03/M-24-10-Advancing-Governance-Innovation-and-Risk-Management-for-Agency-Use-of-Artificial-Intelligence.pdf)).
- The gap is real in practice: only ~21% of agent-adopting organizations have a mature agent governance model, and deployment velocity is outpacing governance ([Gravitee 2026](https://www.gravitee.io/blog/state-of-ai-agent-security-2026-report-when-adoption-outpaces-control)).

**Expectations (inference) — bearing directly on [question-agent-catalog-register-vs-deploy](/features/agent-catalog/knowledge/question-agent-catalog-register-vs-deploy.md).**

- Enterprises will need *some* system of record for deployed agents — regulation makes that non-optional for a growing slice of them. But the evidence supports **registration as an automatic byproduct of deployment, not a gate**: Andrew Ballantyne's objection (GitOps/`oc apply` can't be policed) matches how inventory actually works elsewhere — discovery/reconciliation (the sync-operator ADR seed, ODH ADR #142 direction) rather than a mandatory front door. The CSA register pattern needs owner metadata attached at deploy time — which the catalog deploy flow is uniquely positioned to capture (it knows the template, image version, and requesting user), and raw `oc apply` is not.
- Implication for the Deploy-vs-Register button debate: renaming Deploy to "Register" optimizes for the governance persona at the cost of the ai-engineer's primary job. The requirement is better stated as: *Deploy always registers* (catalog-deployed agents arrive pre-inventoried in the agent registry, 3.6 EA2 backend), and out-of-band deployments get reconciled in. That gives agentops-admins EU-AI-Act/RMF-shaped inventory without policing developer flows.

## 6. Field/enablement: what makes starter kits usable day-zero

**Evidence.**

- Template-driven golden paths demonstrably compress onboarding (55% at Spotify; 75→25h at PagerDuty) *when* they generate a runnable result with CI/CD, secrets handling, and catalog registration baked in ([Backstage/golden path evidence](https://backstage.spotify.com/discover/blog/pagerduty-switches-to-spotify-portal); [Red Hat, designing golden paths](https://www.redhat.com/en/blog/designing-golden-paths)). The design literature is consistent: a golden path must be opinionated, documented, supported, and *fast to first success* — otherwise developers take the dark alleyway ([Forbes on golden paths vs dark alleyways](https://www.forbes.com/sites/adrianbridgwater/2025/06/29/platform-engineering-at-a-crossroads-golden-paths-or-dark-alleyways/)).

**Expectations (inference) — bearing on [question-agent-catalog-quickstarts-overlap](/features/agent-catalog/knowledge/question-agent-catalog-quickstarts-overlap.md).**

- The starter-kit bar that separates them from "glorified quick starts" (Daniele Zonca's worry) is precisely the golden-path checklist: one-command (or one-click) run against platform defaults; a stated support matrix and sizing guidance (CPU/memory per harness pod — cluster-admins will ask before enabling); pre-wired platform binding (model endpoint, MCP gateway, tracing env vars per the Mode 1/Mode 2 contract in [fact-agent-deployment-modes](/features/agent-catalog/knowledge/fact-agent-deployment-modes.md)); and an eval hook ("run this to verify the agent works here") rather than a bare README. Quick starts show *an application*; a starter kit's job is a *working substrate the customer mutates* — the mutation path (swap model, add skill, add MCP server) is the content that must be documented, because it's the first thing every user does.
- Jehlum's bootstrap proposal (harness preconfigured via a platform skill + OpenShift AI MCP server) is the highest-leverage enablement item in the plan: it converts "deployed a harness" into "deployed a harness that can already see the platform" — the six-minutes-to-running-ingress moment ([American Airlines example](https://kodekloud.com/blog/backstage-the-open-source-developer-portal-transforming-how-engineering-teams-ship-software/)) that field teams can demo. If it slips past EA1 (Bill's "really ambitious" flag), the fallback requirement is documented copy-paste binding config per kit — never an undocumented gap.

## Candidate knowledge atoms

- **fact**: Air-gapped/regulated customers prohibit runtime fetching outright — self-contained supported images with disabled self-update are a hard requirement, and stale-but-mirrored beats fresh-but-fetching for this segment; OpenCode needs patch-level work (open upstream issues #16117/#20027) to comply. (resolves direction on the disconnected half of question-agent-catalog-supported-harness-set)
- **fact**: Supported harness images inherit Red Hat's public security machinery automatically — Container Health Index A–F grading, Critical/Important CVE fix policy, Konflux SBOM/SLSA/signing — making image-refresh cadence an externally visible metric and quantifying the maintenance cost behind question-agent-catalog-ownership-packaging.
- **fact**: Claude Code redistribution is prohibited by license (all-rights-reserved, Commercial ToS); Codex CLI and Goose are Apache-2.0 — the viable proprietary-harness pattern is BYOL/Containerfile (customer builds inside their firewall), not Red Hat redistribution.
- **fact**: EU AI Act transparency obligations land August 2026 (one cycle from 3.6) and NIST AI RMF GV-1.6 + CSA's agentic profile expect a per-agent accountability register — supporting "deploy always registers, reconcile the rest" over a mandatory register gate for question-agent-catalog-register-vs-deploy.
- **fact**: Catalog trust is empirically fragile (Port 2025: 3% full trust in portal metadata; 50% doubt) — stale cards or misleading deploy affordances cause permanent routing-around; card accuracy is a requirement, not polish.
- **question**: Does the 3.6 deploy flow capture owner metadata (business + technical owner) at deploy time to seed the agent registry's accountability record, per the CSA agentic-RMF register pattern?
