## 1. Context and Method

The [rhoai-agentic-hub charter](https://github.com/solaius/ai-asset-registry/blob/main/docs/superpowers/specs/2026-07-05-rhoai-agentic-hub-charter.md) requires **in-repo, harness-agnostic memory** — plain, neutral, versioned files that travel with the repo and that any agent can read — and **knowledge registries that point out** to external sources (GDrive, Jira, Slack, upstream repos) rather than copying content in. The charter's research agenda (§8) names OKF as the candidate neutral format to evaluate.

**Method.** This document combines (a) a full first-party close read of `okf/SPEC.md` and the surrounding `GoogleCloudPlatform/knowledge-catalog` repository (raw spec text, complete repo tree, sample bundles, reference tooling); (b) GitHub API data on repo age, contributors, and spec commit history; and (c) a multi-agent deep-research pass (5 search angles, 24 sources fetched, 119 claims extracted, top 25 adversarially verified 3-0 with zero refutations). Claims below are labeled **[verified]** (survived 3-vote adversarial verification or confirmed first-party against the spec/API), **[reported]** (from fetched sources, not independently verified), or **[inference]** (grounded interpretation, flagged as such).

## 2. What OKF Defines

OKF v0.1 is an open format for representing *knowledge* — "the metadata, context, and curated insight that surrounds data and systems" — designed to be "authored by people, generated by agents, exchanged across organizations, and consumed by both" [verified, spec preamble].

**The entire serialization model is a directory tree of UTF-8 markdown files with YAML frontmatter.** No database, SDK, runtime, binary format, manifest, or required tooling. Spec verbatim: *"If you can `cat` a file, you can read OKF; if you can `git clone` a repo, you can ship it."* [verified]

### 2.1 Core constructs

| Construct | Definition |
|---|---|
| **Knowledge Bundle** | A self-contained, hierarchical collection of knowledge documents; the unit of distribution. Distributable as a git repo (recommended), an archive, or **a subdirectory within a larger repository**. |
| **Concept** | One unit of knowledge = one markdown document. May describe a tangible asset (a table, an API) or an abstract idea (a metric, a process, a playbook). |
| **Concept ID** | The file's path within the bundle minus `.md` (e.g., `tables/users.md` → `tables/users`). Identity *is* the filesystem path. |
| **Frontmatter** | YAML block at the top of every concept document. |
| **Link** | Standard markdown link between concepts; expresses relationships beyond the directory hierarchy. |
| **Citation** | Link from a concept to an external source backing a claim. |
| **Reserved files** | `index.md` (directory listing for progressive disclosure) and `log.md` (chronological update history) — the only two reserved names, both optional. |

### 2.2 Frontmatter schema

Exactly six fields are defined; **only `type` is required** [verified]:

```yaml
---
type: <Type name>                  # REQUIRED — e.g. "Playbook", "Metric", "Reference"
title: <display name>              # recommended
description: <one-line summary>    # recommended — used by index generators, search, previews
resource: <canonical URI>          # recommended — the underlying asset the concept describes
tags: [tag, tag]                   # recommended
timestamp: <ISO 8601>              # recommended — last meaningful change
---
```

Type values are **not registered centrally**; producers pick descriptive values and consumers MUST tolerate unknown types gracefully [verified]. Producers MAY add arbitrary extra keys; consumers SHOULD preserve unknown keys and SHOULD NOT reject documents carrying them [verified]. The design convention: *frontmatter for the few fields you want to query, filter, or index on; markdown body for the prose, schemas, and examples that LLMs and humans actually read* [verified, README].

### 2.3 Structural conventions

- **Cross-linking:** bundle-root-absolute links (`/tables/customers.md`) recommended over relative links; link *kind* is conveyed by surrounding prose, not link syntax. Consumers MUST tolerate broken links — a dangling link "is not malformed; it may simply represent not-yet-written knowledge" [verified]. (Notably, this is the same semantic as the `[[name]]` convention in Claude Code auto-memory — see [research 18 §2](18-filesystem-knowledge-and-project-memory.md).)
- **`index.md`:** optional per-directory listing (`* [Title](url) - description` entries grouped under headings) supporting **progressive disclosure** — an agent or human sees what is available before opening documents. May be generated automatically or synthesized on the fly.
- **`log.md`:** optional per-scope chronological history, newest first, ISO-8601 date headings, entries prefixed by a bold convention word (`**Update**`, `**Creation**`, `**Deprecation**`).
- **Citations:** numbered list under a `# Citations` heading; links MAY be absolute URLs, bundle-relative paths, or paths into an optional `references/` subdirectory that mirrors external material as first-class concepts.
- **Conventional body headings:** `# Schema`, `# Examples`, `# Citations` have suggested meaning; nothing is required.

### 2.4 Conformance and versioning

Conformance is deliberately minimal — exactly three hard criteria [verified]:

1. Every non-reserved `.md` file has parseable YAML frontmatter.
2. Every frontmatter block has a non-empty `type`.
3. Reserved files follow their described structure when present.

Consumers MUST NOT reject bundles for missing optional fields, unknown types, unknown keys, broken links, or missing indexes. The spec calls this permissive model intentional: OKF must stay useful "as bundles grow, get refactored, and are partially generated by agents." Versioning is `<major>.<minor>`; a bundle MAY declare `okf_version: "0.1"` in root `index.md` frontmatter (the only place index frontmatter is allowed).

## 3. The Surrounding Ecosystem

The `knowledge-catalog` repo (262 files) frames OKF within Google Cloud's Knowledge Catalog product (formerly Dataplex) but ships the format with a small proof-of-concept ecosystem demonstrating both ends of the producer/consumer contract [first-party repo inspection]:

| Component | Role | Notes |
|---|---|---|
| `okf/SPEC.md` | The format | v0.1 Draft |
| **Reference agent** (`okf/src/reference_agent/`) | *Producer* | Python/Gemini agent; two-pass enrichment (BigQuery metadata pass, then an LLM web-crawl pass with seed URLs, same-domain filters, and page caps) writing OKF concept docs |
| **Sample bundles** (`okf/bundles/`) | Evidence | Three published bundles (GA4, Stack Overflow, Bitcoin) produced by the agent |
| **Visualizer** (`visualize` subcommand) | *Consumer* | Renders any bundle as a self-contained interactive HTML file — force-directed concept graph, per-concept detail panel, backlinks ("Cited by"), search, type filter |
| **`mdcode` toolbox** (`toolbox/mdcode/`) | *Sync* | Push/pull synchronization between OKF markdown and live systems (BigQuery, Dataplex entry groups, knowledge bases) — the "metadata as code" workflow |
| **Enrichment samples** (`samples/`, `toolbox/enrichment/`) | Patterns | Includes agent skills (`SKILL.md`) for knowledge-base search and enrichment pipelines |

Two ecosystem observations relevant to the hub design [inference]:

1. **The visualizer is prior art for the knowledge-hub pattern** — a zero-backend HTML artifact generated from a markdown corpus, directly analogous to the GitHub-Pages HTML hubs this repo already publishes.
2. **`mdcode`'s push/pull model is prior art for registry freshness** — treating external systems as the source of truth and the markdown bundle as a synchronized projection, rather than a copy that drifts.

## 5. Strengths and Gaps

### 5.1 Strengths (for the charter's purposes)

| Strength | Why it matters here |
|---|---|
| Plain markdown + YAML frontmatter, git-native | Structurally identical to what this repo already does; zero migration format risk [verified] |
| Explicitly vendor/harness-neutral | "Not tied to any particular agent, framework, model provider, or serving system" — the charter's harness-agnostic requirement, verbatim [verified; stewardship caveat: single-vendor authorship] |
| One required field | The lowest possible standardization tax; everything else is convention |
| Legal extensibility | Any richer memory/registry schema is a conforming producer extension, not a fork [verified] |
| `resource` URI + external citations + tolerated broken links | Pointer-out architecture is native, not bolted on (see §7) [verified] |
| `index.md` progressive disclosure | Matches the proven index-plus-lazy-load pattern from Claude Code and Letta MemFS ([research 18 §6](18-filesystem-knowledge-and-project-memory.md)) |
| `log.md` chronological capture | A reserved, spec-blessed home for capture-as-you-go history |
| Permissive consumption | Bundles survive refactors and partial agent generation without breaking consumers |
| Bundle = subdirectory | Multiple registries can live as multiple bundles inside one repo [verified, spec §3] |

### 5.2 Gaps

| Gap | Impact | Mitigation |
|---|---|---|
| **No memory semantics.** OKF is a knowledge interchange format; it has no notion of memory types (episodic/semantic/procedural), scopes, consolidation, or decay | Charter pillar C (dogfood the RHOAI memory taxonomy) gets no help from the spec | Acceptable now — pillar C is deferred; type values and extension keys leave the door open [inference] |
| **Single `timestamp` field — no temporal validity model.** Cannot natively express "valid from / superseded on / superseded by" — the exact mechanism needed to keep roadmap dates and strategy facts current | The charter's problem 3 (memory that survives change) needs more than OKF gives | Local frontmatter extension (legal per spec) approximating Zep's bi-temporal fields, plus `log.md` and git history — see [research 18 §6.4](18-filesystem-knowledge-and-project-memory.md) |
| **`resource` URI semantics underspecified** for reliable programmatic resolution (what makes a URI canonical? how to dereference GDrive/Jira/Slack?) [reported, okf.md] | Registry automation (staleness checks, link sweeps) must define its own URI conventions | Document per-source URI conventions locally |
| **Unregistered type values** — interop depends on producers converging by discipline | Cross-team exchange (the future rhoai-atlas scenario) risks type drift | Publish a local type vocabulary as part of the hub's conventions |
| **No per-claim provenance/confidence model** beyond citations | This research series' own Status/confidence conventions are richer than the spec's | Extension keys |
| **v0.1 single-vendor draft; churn risk** | Field names/conformance could change under future governance | Adopt as *conventions*, pin "OKF v0.1 as of 2026-07-05", track the repo and the W3C CG |

## 6. Fit Evaluation A: In-Repo Project Memory Store

Mapping the charter's memory requirements onto OKF affordances:

| Charter requirement | OKF affordance | Verdict |
|---|---|---|
| In-repo, versioned, travels with the repo | Bundle = directory (or subdirectory); git recommended distribution | **Native fit** [verified] |
| Harness-agnostic — any agent can read it | Markdown + frontmatter; no SDK; permissive consumption | **Native fit** [verified] |
| Capture-as-you-go (goal B) | `log.md` reserved for chronological capture; concepts minted per fact/decision; `timestamp` on change | **Good fit**, with the *workflow* (who writes, when) left to the hub's skills — OKF standardizes the artifact, not the behavior |
| Cross-session continuity (goal A) | `index.md` progressive disclosure gives a session a cheap "what do we know" entry point; descriptions in frontmatter feed the index | **Good fit**; matches the MEMORY.md-index and MemFS-tree patterns exactly ([research 18 §6.1](18-filesystem-knowledge-and-project-memory.md)) |
| Facts that change stay current (problem 3) | Only `timestamp` | **Gap — requires local extension** (validity/supersession keys; see §5.2) |
| Future RHOAI memory taxonomy alignment (goal C, deferred) | `type` field + free extension keys | **Extensible**; no conflict, no help [inference] |

**Verdict: strong structural fit with one real gap (temporal validity), which the spec explicitly permits closing via extension keys.** An in-repo memory store can be a conformant OKF bundle whose concept types are memory-flavored (e.g., `Decision`, `Fact`, `Preference`, `Status`) and whose extension keys carry validity metadata. Nothing about that choice locks the hub to Google, to any harness, or to OKF's survival — if OKF dies, the files remain self-describing markdown [inference].

## 7. Fit Evaluation B: Decomposed Knowledge Registries

The charter requires registries that **point out** to external sources rather than copying content in — the pattern `docs/knowledge-registry.md` already uses informally. OKF makes it formal [verified]:

- **`resource:`** carries the canonical URI of the underlying asset — the spec's own example is a console URL to an externally-owned system (a BigQuery table). A registry entry for a GDrive doc, Jira epic, Slack channel, or upstream repo is structurally identical: a concept doc whose `resource` is the external URI and whose body holds the *curated insight about it* — exactly "registry governs metadata and context; asset lives elsewhere," the metadata-first principle this project already applies to AI asset registries ([knowledge-registry §1](https://github.com/solaius/ai-asset-registry/blob/main/docs/knowledge-registry.md)).
- **Citations MAY be absolute URLs** — external evidence stays external. The `references/` mirror option exists but is copy-in; the hub should treat point-out as its default and reserve mirroring for sources at deletion risk [inference].
- **Tolerated broken links** mean a registry can reference not-yet-written registry entries without breaking consumers — decomposition can proceed incrementally.
- **Bundle-as-subdirectory + domain-independent structure** means the monolithic `knowledge-registry.md` (1,105 lines, 13 topical sections as of 2026-07-05) can decompose into one or several bundles along whatever axis the design phase chooses — OKF deliberately does not prescribe the axis ("The directory structure is independent of the domain") [verified]. Evidence on choosing that axis is in [research 18 §7](18-filesystem-knowledge-and-project-memory.md).

Caveats: `resource` is optional, so the pointer-out discipline must be enforced by the hub's own conventions and maintenance skills, not by conformance; and the GDrive/Jira/Slack mapping is grounded inference from verified mechanics — the spec's examples are data-catalog assets, and no published OKF bundle yet demonstrates a PM-knowledge registry [inference, flagged by verifiers].

**Verdict: native fit.** The pointer-out registry is arguably *the* OKF-shaped use case — closer to the spec's motivating scenario (curated context around externally-owned assets) than the memory store is.

## 8. Relationship to the Agent-Memory Research Series

**Updates [research 05](05-standards-and-protocols.md).** Doc 05 (2026-05-17) found no mature standard for agent memory and identified MCP as the de facto *transport*. OKF (published 2026-06-12) does not change the memory-standard verdict — it has no memory semantics (§5.2) — but it is the first *specified* convention for the **file-based knowledge substrate**: the at-rest, git-native layer that AGENTS.md-style conventions gesture at but never specified. The complementary split is: **MCP = transport in motion; OKF = knowledge at rest** [inference]. Doc 05's gap analysis ("the standards opportunity for Red Hat") should be re-read with OKF on the map: the knowledge-interchange layer now has a Google-authored draft, which raises the cost of a Red Hat-originated alternative and raises the value of early engagement (the W3C Holon CG is the venue to watch).

**Feeds the AI Asset Registry.** The registry framework treats **Knowledge Sources** as a first-class asset type with backend TBD ([knowledge-registry §2](https://github.com/solaius/ai-asset-registry/blob/main/docs/knowledge-registry.md)). OKF is now a candidate answer for that asset type's *interchange format* — how a governed knowledge source is packaged, versioned, and exchanged — and its bundle model (metadata-first, pointer-out, git-distributed) matches the registry's own metadata-first principle. Worth an explicit evaluation when the knowledge-sources asset work starts [inference].

**Connects to harness findings.** OKF independently converges on the same architecture that Claude Code auto-memory and Letta MemFS arrived at from the memory side: an index for progressive disclosure over lazily-loaded, frontmatter-described markdown files. That three-way convergence — a knowledge-interchange spec, a commercial harness, and a memory-platform vendor — is the strongest available signal that the pattern is sound. Details in [research 18](18-filesystem-knowledge-and-project-memory.md).

## 9. Recommendation

**PROPOSED — Adopt OKF v0.1 as a lightweight convention layer for rhoai-agentic-hub, with documented local extensions. Do not treat it as a settled standard.**

Concretely:

1. **Structure hub registries and the memory store as conformant OKF bundles** — frontmatter with `type` everywhere, `description` everywhere (it feeds indexes), `resource` on every pointer-out entry, `index.md` at every level, `log.md` where chronology matters. The cost is near zero: this formalizes conventions the repo already half-uses.
2. **Define and document local extension keys** for what OKF lacks — temporal validity (`valid_from`, `superseded_by`, `status`) and any provenance/confidence fields — in a single conventions doc that rhoai-atlas can later inherit. All legal per spec §4.1.
3. **Publish a local type vocabulary** (the hub's `type` values and their meanings) to pre-empt type drift when the pattern generalizes to other PMs.
4. **Pin the dependency**: "OKF v0.1 as of 2026-07-05." Watch the repo and the W3C Holon CG; re-verify before rhoai-atlas hard-commits other teams to the format.
5. **Do not build tooling that only works if OKF wins.** The insurance policy is inherent: a bundle is self-describing markdown, valuable even if the spec is abandoned.

The asymmetry drives the recommendation: adopting OKF conventions costs almost nothing over the markdown-plus-frontmatter approach the hub would use anyway, buys spec-blessed answers to questions the design phase would otherwise settle ad hoc (reserved files, link semantics, index format, conformance floor), and positions both the hub and the RHOAI knowledge-sources asset work to interoperate if adoption materializes — while the downside (spec churn or abandonment) strands nothing.

## 10. Open Questions

1. **OKF trajectory** — Will the W3C Holon CG become real neutral governance? Will non-Google producers/consumers adopt v0.1? Will v0.2+ preserve backward compatibility? (Determines how much rework risk rhoai-atlas inherits.)
2. **Temporal validity design** — Is a frontmatter extension (`valid_from` / `superseded_by` / `status`) plus git history plus `log.md` sufficient to keep roadmap dates and strategy facts current, or does the hub need a first-class invalidation log? (Candidate mechanisms in [research 18 §6.4](18-filesystem-knowledge-and-project-memory.md).)
3. **URI conventions for pointer-out sources** — What are the canonical `resource` URI forms for GDrive docs, Jira issues, Slack channels/threads, and upstream repos, and which of the hub's maintenance skills validate them?
4. **Knowledge Sources asset type** — Should the AI Asset Registry proposal name OKF as a candidate packaging/interchange format for the Knowledge Sources asset type, and is there an upstream (MLflow/Kubeflow) angle?
5. **Nested vs. sibling bundles** — One bundle per registry, or one bundle with registry subdirectories? (A design-phase decision; spec permits both.)
