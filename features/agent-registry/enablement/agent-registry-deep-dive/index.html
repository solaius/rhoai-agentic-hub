WHY A REGISTRY

You can't govern what you can't see

Agents aren't like models. A model sits in a serving endpoint you configured; an agent is a running service — it holds credentials, calls tools, and can be deployed by anyone with cluster access. Three forces make a system of record non-optional:

THE SHADOW FLEET
Agents nobody recorded

Industry discovery data: 82% of organizations that ran agent discovery found agents they didn't know about, and 67% of non-human identities were created outside IAM visibility. Concretely: a teammate oc apply's an agent from a Git repo on Friday. It works. It has a service account. Six months later nobody remembers it exists — but it still holds its credentials.

THE AUDIT LETTER
Regulation already wrote the schema

EU AI Act (applies Aug 2026), US OMB guidance, and NIST/CSA governance frameworks independently converge on the same eight-field agent record: identity+version, owner, purpose, risk class, lineage, deployment status, log references, permissions. The "approved vs running" split they assume maps 1:1 onto the registry's two entities. When the audit letter arrives, this is the query it asks.

THE TRUST HOLE
Nothing verifies agents today

Every agent advertises its own capabilities via its agent card (a JSON manifest served at /.well-known/agent-card.json, defined by the A2A protocol). Today no component on the platform fetches or cryptographically verifies those cards — the schema's verified field has zero producers. Every card is taken at its word. That's the single most urgent gap in this feature.

RUNNING EXAMPLE — MEET TRIAGE-BOT

Throughout this page we'll follow one agent: triage-bot, a support-ticket triage agent built from the LangGraph starter kit. One version was reviewed and approved. Three copies run in production across two clusters. A fourth copy — an experiment someone GitOps'd — runs in a dev namespace nobody governs. A good registry must represent all four truthfully.

THE DATA MODEL

Two entities, because one lie is worse than two truths

The core design decision: a single "Agent" record cannot be both approved once and running three times. So the registry keeps two entities and an explicit join between them.

THE FOURTH COPY
Unlinked instance (version_ref = null)

Triage-bot's GitOps'd experiment matches no approved version — so it's registered as an unlinked instance: visible to governance as shadow inventory instead of invisible. This one nullable field is what turns "you can't police oc apply" from an objection into a feature: the registry doesn't block the side door, it lights it.

How a discovered instance finds its version — the join-key ladder

Three keys, in strength order. The platform path uses the strongest; BYO agents fall through to the weakest.

STRONGEST Stamped identity — the deploy path writes the registry's version ID as an annotation on the workload it creates. Reconciliation becomes a lookup. Costs one line in the deploy flow — and because deployment ships before registry work even starts, stamping from day one is what lets records join retroactively once the registry exists.
FALLBACK Image digest + config hash — digest alone isn't enough: two Mode-2 agents can share the same harness image and differ only in configuration. The composite narrows to a unique version.
WEAKEST Card identity claims — the agent card's name/provider/version. Self-asserted unless the card is signed, but it's the only key a fully out-of-band agent offers.
TRIAGE-BOT, JOINED

The three production copies were deployed through the platform, so their workloads carry stamped annotations — instant, unambiguous join to triage-bot v1.2. The GitOps copy has no annotation; its image digest matches the LangGraph harness (shared by many agents), and its config hash matches nothing approved — so it stays unlinked, and shows up in the shadow-inventory view exactly as it should.

DISCOVERY

The split: knowing an agent exists ≠ knowing what it is

The registry's discovery layer was rebuilt in July 2026. The previous runtime component bundled two jobs in one CRD: it enumerated agents and enriched them (fetched their cards, verified signatures). On today's cluster those jobs are split across different sources — and nobody currently does the second one.

Why not build discovery on the OpenShell gateway? Coupling the registry to a single runtime is exactly the mistake that just cost us a rewrite. Sandbox CRs are the only runtime-agnostic signal on the cluster; the gateway is reserved as a future health/policy enrichment source.

WHAT THE OLD COMPONENT DID THAT NOTHING DOES NOW

Three capabilities were lost with the runtime pivot and have no current equivalent: (1) automatic card fetch and caching, (2) JWS verification bound to workload identity — the whole trust chain behind verified=true, and (3) label-based discovery of BYO agents running as plain Deployments (today they're simply invisible). Each is now a registry-side responsibility — see Your Impact.

Naming note, worth getting right in public docs: Kagenti is off the RHOAI roadmap — Red Hat's runtime bet is OpenShell. The upstream Kagenti project continues independently (rebranding to Rosso); its design concepts inform OpenShell contributions. Two separate facts; don't merge them.

THE ARCHITECTURE

Register on deploy; reconcile the rest

One flow, two doors. Agents deployed through the platform get registered synchronously, with full metadata. Agents that arrive any other way get detected and adopted as unlinked instances. Governance sees a complete inventory either way. One boundary to keep straight: deployment is its own workstream (OpenShell, the Go SDK, the catalog's deploy surface) — the registry doesn't own it; it consumes its events. This diagram shows the target state once both exist.

registration path deploy path enrichment loop out-of-band watch / reconcile
User deploys from the catalog detail page; the BFF resolves the declared configuration (model, MCP servers, skills) against the platform registries.
The BFF calls the OpenShell Go SDK — and stamps the registry's version ID as annotations on the Sandbox CR it creates. One line of code; it makes reconciliation a lookup instead of a guess.
The BFF registers synchronously: confirms the AgentVersion, creates the instance record with version_ref set. The deploy gate refuses versions that aren't Published + Approved — enforcement at the one door the platform controls.
The sync controller watches Sandbox CRs cluster-wide: annotated CRs get confirmed/healed; unannotated CRs (GitOps) become unlinked instances. Same pattern as the MCP registry's sync controller — one mechanism family, two asset types.
CR state changes push to the registry's webhook: pod gone → instance goes STALE, then REMOVED after grace.
An enrichment loop fetches each instance's agent card over mutually-authenticated TLS, verifies the JWS signature, and fills skills/protocol/verified. This loop doesn't exist yet — it's the most urgent open item (see Your Impact).
PLATFORM PATH GUARANTEES

Registration — synchronous, rich, approval-gated. Every catalog deploy produces a complete, owner-attributed, provenance-linked record at birth.

OUT-OF-BAND PATH GUARANTEES

Detection only — reconciled within a watch interval, thin metadata, unlinked. The reasoning: you genuinely cannot police oc apply, so don't pretend to. Compliance needs the inventory to be complete, not every record to be born through the front door.

LIFECYCLES

Two state machines that don't map — on purpose

Governance answers "should this exist?" Runtime answers "does it, and is it well?" Those are different questions with different clocks, so the registry keeps them as independent machines and connects them at four join points instead of forcing a mapping.

ON THE AGENTVERSION — 4 GOVERNANCE TRACKS

Independent tracks, borrowed from the MCP Registry's proven model:

Lifecycle: Draft → Candidate → Published → Deprecated → Retired
Approval: Draft → Pending → Approved → Rejected/Revoked
Verification: Unverified → Verified
Certification: None → Candidate → Certified → Expired
ON THE INSTANCE — RUNTIME STATES

Live-service facts, driven by discovery:

ACTIVE UNHEALTHY STALE — source stopped reporting REMOVED — kept ≥ 6 months for audit

SUSPENDED — missing
Sandbox CRs support operatingMode: Suspended — deliberately paused. That's not ACTIVE, not UNHEALTHY (nothing is wrong), not STALE (still reported). The state model needs a fifth state — a concrete, ready-to-file upstream edit.

The four join points

1 · DEPLOY GATE

Governance → runtime, at the front door: the platform deploy path refuses any version that isn't Published and Approved. The only enforcement point the platform actually controls — so it's the only one the design uses.

2 · IMPACT SURFACING

Deprecating triage-bot v1.2 doesn't kill its three ACTIVE instances — it flags them. A signal to owners, never a kill switch. (Killing running workloads from a metadata store is how you lose the trust of every platform engineer.)

3 · RETIRE SAFETY

Runtime → governance: a version with zero non-REMOVED instances is provably safe to retire. The instance ledger finally gives governance real evidence for the Retired transition — something a deployed-yes/no boolean can never provide.

4 · A NAMING HAZARD

The governance Verification track (a human curation judgment on the record) and the runtime verified field (a cryptographic property of a fetched card) are different facts sharing one word. Rename one — e.g. verification_status vs card_attested — before the backend schema freezes names.

SUPPLY CHAIN

The one question only the registry can answer

"A CVE just landed in the UBI base image — which running agents are affected?" The image registry knows digests but not agents. The cluster knows pods but not versions. The agent registry is the only component that can walk the whole chain.

THE BLAST-RADIUS QUERY, CONCRETELY

CVE lands in the UBI base → resolve which harness digests build on it → find AgentVersions pinning those digests → list their non-REMOVED instances. Answer: "triage-bot's three production copies, plus two other agents — here are the owners." Minutes, not a spreadsheet archaeology week.

One subtlety that matters: automated CVE patching rebuilds images, producing new digests for the "same" harness. If a version pins a digest, every patch forces a governance re-approval (churn); if it pins a tag, the blast-radius query breaks. The design threads the needle: version identity = image line + declared config (stable across patch rebuilds), while each instance records the exact digest it runs (preserving the audit query). Base images themselves are not registry entities — they're supply-chain metadata on versions; Konflux and Quay already govern the images themselves.

UPSTREAM & STRATEGY

Build upstream, own the join, hold the wedge

The April 2026 thesis was "the MLflow registry namespace is empty — move first." That's over: partly because we moved (the upstream RFC stream is now Red Hat-authored), partly because the market GA'd around us. The play now has three parts.

1 · SHAPE THE UPSTREAM
RFC-0008: skills first, agents next

The MLflow registry stream is live as RFC-0008 (MVP Skill Registry, Phase 1 — Red Hat-authored, in draft review). Agent-shaped entities are explicitly Phase 2. The window: get the dual-entity model, the SUSPENDED state, and the card-verification fields into Phase 2 before Phase 1 review concludes. Bonus: the downstream backend and the upstream RFC share an author — the alignment channel already exists.

2 · OWN THE UNOWNED JOIN
Enumeration without enrichment

Nobody — upstream or downstream — currently owns the join between "a workload exists" and "it's a trustworthy agent." The dormant post-deployment proposal (the AgentDiscoveryProvider interface survives; its reference plugin described a dead substrate) gets re-baselined onto Sandbox CRs as Phase 2's runtime companion. This is the differentiated engineering.

3 · HOLD THE WEDGE
Self-managed, disconnected, governed

Nobody ships a self-managed, disconnected-capable, governed fleet registry with lineage — and regulation now effectively mandates the record shape we'd govern. But the window is time-boxed: IBM will eventually port its control plane on-prem, and an OSS registry project is under CNCF Sandbox review this September.

The market, in one strip

Microsoft

Inventory is free; governance is $15/user/mo (GA May 2026). Sets the floor: you can't charge for the list, only for what you do with it.

AWS

Registry in preview; record-centric with approval workflow — and its endpoint-ingestion validates our card-fetch design. Watch GA pricing.

IBM

"Agentic Control Plane" positioning — SaaS-only today. The on-prem window is ours until they port it.

OSS / federation

Solo.io's agentregistry is in CNCF Sandbox review (Sept); the ARD spec (Google+Microsoft) formalizes federated discovery — thin adoption, cheap to hedge via URN-mappable IDs.

Why MLflow at all? Reasoning, not reflex: MLflow is already on the platform (stateless service, PostgreSQL, workspace-scoped RBAC — so versions inherit access control for free), it already has the version-side entity (LoggedModel links a version to git commit, traces, and evals), and Databricks just made agents first-class in Unity Catalog while open-sourcing gateway pieces into MLflow — meaning upstream MLflow is becoming a governance surface with momentum we can ride. The trade-off is honest: Databricks steers the substrate. That's what the RFC channel is for.

WHERE YOU CAN HAVE IMPACT

The open decisions — cheap now, breaking later

This feature is at the moment where one engineer's design input changes the schema everyone lives with. Ordered by urgency:

MOST URGENT
Rebuild the path to verified=true

The card-verification work item closed in mid-July with no successor, while its approved RFE clone still sits open. Result: zero active work items produce agent trust. The design is ready (registry-side card fetch + JWS verify over mTLS); it needs a re-filed home and an owner.

BEFORE BACKEND WORK STARTS (3.6 EA2 EARLIEST)
Fix the backend schema while it's cheap

The tracked backend (work starting 3.6 EA2 at the earliest) is currently drafted single-entity with one lifecycle — no runtime instance states, no SUSPENDED, no version_ref, no risk-tier/log-refs/retention fields, and a sync epic still aimed at the removed runtime. Every one of these is a cheap schema addition today and a breaking migration after the freeze.

PRODUCT CALL NEEDED
What feeds the first registry view?

Registry backend work starts at 3.6 EA2 at the earliest, with DP not expected before ~3.7 — so any registry-view surface shipped before then would predate its own backend. If a stopgap feeds it from Sandbox CRs, it becomes a second deployments view and the registry/deployments distinction collapses on arrival. Feed it from deploy-time registrations (even file-backed), or label it honestly as a preview — or simply don't ship one early.

UNTRACKED REGRESSION
BYO agents are invisible

The old label convention for discovering bring-your-own agents (plain Deployments, no sandbox) lost its owner in the runtime pivot. Sandbox-CR discovery only sees sandbox-shaped workloads. Someone needs to own a new label/CR convention — otherwise Mode-1 agents never reach the inventory at all.

IMPLEMENT-AND-INFLUENCE WINDOW
ARD federation endpoint at TP?

The Google+Microsoft federated-discovery spec (/.well-known/ai-catalog.json) is v0.9 with thin adoption — exactly when implementing it buys influence. Cheap hedge either way: URN-mappable IDs and native-media-type card storage.

BOUNDARY TO DEFEND
The registry is the lookup, not the enforcement

Fleet evidence demands seconds-level agent revocation — which must live in the identity/gateway plane, never behind a registry sync. The registry's job: make the identity linkage one query away. Resist every design that puts the registry in a runtime-critical path; that's how it becomes the outage.

Also open, lower heat: whether the discovery-plugin interface generalizes beyond agents; multi-agent composition graphs (compliance-adjacent via delegation-lineage expectations, still deferred); and the namespace↔workspace mapping for reconciled instances — the MCP sync controller never had to solve it because its deployments are always platform-initiated.

ROADMAP & RESOURCES

Where this lands, release by release

Three separate workstreams, easy to conflate: the catalog (discovery surface), deployment (OpenShell + the deploy path — its own effort, which catalog and registry both consume), and the registry — the latest-starting of the three.
