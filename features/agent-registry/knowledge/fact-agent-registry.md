Draft upstream proposal (Varsha Prasad Narsing, 2026-02-16; Peter Double now taking over) for a post-deployment-only registry: agents as live services with network endpoints, not pre-deployment artifacts (that's a separate, deferred proposal). Governs identity, URL/endpoint, protocol (openai-compatible/a2a/custom), skills, health status, trust verification (SPIFFE/SPIRE), and lifecycle state (ACTIVE → UNHEALTHY → STALE → REMOVED — simpler than MCP Registry's multi-track governance since it's runtime-focused).

Architecture: pluggable discovery providers via `mlflow.agent_discovery` entry points (poll/watch/webhook sync), with reference plugins for Kubernetes (via kagenti AgentCard CRD), Docker, Consul, and Static. The Kubernetes/kagenti plugin doesn't re-fetch agent cards — kagenti has already done fetching, JWS verification, and SPIFFE binding; three integration options are on the table: Pull (MLflow watches the CRDs), Push (kagenti POSTs to a webhook, extending kagenti's existing `MLflowReconciler`), or Hybrid (push plus poll fallback). Trust verification precisely: `verified` = both JWS signature valid AND SPIFFE identity bound; `identity` = the SPIFFE ID from the leaf cert's SAN URI; `trust_domain` = the organizational boundary used for filtering; the Gateway prefers routing to `verified=true` agents.

Optional auto-sync into the Gateway as endpoints (opt-in via `MLFLOW_AGENT_GATEWAY_AUTO_SYNC=true`) for tracing/routing/fallback — REMOVED agents become deactivated endpoints. Python client: `mlflow.agents.list_agents()`, `get_agent()`, `discover()`, `invoke()`. Key distinction from MCP Registry: this governs live services with runtime state, not static assets with multi-track governance — complementary, not overlapping.

**What the proposal does NOT cover** (gaps for Red Hat to address): pre-deployment agent registry (versioning agent definitions/images — a separate follow-up); multi-track governance (approval/certification/verification tracks like MCP Registry); agent-to-agent relationships or composition graphs; workspace-scoped RBAC; evaluation integration; cost tracking/chargeback.

**Implications for Red Hat**: this is a starting point, not a finished design — Peter is developing it into a full RFC with requirements, user stories, and governance extensions. It complements rather than competes with MCP Registry (MCP governs static assets, this governs live services — same framework, different lifecycle models). The kagenti push-model integration is a natural fit for RHOAI deployments where kagenti and MLflow coexist. The governance gap (runtime lifecycle exists; no approval/certification/trust-tier layer) is Red Hat's opportunity to add value — the same pattern seen in the Skills Registry MVP (see [fact-skills-registry-mvp-analysis.md](/features/skills-registry/knowledge/fact-skills-registry-mvp-analysis.md)). Open reconciliation question: agent-as-asset (governance-focused) vs. agent-as-service (runtime-focused) — both views are needed.
