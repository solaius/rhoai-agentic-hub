# Agent Registry research — registry & fleet-management landscape refresh

Research date: 2026-07-16. Scope: the **registry / fleet-governance layer only** — registries of running/registered agents, agent inventory, agent identity, approval and audit. The catalog/storefront layer is covered by the sibling agent-catalog series ([01-landscape](/features/agent-catalog/research/01-landscape.md), [05-competitive](/features/agent-catalog/research/05-competitive.md)) and is deliberately not repeated here. This document refreshes the April 2026 claims in [04-agent-management-landscape](/features/agent-registry/research/04-agent-management-landscape.md) and [01-agent-ecosystem](/features/agent-registry/research/01-agent-ecosystem.md).

**Bottom line:** In the twelve weeks since the April series, the registry layer went from "AWS just launched the first one" to a crowded, shipping, partially-federated market. Microsoft closed its "notable gap" spectacularly — Agent 365 GA'd May 1 as a full agent control plane, and the *inventory* part of it is reportedly free, with the $15/user/month charge attached to the governance/security stack around it. Google shipped a documented Agent Registry product inside the Gemini Enterprise Agent Platform. IBM renamed its whole strategy "the Agentic Control Plane" — but launched it SaaS-first on AWS and IBM Cloud, leaving the self-managed on-prem control-plane slot open for now. Salesforce's Agent Fabric registry with cross-cloud Agent Scanners went GA in January. Meanwhile AWS — April's first mover — is *still in preview*. Two structural shifts matter most for RHOAI: (1) the April claim that "no open-source platform has a dedicated agent registry" is no longer safely true (Solo.io's `agentregistry` is before the CNCF Sandbox board; AGNTCY's Agent Directory is federating registries under the Linux Foundation), and (2) monetization is migrating *off* the registry inventory itself and *onto* the fleet-governance capabilities around it — which means an MLflow-based registry is necessary but not sufficient; the wedge is the registry *plus* governance (gateway, identity, policy) as one self-managed, air-gappable stack.

## AWS — first mover, still in preview

The April series treated the AWS Agent Registry (preview 2026-04-09) as the pace-setter. Three months on, the honest read is that AWS validated the category and then let others GA past it:

- **Still preview.** The [AgentCore release notes](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/release-notes.html) (checked 2026-07-16) show no Agent Registry GA between April and July 2026, and the [registry documentation](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/registry.html) still carries the "(Preview)" designation. Feature set is as the April series described it: registries + records for MCP servers, agents, skills, tools, and custom resources; hybrid (semantic + keyword) search; the publish → pending-approval → approved/rejected → deprecated curation workflow with Admin/Publisher/Curator/Consumer personas; the registry exposed as a **remote MCP endpoint** queryable by any MCP client; IAM or corporate-IdP JWT authorization; EventBridge notifications and CloudTrail audit. A "Migration from public preview FAQ" section in the docs signals GA preparation.
- **Pricing signal.** Community analysis of the preview reports the GA pricing model will be per **"Net Records"** — records alive at any moment, with deletions decrementing the count ([Gerardo Arroyo — AWS Agent Registry: a private catalog to stop agent sprawl](https://gerardo.dev/en/aws-agent-registry.html), 2026). Treat as reported-not-official, but it is the first per-inventory-record meter proposed in the market.
- **The fleet-ops motion moved elsewhere in AgentCore.** What did GA between April and July is the operational loop around registered agents: the managed **AgentCore harness** (GA June 2026), **Recommendations, Batch Evaluations, and A/B testing** (GA June 2026) — explicitly working "regardless of where agents run: on AgentCore runtime, AWS Lambda, Amazon EKS, or non-AWS environments" — plus cross-account observability, Bedrock Guardrails enforcement at the gateway layer ("outside the agent's code, where the agent cannot reason around them"), SOC 1/2/3 compliance, and **AgentCore GA in AWS GovCloud (US-West)** (May 2026) ([release notes](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/release-notes.html)). The GovCloud GA matters for us: the "regulated workloads need on-prem" assumption erodes the same way Google's air-gapped GDC eroded it — sovereign-adjacent clouds are absorbing part of the compliance market.
- April-roadmap items — automatic indexing on deployment, cross-registry federation, operational data folded into registry records — **have not shipped** per the release notes; they remain roadmap.

## Microsoft — the "notable gap" closed, and it redefined the category

The April series' single most-wrong-by-July claim is doc 04's "Microsoft does not offer a dedicated agent registry product." As of **May 1, 2026, Microsoft Agent 365 is GA** — and it is now the reference implementation of the *fleet-governance* registry:

- **What it is.** A control plane/registry for all agents in a tenant "regardless of where they were built" — Copilot agents, Foundry agents, third-party — with each agent receiving a **Microsoft Entra Agent ID** (a real directory object with governance policies and audit trails), governed through Defender (threat detection, conditional access), Purview (DLP, communication compliance, retention), and Intune-style controls ([Microsoft Security blog — Agent 365 now generally available](https://www.microsoft.com/en-us/security/blog/2026/05/01/microsoft-agent-365-now-generally-available-expands-capabilities-and-integrations/), 2026-05-01).
- **Pricing — and the nuance the "registries are monetized" framing misses.** The headline is $15/user/month (one user license covers all agents that user owns/uses/sponsors; bundled in M365 E7 at $99/user/mo) ([SAMexpert — Agent 365 licensing](https://samexpert.com/agent-365/)). But licensing analyses report that the **core registry functions require no license**: agent identity, the agent registry itself, visibility with creator/timestamp data, and admin actions including publish, block, delete, and approve — plus shadow-AI detection — sit in the free tier, while the $15 buys the surrounding security/governance/observability stack; autonomous (non-user-attached) agents remain in preview and uncharged ([Cloudpartner — Agent 365 GA: what it governs, what it costs](https://learn.cloudpartner.fi/posts/agent365-ga-governance-licensing), May 2026). Read: Microsoft is **commoditizing the inventory to sell the control plane**.
- **Consolidation.** The standalone Agent Registry and Agent Collections blades in the Entra admin center were retired at GA, consolidating into Agent 365 as the single source of truth ([Cloudpartner](https://learn.cloudpartner.fi/posts/agent365-ga-governance-licensing); [Wintive — Entra ID + Agent ID guide](https://www.wintive.com/tutorials/microsoft-365/microsoft-entra-id-complete-guide/)). Entra Agent ID is the identity substrate; Agent 365 is the governance surface over it.
- **Identity-security vendors now position against it** — e.g., [Oasis Security on Agent 365 vs. hybrid-environment agent governance](https://www.oasis.security/blog/agent-365-oasis-for-ai-agent-governance) — which is itself evidence the category is real enough to attack. The consistent critique: Agent 365 governs the Microsoft estate deeply but hybrid/multi-cloud agent fleets shallowly.

## IBM — from catalog to "Agentic Control Plane," SaaS-first

IBM made the loudest strategic repositioning of the quarter, and it is aimed exactly at the layer this document covers:

- At **Think 2026 (2026-05-05)**, IBM announced the next generation of watsonx Orchestrate as "an agentic control plane for the multi-agent era," in **private preview**, pitched at deploying "agents from any source with consistent policy enforcement and accountability" as organizations go from a handful of agents to thousands ([IBM newsroom — Think 2026](https://newsroom.ibm.com/2026-05-05-think-2026-ibm-delivers-the-blueprint-for-the-ai-operating-model-as-the-ai-divide-widens)).
- In **June 2026** IBM introduced the **Agentic Control Plane** in watsonx Orchestrate **on AWS and IBM Cloud**: end-to-end visibility into agent activity, built-in behavioral controls, a shared catalog for reuse, native scheduling, and **AgentOps** capabilities — runtime policies, audit trails, anomaly alerts ([IBM — Introducing the Agentic Control Plane](https://www.ibm.com/new/announcements/introducing-the-agentic-control-plane), June 2026; [IBM — Agent Control Plane product page](https://www.ibm.com/products/watsonx-orchestrate/agent-control-plane); [AI CERTs coverage](https://www.aicerts.ai/news/watsonx-orchestrate-preview-positions-ibm-as-agentic-control-hub/)).
- **The on-prem observation that matters for us:** the new control plane launched on **AWS and IBM Cloud** — not on-premises. IBM's documented on-prem/air-gapped path (Orchestrate on OpenShift via Software Hub — see the sibling [agent-catalog competitive doc](/features/agent-catalog/research/05-competitive.md)) is the *previous-generation* product. Until the Agentic Control Plane lands in Software Hub, the "self-managed on-prem agent fleet-governance control plane" slot that IBM would naturally occupy is open. IBM will very likely close it — its whole hybrid story depends on it — so this is a time-boxed observation, not a durable gap.
- Governance depth remains IBM's strongest card (watsonx.governance AI-Agent lifecycle objects, Risk Atlas, EU AI Act automation, per the April series — still accurate), and it is now packaged as control-plane-first rather than catalog-first.

## Google — the Agent Registry is now a shipped, documented product

The April series described Google's Agent Registry as a Cloud Next announcement. It is now real:

- **Agent Registry** is documented as "the central hub for governance and inventory for your AI ecosystem" in the Gemini Enterprise Agent Platform (launched 2026-04-22): a centralized, queryable catalog for **AI agents, MCP servers and their tools, and external API endpoints**, with automatic registration from supported runtimes and manual registration for custom deployments, centralized IAM-based access control, keyword/prefix search, and an auth manager/binding system ([Google Cloud docs — Agent Registry](https://docs.cloud.google.com/gemini-enterprise-agent-platform/govern/agent-registry); [Agent Registry overview](https://docs.cloud.google.com/agent-registry/overview)).
- **ADK integration** lets agents dynamically resolve endpoints and discover other registered agents; **Gemini Enterprise imports A2A agents directly from the Agent Registry** into the end-user governance surface ([Google Cloud docs — Import A2A agents from Agent Registry](https://docs.cloud.google.com/gemini/enterprise/docs/import-govern-agent-registry)). Registry (govern) → Gallery (consume) is the same two-surface split Microsoft runs.
- The registry itself is cloud-only. The watch item stands from the catalog sibling: Gemini is GA on air-gapped Google Distributed Cloud; if the Agent Platform follows, the disconnected wedge narrows further.

## Databricks — the registry became an OSS governance play inside MLflow

Directly relevant, since RHOAI's Agent Registry **is** MLflow:

- At DAIS 2026 (June 16), Unity Catalog's registration of "Databricks-hosted models, external model providers, MCP services, agents, and skills" under one governance framework went GA, and **Unity AI Gateway** (runtime governance over interactions between models, agents, MCPs, skills, and tools — discovery, fine-grained access control, budget enforcement, policy routing) entered preview ([Databricks — What's new with Unity Catalog at DAIS 2026](https://www.databricks.com/blog/whats-new-unity-catalog-data-ai-summit-2026); [Unity AI Gateway](https://www.databricks.com/blog/ai-governance-data-ai-summit-2026-whats-new-unity-ai-gateway)).
- The structurally important move: **Unity AI Gateway is open-sourced as part of both Unity Catalog and MLflow** ([Databricks — Unity AI Gateway](https://www.databricks.com/blog/ai-governance-data-ai-summit-2026-whats-new-unity-ai-gateway)). Upstream MLflow is thus becoming a *governance* surface, not just tracking/registry — recent MLflow releases (3.13.0) focus on "access control, the lifecycle of trace data, and richer support for agents," and upstream has an established metadata-first registry pattern (the MCP Server Registry proposal, with a [skill-registry follow-on filed as mlflow#22833](https://github.com/mlflow/mlflow/issues/22833)) ([MLflow releases](https://mlflow.org/releases/)).
- Two-edged for us: building the RHOAI registry on MLflow now inherits governance primitives faster than we could build them — and it means Databricks, a serverless-only competitor at the platform level, steers the substrate. Upstream influence on the agent/skill registry threads is a strategic activity, not housekeeping.

## Salesforce — Agent Fabric: the cross-vendor fleet registry went GA first

Salesforce quietly shipped the thing several vendors still describe on slides:

- **MuleSoft Agent Fabric** delivers "a central registry supported with automated discovery, where every agent and MCP server can be cataloged, discovered, and reused." **Agent Scanners went GA January 2026** for Salesforce Agentforce, Amazon Bedrock, Google Vertex AI, and Microsoft Copilot Studio — automatically extracting each agent's capabilities, underlying LLM, and data access, then **normalizing the metadata to A2A agent-card specifications**; a curated set of third-party MCP servers in the Agent Registry also went GA January 2026 ([Salesforce — MuleSoft Agent Fabric automated discovery](https://www.salesforce.com/news/stories/mulesoft-agent-fabric-automated-agent-discovery/); [MuleSoft docs — Agent Fabric overview](https://docs.mulesoft.com/general/agent-fabric-overview)).
- Agent Broker (deterministic orchestration over the registry) hit GA June 2026, with governance controls positioned explicitly as the control plane for "multi-vendor AI" ([Salesforce — Agent Fabric control plane announcement](https://www.salesforce.com/news/stories/agent-fabric-control-plane-announcement/)).
- Lesson for the RHOAI registry: **scanner-based auto-discovery of agents living in other vendors' runtimes, normalized to A2A cards, is now table stakes** for anything claiming fleet coverage. The MLflow proposal's pluggable discovery providers are the right shape; the provider list needs to include non-RHOAI runtimes to be credible.

## ServiceNow — enforcement muscle plus the first shipped cross-registry federation

- **AI Control Tower** is ServiceNow's "control plane for discovering, governing, securing, observing, and measuring AI across the enterprise." At Knowledge 2026 (May 5) it gained enforcement capabilities including **agent kill switches** ([The Register — ServiceNow adds agent kill switches](https://www.theregister.com/software/2026/05/05/servicenow-adds-agent-kill-switches-to-ai-control-tower/5228579), 2026-05-05).
- The **June 2026 release** shipped something the April series listed as existing nowhere: working cross-registry federation between vendors. AI asset owners can **publish a managed ServiceNow agent into the Microsoft Agent 365 directory directly from AI Control Tower**, after which the agent is visible and manageable in both registries — plus day-one compliance content packs (EU AI Act, Colorado AI Act, California) and MCP-server import from the community catalog with an approval lifecycle ([ServiceNow community — AI Control Tower June 2026 release](https://www.servicenow.com/community/ai-control-tower-articles/ai-control-tower-what-s-new-in-the-june-2026-release/ta-p/3561445); [ServiceNow newsroom — expanded Microsoft integration](https://newsroom.servicenow.com/press-releases/details/2026/ServiceNow-expands-AI-agent-governance-through-deeper-integration-with-Microsoft/default.aspx)).
- Federation is arriving as **bilateral vendor integrations**, not as the neutral standard the April series imagined. That creates a real opening for standards-based federation (ARD, AGNTCY OASF) — and a risk that pairwise walled-garden bridges win by default.

## New category since April — shadow-agent discovery and agent security posture

The agent-interop series (2026-07-11) flagged shadow-agent discovery as "a recognized gap with no mature solution." That needs updating: a commercial category has formed, even if maturity is debatable.

- **Zenity** — agentic-AI security posture management: automatic agent discovery with a real-time inventory including ownership and dependency mapping across low-code copilot builders, SaaS agent platforms, and home-grown agents on cloud model services; named a Gartner Cool Vendor in Agentic AI TRiSM (2025). Its field data is the best public evidence of the problem's scale: a Fortune 50 financial-services firm with 150,000+ agent/automation-linked resources; a Fortune 50 pharma with 2,000+ shared agent instances, 82% built by non-developers ([Zenity — AI agent sprawl is the new shadow IT](https://zenity.io/blog/security/incomplete-agent-inventory-risk), 2026).
- **WitnessAI** — shadow-AI discovery plus a full AI inventory (applications, MCP servers, agents) with intent-based behavioral controls and runtime defense ([WitnessAI — Observe](https://witness.ai/observe/)).
- **Oasis Security** — agent identity governance positioned explicitly for the hybrid gap Agent 365 leaves ([Oasis — Agent 365 & Entra Agent ID vs. Oasis](https://www.oasis.security/blog/agent-365-oasis-for-ai-agent-governance)).
- **Lasso Security** — open-source, self-hostable MCP gateway with server reputation scanning and prompt-injection detection ([roundup: Top agentic AI security solutions 2026](https://www.linx.security/blog/top-agentic-ai-security-solutions)).
- Observability platforms (LangSmith, Langfuse, Arize/Phoenix, Datadog LLM Observability) deepened *tracing and evals* through H1 2026 but none has shipped a governance registry — deployment inventories at best. The registry/observability boundary is holding; the registry/security-posture boundary is the one blurring.

## Open source — the registry slot is no longer empty

The April series' differentiation claim — "no open-source platform has a dedicated agent registry" — now needs narrowing:

- **Solo.io `agentregistry`** ([agentregistry-dev/agentregistry](https://github.com/agentregistry-dev/agentregistry)) — "a cloud-native registry for discovering, curating, and deploying MCP servers, agents, and skills across local development and Kubernetes environments," with registry curation, Kubernetes deployment, and kagent integration for runtime execution. Proposed to the **CNCF Sandbox on 2026-03-25**, pending board review ([cncf/sandbox#477](https://github.com/cncf/sandbox/issues/477)). This is a direct open-source occupant of the k8s-native agent-registry slot, from the same vendor whose kagent is applying for CNCF Incubation.
- **AGNTCY Agent Directory** (Linux Foundation; Cisco, Dell, Google Cloud, Oracle, and **Red Hat** as formative members) — a peer-to-peer, OCI-storage-based directory for publishing and federating agents across organizations using the Open Agent Schema Framework; testbed v0.3 demonstrates **synchronization between different agent registries**, with planned MCP/A2A/NANDA support and a Directory MCP server for IDE-side discovery ([Linux Foundation — AGNTCY announcement](https://www.linuxfoundation.org/press/linux-foundation-welcomes-the-agntcy-project-to-standardize-open-multi-agent-system-infrastructure-and-break-down-ai-agent-silos); [AGNTCY blog — Directory MCP server](https://blogs.agntcy.org/technical/2026/02/19/directory-mcp-server.html), 2026-02-19).
- The academic taxonomy ([arXiv 2508.03095 — Evolution of AI Agent Registry Solutions](https://arxiv.org/abs/2508.03095)) now lists five live approaches: MCP Registry (centralized), A2A Agent Cards (self-describing), AGNTCY Directory (distributed DHT + OCI), Entra Agent ID (enterprise directory), NANDA AgentFacts (verifiable credentials).
- **What remains true:** none of these is an *enterprise fleet-governance* registry — versioned lifecycle, approval workflow, audit trail, lineage to models/evals/traces, running-instance health — shipped as a supportable open-source product. `agentregistry` is closest in intent but pre-Sandbox; AGNTCY solves federation/discovery, not governance. The corrected claim: *open-source agent registries now exist for discovery and distribution; an open-source governed fleet registry with lineage remains unshipped* — which is precisely the MLflow-based registry's lane.

## Standards — ARD formalizes the discovery layer above MCP + A2A

- On **2026-06-17**, Google and Microsoft jointly introduced the **Agentic Resource Discovery (ARD)** specification — Apache 2.0, a machine-readable catalog published under an organization's own domain so agents can search, verify, and use tools/agents across sources; MCP defines invocation, ARD defines the earlier discovery stage ([Google Developers Blog — Announcing ARD](https://developers.googleblog.com/announcing-the-agentic-resource-discovery-specification/); [Microsoft Command Line — Introducing ARD](https://commandline.microsoft.com/agentic-resource-discovery-specification-ard/); [InfoQ coverage](https://www.infoq.com/news/2026/07/agentic-resource-discovery-spec/), July 2026).
- Collaborators include Cisco, Databricks, GitHub, GoDaddy, Hugging Face, NVIDIA, Salesforce, ServiceNow, and Snowflake; GitHub Copilot's **Agent Finder** is the first shipping implementation. Notably absent: Anthropic and OpenAI ([InfoQ](https://www.infoq.com/news/2026/07/agentic-resource-discovery-spec/)).
- For the RHOAI registry this is the concrete successor to the April series' "unified entity card" watch item: a registry that can **emit and consume ARD catalogs** gets standards-based federation with the Microsoft/Google ecosystems without bilateral integrations. Adoption is early and contestable — the right moment to implement and influence, and the interop series' recommendation to track ARD is upgraded to "implement behind a flag."

## Cross-cutting trends

1. **Inventory is commoditizing; governance is monetizing.** Microsoft gives the registry away and charges $15/user for the control plane; AWS plans to meter records but gives the curation workflow away in preview; Google and Databricks bundle the registry into the platform. The claim "registries are now monetized" (catalog sibling) is better stated as: *the registry is becoming the free front door to a paid fleet-governance stack.* Consequence for RHOAI: the MLflow registry's value story must be the governed stack (registry + MCP gateway + identity + policy + lineage), never the inventory alone.
2. **Fleet governance converged on a shared shape.** Every shipped control plane now offers: inventory with ownership, identity per agent, approval/blocking, audit, and increasingly a kill switch (ServiceNow) and runtime policy outside the agent's code (AWS Guardrails-at-gateway, IBM AgentOps runtime policies, Unity AI Gateway). This is the checklist a 3.6 registry TP will be demoed against.
3. **Cross-registry federation started shipping — bilaterally.** ServiceNow↔Agent 365 (June 2026), Salesforce scanners ingesting Bedrock/Vertex/Copilot Studio agents (Jan 2026), AGNTCY registry-sync testbed, ARD as the would-be neutral layer. The April "nobody has federation" claim is dead; the open question is whether federation is standards-based or pairwise.
4. **Auto-discovery beats manual registration.** Google auto-registers from its runtimes; Salesforce scans other clouds; Zenity/WitnessAI scan everything including low-code sprawl; Microsoft detects shadow agents in-tenant. A registry that only knows what was explicitly published to it is a 2025 design.
5. **Agent identity became a directory problem.** Entra Agent ID (agents as directory objects) is the clearest expression; SPIFFE-based workload identity (our lane, per the interop series) remains the strongest *cryptographic* substrate but has no enterprise-directory UX yet. The two will have to meet; whoever bridges k8s workload identity and enterprise directory semantics for agents owns a durable layer.
6. **The sovereign/regulated escape hatch is narrowing but not closed.** AgentCore in GovCloud, Gemini on air-gapped GDC — yet **no vendor ships a self-managed agent fleet-governance control plane a customer can run disconnected**, and even IBM's new Agentic Control Plane launched cloud-first. That exact artifact is what RHOAI 3.6's registry + gateway can be.

## Honest positioning — the RHOAI wedge as of July 2026

**What still holds:**
- **The self-managed, disconnected, governed fleet registry remains unclaimed.** Every GA control plane (Agent 365, Agent Fabric, AI Control Tower, Google Agent Registry, IBM ACP) is SaaS. IBM is the only credible near-term claimant on-prem and its new control plane hasn't landed there yet. An MLflow-based registry with MCP-gateway policy, SPIFFE-track identity, and air-gap operation is a combination with zero shipping competitors today.
- **Infra pricing is a stronger TCO argument than in April** — now that the meters are visible: $15/user/mo (Microsoft), per-Net-Record (AWS at GA, reported), platform-bundled seats (Google $30/seat Gemini Enterprise). At fleet scale (thousands of agents, all employees as sponsors/users), infra-priced fleet governance is a clean pitch — with the caveat that Microsoft's free registry floor blunts "you pay just to inventory" as a talking point.
- **MLflow alignment aged well.** Databricks open-sourcing Unity AI Gateway into MLflow means the upstream substrate is accreting exactly the governance primitives a fleet registry needs, and the Databricks catalog→registry architecture (the sibling's finding) is now also a governance architecture. We ride that current; cloud-only Databricks can't follow us on-prem.

**What eroded since April:**
- **"First/only" claims are gone.** AWS is no longer the lone registry; Microsoft, Google, Salesforce, ServiceNow, IBM all ship registry-layer product, most at GA while ours targets TP in 3.6 EA1 with the backend in EA2. The lag story from the catalog sibling (~12–18 months on catalogs) is now roughly a 6–12 month lag on registries — smaller, because the category is younger, but closing fast.
