Bring Your Own MCP Server

The MCP Catalog isn't limited to Red Hat and partner listings: a customer can package and publish their own MCP server so it deploys through the same catalog-to-Lifecycle-Operator-to-Gateway path as any curated entry. This tier is called Bring your own in the trust tier glossary; see Trust & Policy for how it sits relative to the Red Hat, Partner, and Community tiers.

Status: Manual Today

Packaging and publishing a Bring Your Own (BYO) server is a manual workflow in RHOAI 3.4 DP. It's tracked for future catalog automation under RHAIRFE-2081 and RHAIRFE-2082: open asks, not shipped.

Container-Packaging Checklist

  • Base image: build on a UBI (Universal Base Image) base.
  • Transport: use explicit HTTP transport, not stdio. A stdio-only server run this way will start, produce no error output, and exit silently; the failure won't surface.
  • Bind address: bind 0.0.0.0, not 127.0.0.1.
  • Credentials: accept credentials via HTTP headers, not environment variables or baked-in config, so the Gateway/Vault credential-injection path works.
  • MCP path: declare a stable, documented MCP path (e.g. /mcp).
  • Health endpoint: expose a health endpoint for liveness/readiness probing.
Why this matters: the Catalog's stated technical bar for partner servers (Streamable HTTP only, no stdio) is the same bar a BYO server has to clear to work through the Gateway's credential-injection and routing path; see Partner Onboarding Pipeline for the equivalent partner-side requirements.

Restricted-SCC Compliance

OpenShift's restricted Security Context Constraint is the default for MCP server workloads:

Requirement Detail
Non-root Container must not run as UID 0.
Drop all capabilities No Linux capabilities beyond the restricted default set.
No privilege escalation allowPrivilegeEscalation: false.
Read-only root filesystem Container filesystem is read-only; use storage mounts (ConfigMap/Secret/EmptyDir) for writable paths.

Table carried verbatim from the management hub's Configuration Reference page.

Catalog-YAML Metadata Spec

  • Top level: name, provider, description, version, transports, artifacts, deploymentMode
  • Per-tool accessType: read_only, read_write, or execute
  • runtimeMetadata: drives the deployment modal's pre-filled prerequisites
  • securityIndicators: mapped from the restricted-SCC validation checklist results above

Worked Example

An illustrative catalog-yaml entry using the field names above (values are illustrative, not a live listing):

name: example-mcp-server
provider: Example Corp
description: Illustrative BYO MCP server entry
version: 1.0.0
transports:
  - streamable-http
artifacts:
  - image: quay.io/example/example-mcp-server:1.0.0
deploymentMode: on-cluster
tools:
  - name: list_items
    accessType: read_only
  - name: create_item
    accessType: execute
  - name: update_item
    accessType: read_write
runtimeMetadata:
  serviceAccount: mcp-viewer
  configMap: example-mcp-server-config
securityIndicators:
  verifiedSource: true
  sast: true
  secureEndpoint: true
  readOnlyTools: false
