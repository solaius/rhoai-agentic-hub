Positioning & Trust Tiers

The MCP Catalog is the storefront for MCP discovery, not a discovery-only list. It's the governance and management entry point of the MCP ecosystem within RHOAI.

The Storefront Story

The Catalog is "a curated catalog of MCP servers that you can discover, deploy, and manage directly on Red Hat OpenShift." Where discovery-only registries stop at "here's a list," the Catalog provides a governed path all the way to consumption: discover in the catalog UI, deploy via the MCP Lifecycle Operator, connect through the MCP Gateway, consume in Gen AI Studio. AI Hub treats MCP servers as first-class citizens alongside models, not a bolt-on category.

Trust Tiers

The Catalog surfaces four tiers of provenance:

Tier Meaning
Red Hat built and supported Authored and supported by Red Hat; the highest trust tier in the Catalog
Partner built and certified Built by an ISV or technology partner; verified through the partner onboarding pipeline (consent, technical attestation, image build)
Community curated Sourced from the open community; included based on curation criteria rather than a formal partner agreement
Bring your own A customer-supplied MCP server deployed through the same Catalog to Lifecycle Operator to Gateway path, outside the curated tiers

The 3.4 DP catalog shipped 10 servers across the Red Hat, Partner, and Community tiers (see What Is the MCP Catalog?); trust tier display itself is sourced from the Registry starting in 3.5+.

RBAC and Approval Workflow

Roadmap, not shipped. "RBAC and approval workflow integration with Registry" is listed as a Catalog key feature, but it depends on the Registry, which is still in Design phase: Dev Preview is not landing in RHOAI 3.5 stable, and Tech Preview is targeted for RHOAI 3.6 EA1. The Registry's current data model has exactly two governance enums (MCPServerStatus: active, deprecated, retired; MCPPublishState: draft, published, deprecated, retired) and no approval_status, verification_status, or certification_status field. The earlier four-track governance model (Lifecycle, Approval, Verification, Certification) described in early design drafts is a deferred proposal for a post-MVP follow-on, not a shipped capability.

Admin UI: RHAISTRAT-1859

The Catalog Admin UI (a Settings page with CRUD on Catalog source ConfigMaps) is in progress, targeted for the RHOAI 3.5 release, tracked as RHAISTRAT-1859. This is a feature landing within a specific RHOAI release, which is a separate axis from the Catalog's own DP/TP/GA maturity ladder: the Catalog itself remains DP through 3.5, with TP targeted for RHOAI 3.6 EA1 and GA targeted at RHOAI 3.6 (code freeze Oct 23, 2026).

PM Vision: Not Shipped

PM vision, not shipped. The Catalog's long-term vision extends beyond MCP servers to manage Tools, Resources, Prompts, and Workloads as first-class catalog entities, including workload-to-tool mappings. Planned capabilities:
  • Tool Masking: expose only the tools a given workload actually needs
  • Input Simplification: split multi-purpose tools into single-purpose variants
  • Output Filtering: trim tool responses to only the fields relevant to the calling LLM
None of this is shipped. Current scope is MCP server discovery, browsing, and selection. Federation with external registries (Docker MCP Catalog, GitHub MCP Registry, Smithery, Microsoft Copilot Catalog, Databricks Unity Catalog) is part of the same planned scope; Red Hat's own federation posture toward the official MCP registry is a separate, still-open question (see Upstream & Ecosystem).
