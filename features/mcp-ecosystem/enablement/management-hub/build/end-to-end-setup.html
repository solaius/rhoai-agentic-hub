Integration Scenarios

Choose an integration scenario for deploying MCP servers in RHOAI, then use the folded reference below for the operator tiers, seam artifacts, and cross-component issues. The per-component mechanics (MCPServer CRD fields, Gateway CRDs, Catalog packaging) now live in each component hub; this page keeps the cross-component seams.

Do I Need the Gateway?

Gateway is NOT required for basic playground usage. Use Track A for demos and testing. Use Track B for production with identity-based access control, per-tool metrics, and credential brokering.

Confirmed by the MCP product management team and a Red Hat solutions architect.

Track A: Minimal Setup (Playground without Gateway)

Get MCP working in Gen AI Studio playground with minimal infrastructure. No Gateway, Keycloak, or Vault required.

1Install MCP Lifecycle Operator

The MCPLO is not available via OLM or Helm. Install manually from the GitHub release asset (repo is kubernetes-sigs/mcp-lifecycle-operator):

# Download and apply the manifest
oc apply -f https://github.com/kubernetes-sigs/mcp-lifecycle-operator/releases/download/v0.2.0/install.yaml

# Wait for operator to be healthy
oc get pod -n mcp-lifecycle-operator-system

Verify all pods are Running before proceeding.

2Enable MCP Catalog in RHOAI

The MCP Catalog is not enabled by default. Patch the dashboard config:

oc patch odhdashboardconfigs odh-dashboard-config \
    -n redhat-ods-applications \
    --type=merge \
    -p '{"spec":{"dashboardConfig":{"mcpCatalog":true}}}'

Wait for the RHOAI web UI to update. Refresh your browser. AI Hub → MCP Servers should now appear in the navigation.

3Create Namespace with Dashboard Label

Critical: The namespace must have a special label to appear in the RHOAI web UI. This label cannot be applied via the OCP console or oc new-project — you must use a manifest:

oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  labels:
    opendatahub.io/dashboard: "true"
  name: my-mcp
EOF
Common Issue: Namespace Doesn't Appear in RHOAI UI

Cause: Missing opendatahub.io/dashboard: "true" label.

Fix: Cannot set via oc new-project or OCP console. Must apply via manifest as shown above.

4Create Prerequisites

The MCP Catalog does NOT create prerequisites automatically (this would require privilege escalation). You must create them manually for each MCP server:

# Service account with cluster-wide read access
oc create serviceaccount mcp-viewer -n my-mcp
oc create clusterrolebinding my-mcpviewer-read \
    --clusterrole view \
    --serviceaccount my-mcp:mcp-viewer

# Configuration for the OpenShift MCP server
oc apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: openshift-mcp-server-config
  namespace: my-mcp
data:
  config.toml: |
    log_level = 5
    port = "8080"
    read_only = true
    toolsets = ["core", "config"]
EOF
Common Issue: "failed to create Deployment / ConfigMap not found"

Cause: Prerequisites not created before deploying from catalog.

Fix: The catalog does NOT create ServiceAccounts, ConfigMaps, or RBAC automatically. Create them BEFORE deploying the MCP server.

5Deploy MCP Server

Option A — From the Catalog UI:

  1. Navigate to AI Hub → MCP Servers
  2. Click the openshift-mcp-server tile
  3. Read the README for prerequisites (you already created them in Step 4)
  4. Click Deploy MCP Server
  5. Name the deployment (e.g., "ocp-mcp") and select the my-mcp namespace
  6. Click Deploy

Option B — From manifests (bypassing catalog):

oc apply -f - <<EOF
apiVersion: mcp.x-k8s.io/v1alpha1
kind: MCPServer
metadata:
  name: ocp-mcp
  namespace: my-mcp
spec:
  source:
    type: ContainerImage
    containerImage:
      ref: registry.redhat.io/openshift-mcp-beta/openshift-mcp-server-rhel9:0.2
  config:
    port: 8080
    path: /mcp
    storage:
    - path: /etc/mcp-config
      permissions: ReadOnly
      source:
        configMap:
          name: openshift-mcp-server-config
  runtime:
    security:
      serviceAccountName: mcp-viewer
EOF

The full field-by-field MCPServer CRD schema and a complete example live in the MCP Lifecycle Operator Knowledge Hub: MCPServer CRD Reference.

Verify the deployment:

oc get mcpserver -n my-mcp
oc get pods -n my-mcp

6Register MCP Server in Gen AI Studio

This is the critical bridge. The gen-ai-aa-mcp-servers configmap tells Gen AI Studio about available MCP servers:

oc apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: gen-ai-aa-mcp-servers
  namespace: redhat-ods-applications
data:
  OpenShift-MCP: |
    {
      "url": "http://ocp-mcp.my-mcp.svc.cluster.local:8080/mcp",
      "description": "Red Hat OpenShift MCP server"
    }
EOF

Verify the server appears in the RHOAI UI:

  • Navigate to Gen AI Studio → AI asset endpoints → MCP Servers
  • You should see "OpenShift-MCP" listed
Common Issue: "MCP tab empty in playground"

Cause: Configmap not created, created in wrong namespace, or server URL incorrect.

Fix: Ensure the configmap is in redhat-ods-applications namespace and the server URL matches the actual service name and port.

7Create Playground and Enable MCP

  1. Navigate to Gen AI Studio → Playgrounds
  2. Click Create Playground
  3. Select a project with a tool-calling model deployed (e.g., mistral-7b-instruct or larger)
  4. In the playground settings panel, click the MCP tab
  5. Check the OpenShift-MCP checkbox
Common Issue: "Model can't handle tool output"

Cause: Small models (like llama-3.2-3b-instruct) struggle with large tool responses (e.g., listing all pods with all attributes).

Fix: Use focused prompts like "list all projects" instead of "list all pods". Or use a larger tool-calling model.

8Verify with Test Prompt

In the playground, submit a test prompt that would use the MCP tools:

list all projects in the cluster

Expected behavior:

  1. The model recognizes this requires the OpenShift MCP server
  2. The playground calls the list_projects tool
  3. The model responds with a summary of all OpenShift projects

Example output:

I found 15 projects in your OpenShift cluster: default, kube-system, my-mcp, openshift-operators, redhat-ods-applications...

Track B: Full Setup (Gateway + Keycloak + Vault)

Production-grade setup with identity-based access control, per-tool metrics, credential brokering, and virtual MCP server aggregation.

Complexity Warning

This track requires Service Mesh, Keycloak, Vault, and cert-manager. Only proceed if you need production-grade auth and governance. For demos and testing, use Track A.

1Install Tier 1 Operators (via OperatorHub)

These are platform prerequisites installed via OperatorHub:

  • OpenShift Service Mesh (stable) — Programs Envoy for Gateway; handles mTLS
  • cert-manager (stable-v1) — TLS certificates and ClusterIssuers
  • Red Hat Connectivity Link (stable) — Kuadrant, Authorino, Limitador
  • RHOAI (may require custom CatalogSource for 3.4.0+)

See Operator Tiers & Validated Versions below for the two-tier model, install order, and validated version matrix.

2Install Tier 2 Operators (post-platform)

These are MCP ecosystem components installed after platform operators:

  • MCP Gateway Operator — OLM via custom CatalogSource; watches MCPGatewayExtension CRs
  • RHBK (Keycloak) — Identity management
  • MCP Lifecycle Operator — Manual YAML apply from v0.2.0
  • HashiCorp Vault — Helm chart for dynamic secret injection (optional)

See Operator Tiers & Validated Versions below for installation order and validated versions.

3Set Up Gateway Namespace

Create a dedicated namespace for Gateway components and install via Helm:

oc new-project mcp-gateway

# Install MCP Gateway with controller disabled (Kuadrant manages routing)
helm upgrade -i mcp-gateway oci://ghcr.io/kuadrant/charts/mcp-gateway \
  -n mcp-gateway \
  --set controller.enabled=false \
  --set publicHost=gateway.apps.example.com

Create an OpenShift Route with edge TLS termination pointing to the Gateway service.

4Configure Keycloak

Set up Keycloak realm with two OIDC clients:

  • Confidential client for Gateway broker (service-to-service with client secret)
  • Public client for Gen AI Studio/Playground (authorization code flow, no secret)

Configure realm roles (user, admin) and map per-server tool roles in resource_access claim.

See MCP Gateway Auth Setup for detailed Keycloak configuration.

5Create Wristband Signing Keys

Generate ECDSA key pair for wristband signing:

# Generate private key
openssl ecparam -name prime256v1 -genkey -noout -out wristband-private.pem

# Extract public key
openssl ec -in wristband-private.pem -pubout -out wristband-public.pem

# Create secrets
oc create secret generic wristband-signing-key \
  -n mcp-gateway \
  --from-file=key.pem=wristband-private.pem

oc create secret generic wristband-public-key \
  -n mcp-gateway \
  --from-file=key.pem=wristband-public.pem

Patch MCPGatewayExtension to reference the public key:

oc patch mcpgatewayextension mcp-gateway \
  -n mcp-gateway \
  --type=merge \
  -p '{"spec":{"trustedHeadersKey":"wristband-public-key"}}'

6Create Gateway-Level AuthPolicy

Single AuthPolicy targeting Gateway's MCP listener handles:

  • JWT authentication — Validates Bearer tokens against Keycloak OIDC
  • Wristband signing — OPA Rego extracts per-server tool roles from JWT resource_access claim, signs into ES256 wristband JWT in x-authorized-tools header
  • VirtualMCPServer routing — CEL reads groups claim, sets x-mcp-virtualserver header

See the Configuration Artifacts section below for the AuthPolicy and MCPGatewayExtension shapes, and the MCP Gateway Knowledge Hub for the full CRD reference.

7Deploy MCP Servers and Register Gateway URL

Deploy MCP servers as in Track A, but register the gateway URL (not individual server URLs) in gen-ai-aa-mcp-servers:

oc apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: gen-ai-aa-mcp-servers
  namespace: redhat-ods-applications
data:
  MCP-Gateway: |
    {
      "url": "https://gateway.apps.example.com/mcp",
      "description": "MCP Gateway (all servers)"
    }
EOF

The Gateway will route to backend MCP servers based on VirtualMCPServer configuration and user identity.

8Verify from Playground

Create a playground, select the Gateway endpoint, and authenticate via Keycloak:

  1. Navigate to Gen AI Studio → Playgrounds
  2. Click Create Playground
  3. In settings, enable MCP-Gateway
  4. You'll be redirected to Keycloak for login
  5. After login, submit a test prompt: "list all projects in the cluster"

The Gateway will validate your JWT, extract tool permissions from the wristband, and route to the appropriate MCP server based on your identity and groups.

Alternative Validation & Deployment Paths (Unsupported)

Two additional paths surfaced in field notes. Neither is part of Track A or Track B; treat both as unsupported-but-useful alternatives.

Claude Code Local Validation

Validate the OpenShift MCP server independent of RHOAI/Gen AI Studio using Claude Code:

claude mcp add-json kubernetes-mcp-server '{"command":"npx","args":["-y","kubernetes-mcp-server@latest","--read-only"],"env":{"KUBECONFIG":"..."}}'

Caveat: this runs the MCP server locally via npx, not against the remote cluster-deployed instance. Useful for isolating client-side issues, not a substitute for Track A/B verification.

Direct Helm Path (Bypasses MCPLO and Catalog)

Deploy the MCP server directly via Helm chart, skipping the MCP Lifecycle Operator and Catalog UI entirely:

helm upgrade -i ocp-mcp oci://ghcr.io/containers/charts/kubernetes-mcp-server \
  -n my-mcp

Wire the resulting service into gen-ai-aa-mcp-servers manually, by local service hostname, the same manual bridge step as Track A step 6.

Operator Tiers & Validated Versions

The MCP ecosystem on RHOAI installs in two tiers. Installation order matters: Tier 1 must complete before Tier 2. Per-operator install mechanics are standard OperatorHub subscriptions; the one exception, MCPLO, installs manually from its upstream release and is documented in the MCP Lifecycle Operator Knowledge Hub: Installation.

Tier Components Installation Method Purpose
Tier 1
Platform Prerequisites		 • OpenShift Service Mesh
• cert-manager
• Red Hat Connectivity Link
• RHOAI 		OperatorHub Platform-level infrastructure for API management, TLS, and AI workloads
Tier 2
MCP Ecosystem		 • MCP Gateway Operator
• MCP Lifecycle Operator
• RHBK (Keycloak)
• HashiCorp Vault 		OLM custom catalog, Manual YAML, Helm MCP-specific management, auth, and credential injection
Installation Order Matters

All Tier 1 operators must be installed and healthy before installing Tier 2 components. MCP Gateway requires Service Mesh and Kuadrant to be fully operational. Deploying MCPServer CRs before MCPLO is installed, or MCPGatewayExtension before Gateway API CRDs are present, is the leading cause of operator pods crashlooping.

After Tier 1, apply a Kuadrant CR to activate the control plane, and ensure the DataScienceCluster has llamastackoperator: Managed. MCP Catalog and Gen AI Studio are enabled with odhdashboardconfigs patches (see the Configuration Artifacts section). Verify each tier is healthy before proceeding:

# Tier 1
oc get csv -n openshift-operators | grep -E "(servicemesh|cert-manager|kuadrant)"
oc get kuadrant -n gateway-system

# Tier 2
oc get pods -n mcp-lifecycle-operator-system
oc get csv -n openshift-operators | grep mcp-gateway
oc get keycloak -A
oc get pods -n vault  # if Vault installed

Version Support Gap — August 2026

RHOAI 3.5 (Aug 2026) supports OCP 4.19, but RHCL 1.4+ will likely require OCP 4.20+. The configuration (RHOAI 3.5 + RHCL + OCP 4.19) is untested and best-effort support only.

OCP 4.19 maintenance support ends December 2026 (only 4 months overlap). Customers are advised to upgrade to OCP 4.20+ for full support.

The Gen AI Studio Bridge ConfigMap

The gen-ai-aa-mcp-servers ConfigMap is the cross-component seam between deployed servers and the Studio. Register individual server URLs (Track A) or the gateway URL (Track B):

apiVersion: v1
kind: ConfigMap
metadata:
  name: gen-ai-aa-mcp-servers
  namespace: redhat-ods-applications
data:
  OpenShift-MCP: |
    {
      "url": "http://ocp-mcp.my-mcp.svc.cluster.local:8080/mcp",
      "description": "Red Hat OpenShift MCP server"
    }
  # When using the Gateway, register the gateway URL instead of per-server URLs:
  MCP-Gateway: |
    {
      "url": "https://gateway.apps.example.com/mcp",
      "description": "MCP Gateway (all servers)"
    }

Cross-Component Issues

The ecosystem-level failure points that span more than one component, organized as Symptom → Cause → Fix. Component-specific issues point to the owning hub.

1. Namespace Doesn't Appear in RHOAI Project Selector

Symptom

After creating a namespace with oc new-project my-mcp or via the OCP console, the namespace does not appear in the RHOAI web UI project selector.

Cause

Missing opendatahub.io/dashboard: "true" label on the namespace. This label cannot be applied via oc new-project or the OCP console namespace creation form.

Fix

Delete the namespace and recreate it using a manifest with the label pre-set (see Track A step 3).

2. Catalog Deploy Fails: "ConfigMap not found" / Server Not Appearing

Deploying an MCP server from the catalog without pre-creating prerequisites (ServiceAccount, ConfigMap, RBAC), or a missing namespace label / failed MCPServer CR. Full symptom/cause/fix lives in the MCP Catalog Knowledge Hub: Enablement & Troubleshooting.

3. MCP Tab Empty in Playground

Symptom

In the Gen AI Studio playground settings, the MCP tab is empty or doesn't show any servers.

Cause

The gen-ai-aa-mcp-servers configmap not created, created in wrong namespace, or server URL is incorrect.

Fix

Create or verify the configmap in redhat-ods-applications namespace (see Track A step 6). Verify the server URL matches the actual service name and port, then refresh the playground UI.

4. Model Returns Empty or Error for Tool Calls

Symptom

The model calls an MCP tool but returns an empty response, error, or gets stuck.

Cause

Small models (like llama-3.2-3b-instruct) struggle with large tool responses. For example, listing all pods with all attributes returns a massive JSON payload that exceeds the model's context window or reasoning capacity.

Fix

Option 1: Use focused prompts ("list all projects" instead of "list all pods in the cluster"). Option 2: Use a larger tool-calling model with better reasoning capacity (e.g., mistral-7b-instruct or larger).

5. Workbench Can't Call MCP Servers

Symptom

Trying to call MCP servers from Python code in a workbench (e.g., using Langchain) fails or returns connection errors.

Cause

No documented way to call MCP servers from Python code in workbenches. The playground's exported code doesn't work for MCP calls.

Fix

Status: Open gap (as of June 2026). A Python MCP client library for remote HTTP-based MCP servers is needed. Workaround: use the playground for MCP interactions, or implement a custom HTTP client following the MCP protocol spec.

6. tool/list Fails with OAuth Enabled (Gateway)

Symptom

When OAuth authentication is enabled on an MCP server, the tool/list call fails with a 401 Unauthorized error.

Cause

Long-running credentials needed for OAuth flows. Short-lived tokens expire before the tool/list operation completes.

Fix

Option 1: Use per-user tool/list with refreshable tokens (requires Gateway + Vault credential injection). Option 2: Configure the MCP server with a skip-auth mode for tool/list only (tool invocations still require auth). Option 3: Use service account credentials with longer TTL instead of user OAuth tokens.

7. Wristband Signature Verification Fails (Gateway)

Symptom

MCP Gateway returns 403 Forbidden with an error like "wristband signature verification failed".

Cause

Mismatch between the private key used by Authorino to sign wristbands and the public key configured in MCPGatewayExtension.spec.trustedHeadersKey.

Fix

Verify the key pair and secret references match (see Track B step 5), and regenerate the ECDSA key pair and secrets if they have drifted:

# Check the public key secret referenced in MCPGatewayExtension
oc get mcpgatewayextension mcp-gateway -n mcp-gateway -o yaml | grep trustedHeadersKey

# Verify the secret exists
oc get secret wristband-public-key -n mcp-gateway

8. Operator Pods Crashlooping

Usually a missing CRD dependency: MCPServer CRs deployed before MCPLO is installed, or MCPGatewayExtension deployed before Gateway API CRDs are present. Full symptom/cause/fix lives in the MCP Lifecycle Operator Knowledge Hub: Supported Servers & Troubleshooting.

9. "cluster-admin required" for ConfigMap Patches

Symptom

Non-admin users can't enable MCP Catalog or register MCP servers in Gen AI Studio.

Cause

The gen-ai-aa-mcp-servers configmap approach requires write access to redhat-ods-applications namespace, which is typically restricted to cluster admins.

Fix

Current (RHOAI 3.4): Cluster admin access required. Future: resolved when the MCP Registry replaces the manual configmap approach, with proper RBAC controls. See Gaps & Open Questions.
