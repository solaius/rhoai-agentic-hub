Troubleshooting

Known issues, common failure points, and solutions for MCP deployment on RHOAI. Organized as Symptom → Cause → Fix.

Documentation Gaps (as of June 2026)

Many of these issues stem from circular documentation references: product docs reference MCP features but don't explain prerequisites. The catalog UI references blogs, blogs reference GitHub repos, and GitHub repos reference product docs — creating a loop with no complete end-to-end guide.

This troubleshooting page consolidates the gaps from Fernando Lozano's research notes, Jaideep Rao's testing, and internal Slack discussions.

Deployment Issues

1. Namespace Doesn't Appear in RHOAI Project Selector

Symptom

After creating a namespace with oc new-project my-mcp or via the OCP console, the namespace does not appear in the RHOAI web UI project selector.

Cause

Missing opendatahub.io/dashboard: "true" label on the namespace. This label cannot be applied via oc new-project or the OCP console namespace creation form.

Fix

Delete the namespace and recreate it using a manifest:

oc delete namespace my-mcp

oc apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  labels:
    opendatahub.io/dashboard: "true"
  name: my-mcp
EOF

2. "failed to create Deployment: ConfigMap not found"

Deploying an MCP server from the catalog without pre-creating prerequisites (ServiceAccount, ConfigMap, RBAC). Full symptom/cause/fix now lives in the MCP Catalog Knowledge Hub: Enablement & Troubleshooting.

3. MCP Server Not Appearing in Catalog Deployments

Usually a missing namespace label or a failed MCPServer CR. Full symptom/cause/fix now lives in the MCP Catalog Knowledge Hub: Enablement & Troubleshooting.

Playground Integration Issues

4. MCP Tab Empty in Playground

Symptom

In the Gen AI Studio playground settings, the MCP tab is empty or doesn't show any servers.

Cause

The gen-ai-aa-mcp-servers configmap not created, created in wrong namespace, or server URL is incorrect.

Fix

Create or verify the configmap in redhat-ods-applications namespace:

oc get configmap gen-ai-aa-mcp-servers -n redhat-ods-applications

# If not found, create it:
oc apply -f - <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: gen-ai-aa-mcp-servers
  namespace: redhat-ods-applications
data:
  OpenShift-MCP: |
    {
      "url": "http://ocp-mcp.my-mcp.svc.cluster.local:8080/mcp",
      "description": "Red Hat OpenShift MCP server"
    }
EOF

Verify the server URL matches the actual service name and port. Refresh the playground UI.

5. Model Returns Empty or Error for Tool Calls

Symptom

The model calls an MCP tool but returns an empty response, error, or gets stuck.

Cause

Small models (like llama-3.2-3b-instruct) struggle with large tool responses. For example, listing all pods with all attributes returns a massive JSON payload that exceeds the model's context window or reasoning capacity.

Fix

Option 1: Use focused prompts that return smaller responses:

  • Instead of: "list all pods in the cluster"
  • Try: "list all projects" or "list pods in namespace my-mcp"

Option 2: Use a larger tool-calling model with better reasoning capacity (e.g., mistral-7b-instruct or larger).

6. Workbench Can't Call MCP Servers

Symptom

Trying to call MCP servers from Python code in a workbench (e.g., using Langchain) fails or returns connection errors.

Cause

No documented way to call MCP servers from Python code in workbenches. The playground's exported code doesn't work for MCP calls.

Fix

Status: Open gap (as of June 2026). This is a known limitation documented by Fernando Lozano. A Python MCP client library for remote HTTP-based MCP servers is needed.

Workaround: Use the playground for MCP interactions, or implement a custom HTTP client following the MCP protocol spec.

Gateway Issues

7. tool/list Fails with OAuth Enabled

Symptom

When OAuth authentication is enabled on an MCP server, the tool/list call fails with a 401 Unauthorized error.

Cause

Long-running credentials needed for OAuth flows. Short-lived tokens expire before the tool/list operation completes.

Fix

Option 1: Use per-user tool/list with refreshable tokens (requires Gateway + Vault credential injection).

Option 2: Configure the MCP server with a skip-auth mode for tool/list only (tool invocations still require auth).

Option 3: Use service account credentials with longer TTL instead of user OAuth tokens.

8. Wristband Signature Verification Fails

Symptom

MCP Gateway returns 403 Forbidden with an error like "wristband signature verification failed".

Cause

Mismatch between the private key used by Authorino to sign wristbands and the public key configured in MCPGatewayExtension.spec.trustedHeadersKey.

Fix

Verify the key pair and secret references:

# Check the public key secret referenced in MCPGatewayExtension
oc get mcpgatewayextension mcp-gateway -n mcp-gateway -o yaml | grep trustedHeadersKey

# Verify the secret exists
oc get secret wristband-public-key -n mcp-gateway

# Regenerate keys if needed
openssl ecparam -name prime256v1 -genkey -noout -out wristband-private.pem
openssl ec -in wristband-private.pem -pubout -out wristband-public.pem

# Update secrets
oc create secret generic wristband-signing-key \
  -n mcp-gateway \
  --from-file=key.pem=wristband-private.pem \
  --dry-run=client -o yaml | oc apply -f -

oc create secret generic wristband-public-key \
  -n mcp-gateway \
  --from-file=key.pem=wristband-public.pem \
  --dry-run=client -o yaml | oc apply -f -

Operator Issues

9. Operator Pods Crashlooping

Symptom

MCP Lifecycle Operator or MCP Gateway Operator pods crashloop with CRD-related errors.

Cause

Missing CRD dependency. For example, deploying MCPServer CRs before MCPLO is installed, or deploying MCPGatewayExtension before Gateway API CRDs are present.

Fix

Verify installation order. Ensure all Tier 1 operators are healthy before installing Tier 2:

# Check Gateway API CRDs
oc get crd gateways.gateway.networking.k8s.io

# Check MCPLO installation
oc get pods -n mcp-lifecycle-operator-system

# Check Kuadrant
oc get kuadrant -n gateway-system

# If CRDs are missing, install in the correct order:
# 1. Service Mesh
# 2. cert-manager
# 3. RHCL (Kuadrant)
# 4. RHOAI
# 5. MCPLO
# 6. MCP Gateway Operator

10. "cluster-admin required" for ConfigMap Patches

Symptom

Non-admin users can't enable MCP Catalog or register MCP servers in Gen AI Studio.

Cause

The gen-ai-aa-mcp-servers configmap approach requires write access to redhat-ods-applications namespace, which is typically restricted to cluster admins.

Fix

Current (RHOAI 3.4): Cluster admin access required.

Future (RHOAI 3.5): This limitation will be resolved when the MCP Registry replaces the manual configmap approach. Registry-based registration will have proper RBAC controls.
