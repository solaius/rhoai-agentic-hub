CRDs & Configuration

MCP Gateway introduces two Kubernetes Custom Resource Definitions (CRDs) for declarative configuration: MCPServer and MCPGatewayExtension.

MCPServer CRD

The MCPServer CRD registers an MCP server with the gateway. It tells the gateway: "This backend service exposes MCP tools under this prefix."

MCPServer Fields

Field Type Description
metadata.name string Name of the MCPServer resource (used in logs and references).
metadata.namespace string Namespace where the MCPServer is defined (typically same as backend service).
spec.toolPrefix string Tool namespace this server owns (e.g., jira/, github/). All tools from this server are prefixed with this string.
spec.targetRef.group string API group of the target resource (typically gateway.networking.k8s.io).
spec.targetRef.kind string Kind of the target resource (typically HTTPRoute).
spec.targetRef.name string Name of the HTTPRoute that routes traffic to this backend MCP server.
spec.targetRef.namespace string Namespace of the HTTPRoute (can be different from MCPServer namespace).

MCPServer Example

apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: jira-mcp
  namespace: mcp-backends
spec:
  # All tools from this server are prefixed with "jira/"
  # Example: "jira/create_issue", "jira/search_issues"
  toolPrefix: jira/

  # Reference to the HTTPRoute that routes to this backend
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
    namespace: mcp-backends

What this does: When the MCP Gateway Controller sees this MCPServer, it:

  1. Reads the jira-mcp-route HTTPRoute to find the backend service
  2. Adds this backend to the Broker's configuration with the jira/ prefix
  3. Updates the Router so it knows to route jira/* tool calls to this backend

MCPGatewayExtension CRD

The MCPGatewayExtension CRD configures gateway-level behavior. This is where you specify which Gateway API Gateway resource the MCP Gateway should attach to, and any global broker/router settings.

MCPGatewayExtension Fields

Field Type Description
metadata.name string Name of the MCPGatewayExtension resource (typically one per gateway deployment).
metadata.namespace string Namespace where the MCPGatewayExtension is defined.
spec.targetRef.group string API group of the target Gateway (gateway.networking.k8s.io).
spec.targetRef.kind string Kind of the target resource (Gateway).
spec.targetRef.name string Name of the Gateway resource to attach to.
spec.targetRef.namespace string Namespace of the Gateway (can be different from MCPGatewayExtension namespace).
spec.broker object Broker configuration (optional, for advanced use cases).
spec.router object Router configuration (optional, for advanced use cases).

MCPGatewayExtension Example

apiVersion: kuadrant.io/v1alpha1
kind: MCPGatewayExtension
metadata:
  name: mcp-gateway-config
  namespace: mcp-system
spec:
  # Reference to the Gateway API Gateway resource
  targetRef:
    group: gateway.networking.k8s.io
    kind: Gateway
    name: istio-gateway
    namespace: gateway-system

  # Optional: Broker configuration
  broker:
    # Enable identity-aware tool filtering
    enableIdentityFiltering: true

  # Optional: Router configuration
  router:
    # Enable detailed request logging
    logLevel: debug

What this does: Tells the MCP Gateway Controller to configure the Broker and Router components to work with the istio-gateway in the gateway-system namespace.

Common Configuration Scenarios

Scenario 1: Register a Single MCP Server

Use case: You have a Jira MCP server running as a Kubernetes service and want to make it available through the gateway.

Step 1: Create an HTTPRoute for your backend service:

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: jira-mcp-route
  namespace: mcp-backends
spec:
  parentRefs:
  - name: istio-gateway
    namespace: gateway-system
  hostnames:
  - "gateway.example.com"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /jira-backend
    backendRefs:
    - name: jira-mcp-service
      port: 8080

Step 2: Create an MCPServer that references this HTTPRoute:

apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: jira-mcp
  namespace: mcp-backends
spec:
  toolPrefix: jira/
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
    namespace: mcp-backends

Result: Agents calling /mcp on the gateway will see tools like jira/create_issue, jira/search_issues. Calls to these tools are routed to the jira-mcp-service.

Scenario 2: Create a Virtual MCP Server Aggregating 3 Backends

Use case: You want to present a single "internal-tools" virtual MCP server that aggregates tools from Jira, GitHub, and Confluence backends.

Step 1: Register each backend as an MCPServer (as shown in Scenario 1):

apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: jira-mcp
  namespace: mcp-backends
spec:
  toolPrefix: jira/
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
---
apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: github-mcp
  namespace: mcp-backends
spec:
  toolPrefix: github/
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: github-mcp-route
---
apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: confluence-mcp
  namespace: mcp-backends
spec:
  toolPrefix: confluence/
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: confluence-mcp-route

Step 2: Create an MCPGatewayExtension that references all three:

apiVersion: kuadrant.io/v1alpha1
kind: MCPGatewayExtension
metadata:
  name: internal-tools-virtual-server
  namespace: mcp-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: Gateway
    name: istio-gateway
    namespace: gateway-system
  broker:
    # Enable virtual server mode — aggregate tools from multiple backends
    virtualServer:
      enabled: true
      name: "internal-tools"
      # List of MCPServer resources to aggregate
      backends:
      - name: jira-mcp
        namespace: mcp-backends
      - name: github-mcp
        namespace: mcp-backends
      - name: confluence-mcp
        namespace: mcp-backends

Result: Agents calling /mcp see a single combined tool list: jira/create_issue, github/create_pr, confluence/create_page, etc. The Broker aggregates them into one response.

Scenario 3: Add Tool Prefix to Avoid Naming Collisions

Use case: You have two different Slack MCP servers (one for internal Slack, one for customer Slack) and need to distinguish their tools.

Step 1: Register each with a unique toolPrefix:

apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: slack-internal
  namespace: mcp-backends
spec:
  # Prefix all tools with "slack-internal/"
  toolPrefix: slack-internal/
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: slack-internal-route
---
apiVersion: kuadrant.io/v1alpha1
kind: MCPServer
metadata:
  name: slack-customer
  namespace: mcp-backends
spec:
  # Prefix all tools with "slack-customer/"
  toolPrefix: slack-customer/
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: slack-customer-route

Result: Agents see tools like slack-internal/post_message and slack-customer/post_message. The Router knows which backend to send each request to based on the prefix.

Why this matters: Without prefixes, both backends would expose a tool called post_message, and the Router wouldn't know which one to call. The toolPrefix field makes every tool name globally unique within the gateway.

Best Practices

  • Use descriptive tool prefixes: Choose prefixes that make it obvious which backend a tool belongs to (jira/, github/, not j/, g/).
  • One MCPServer per backend service: Don't try to register multiple tool prefixes in a single MCPServer. Create one MCPServer per backend.
  • Use ReferenceGrants for cross-namespace references: If your MCPServer is in a different namespace than its HTTPRoute, you need a ReferenceGrant to allow the reference.
  • Keep MCPGatewayExtension minimal: Most deployments only need the targetRef field. Advanced broker/router config is optional and rarely needed.
  • Version your CRDs: MCP Gateway CRDs are currently v1alpha1. Expect schema changes before v1. Use kubectl apply --server-side to safely update existing resources.
