Deployment Guide

Step-by-step instructions for deploying MCP Gateway on OpenShift. This guide covers prerequisites, installation, verification, and production considerations.

Prerequisites

Before deploying MCP Gateway, ensure your OpenShift cluster meets these requirements:

Component Minimum Version Notes
OpenShift Container Platform 4.19+ Gateway API is GA in OCP 4.19. Earlier versions require manual Gateway API CRD installation.
Gateway API v1 (GA) Included by default in OCP 4.19+. Provides HTTPRoute, Gateway, and related CRDs.
OpenShift Service Mesh 3.x Provides Envoy-based ingress gateway and ext_proc support required by MCP Gateway Router.
Red Hat Connectivity Link (RHCL) 1.3.3+ Kuadrant platform providing Authorino (auth) and Limitador (rate limiting). MCP Gateway 0.6.0 TP ships in RHCL 1.3.3.
Version Support Gap — August 2026

RHOAI 3.5 (Aug 2026) supports OCP 4.19, but RHCL 1.4+ will likely require OCP 4.20+. This configuration (RHOAI 3.5 + RHCL + OCP 4.19) is untested and best-effort support only.

OCP 4.19 maintenance support ends December 2026 (only 4 months overlap). Customers are advised to upgrade to OCP 4.20+ for full support.

Installation Steps

Step 1: Enable Gateway API

If you're on OCP 4.19 or later, Gateway API CRDs are already installed. Verify:

oc get crd gateways.gateway.networking.k8s.io
oc get crd httproutes.gateway.networking.k8s.io
oc get crd referencegrants.gateway.networking.k8s.io

If you're on OCP 4.18 or earlier (not recommended), install Gateway API CRDs manually:

kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.0.0/standard-install.yaml

Step 2: Install OpenShift Service Mesh 3

Install the Service Mesh Operator from OperatorHub:

oc apply -f - <<EOF
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: servicemeshoperator
  namespace: openshift-operators
spec:
  channel: stable
  name: servicemeshoperator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Wait for the operator to be ready:

oc wait --for=condition=Ready pod -l name=istio-operator -n openshift-operators --timeout=300s

Create an Istio control plane in the gateway-system namespace:

oc new-project gateway-system

oc apply -f - <<EOF
apiVersion: operator.istio.io/v1alpha1
kind: Istio
metadata:
  name: default
  namespace: gateway-system
spec:
  version: v1.23
  namespace: gateway-system
  values:
    global:
      istioNamespace: gateway-system
EOF

Step 3: Install Red Hat Connectivity Link (Kuadrant)

Install the Kuadrant Operator from OperatorHub:

oc apply -f - <<EOF
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: kuadrant-operator
  namespace: openshift-operators
spec:
  channel: stable
  name: kuadrant-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Wait for the operator to be ready:

oc wait --for=condition=Ready pod -l control-plane=kuadrant-operator -n openshift-operators --timeout=300s

Create a Kuadrant instance in the gateway-system namespace:

oc apply -f - <<EOF
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata:
  name: kuadrant
  namespace: gateway-system
EOF

Step 4: Deploy MCP Gateway via Helm

Create a namespace for MCP Gateway components:

oc new-project mcp-system

Add the MCP Gateway Helm repository:

helm repo add kagenti https://ghcr.io/kagenti/charts
helm repo update

Install the MCP Gateway chart:

helm install mcp-gateway kagenti/mcp-gateway \
  --namespace mcp-system \
  --set gateway.namespace=gateway-system \
  --set gateway.name=istio-gateway

Wait for MCP Gateway components to be ready:

oc wait --for=condition=Ready pod -l app=mcp-gateway -n mcp-system --timeout=300s
Memory Allocation is Critical

"Memory allocation is critical for MCP Gateway — size replicas higher than typical API management."

— Red Hat engineering, network discussion with a major telecommunications provider (2026-05-08)

MCP Gateway components (especially the Broker) maintain in-memory caches for tool lists and routing tables. Production deployments should allocate at least 1 GB memory per replica for the Broker and Router components.

For high-traffic environments (100+ MCP servers, 1000+ tools), consider 2-4 GB per replica and horizontal scaling (3+ replicas with load balancing).

Verification

After installation, verify that MCP Gateway is responding correctly:

Check Component Status

oc get pods -n mcp-system
oc get mcpgatewayextension -n mcp-system
oc get mcpserver -n mcp-system

Test the MCP Endpoint

Expose the gateway temporarily for testing (production deployments use proper ingress):

oc port-forward -n gateway-system svc/istio-gateway 8080:80

In another terminal, send an MCP initialize request:

curl -X POST http://localhost:8080/mcp \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "method": "initialize",
    "params": {
      "protocolVersion": "2024-11-05",
      "capabilities": {},
      "clientInfo": {
        "name": "test-client",
        "version": "1.0.0"
      }
    },
    "id": 1
  }'

Expected response (server capabilities):

{
  "jsonrpc": "2.0",
  "result": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "tools": {}
    },
    "serverInfo": {
      "name": "mcp-gateway",
      "version": "0.7.1"
    }
  },
  "id": 1
}

Production Considerations

DNS Management

For external access, configure DNS to point to the OpenShift router or gateway service. Options:

  • OpenShift Cluster Ingress Operator: Automatically creates DNS records for Routes (not applicable for Gateway API)
  • ExternalDNS: Automates DNS record creation for Gateway resources (recommended for Gateway API)
  • Manual DNS: Create A/CNAME records pointing to the ingress gateway LoadBalancer IP

With RHCL's full feature set, DNSPolicy CRDs provide automated DNS management including health checks and geo-routing.

TLS Certificate Management

MCP Gateway requires TLS for production. Options:

  • Gateway-level TLS: Configure TLS termination directly on the Gateway resource by referencing a Kubernetes Secret containing the certificate and key.
  • TLSPolicy (RHCL): Use Kuadrant's TLSPolicy CRD to automate certificate provisioning via cert-manager and Let's Encrypt.
  • Route-level TLS: For simpler setups, use OpenShift Routes with automatic cert provisioning instead of Gateway API.

Example TLSPolicy:

apiVersion: kuadrant.io/v1alpha1
kind: TLSPolicy
metadata:
  name: mcp-gateway-tls
  namespace: mcp-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: Gateway
    name: istio-gateway
    namespace: gateway-system
  issuerRef:
    group: cert-manager.io
    kind: ClusterIssuer
    name: letsencrypt-prod

Cross-Namespace ReferenceGrants

If your MCPGatewayExtension is in a different namespace than the Gateway it references, you need a ReferenceGrant to allow the cross-namespace reference:

apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-mcp-to-gateway
  namespace: gateway-system
spec:
  from:
  - group: kuadrant.io
    kind: MCPGatewayExtension
    namespace: mcp-system
  to:
  - group: gateway.networking.k8s.io
    kind: Gateway

Without this, the MCP Gateway controller will fail to reconcile the MCPGatewayExtension and you'll see permission errors in the logs.

Backend TLS (Future)

Currently, MCP Gateway communicates with backend MCP servers over plaintext HTTP within the cluster. For end-to-end encryption, Kubernetes Gateway API supports BackendTLSPolicy (currently alpha).

When BackendTLSPolicy becomes stable, you'll be able to enforce TLS re-encryption for traffic between the gateway and backend MCP servers:

apiVersion: gateway.networking.k8s.io/v1alpha2
kind: BackendTLSPolicy
metadata:
  name: mcp-backend-tls
  namespace: mcp-system
spec:
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
  tls:
    caCertRefs:
    - name: jira-ca-cert
      kind: ConfigMap

For now, backend TLS is achievable via Service Mesh mTLS (requires all services to be in the mesh).
