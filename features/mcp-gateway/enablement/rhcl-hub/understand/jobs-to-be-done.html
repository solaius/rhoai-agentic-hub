Govern agent tool and model access

Persona: agentops-admin   Status: candidate

When I manage a fleet of agents,
I want to govern which tools and models each agent can access,
so I can enforce least-privilege, rate-limit token usage per team, and audit every agent action.

How MCP Gateway addresses it:

  • The job names MCP Gateway specifically as the tool-access layer of a three-layer governance model (model access, tool access, identity), distinct from AI Gateway (model routing) and SPIFFE/SPIRE (workload identity)
  • Tool-level access control is enforced through OAuth 2.0 token exchange: a per-tool, per-agent scoped token (Token 3 in the gateway's three-tier flow) is issued for each backend MCP server call, not a single broad credential
  • On-Behalf-Of (OBO) delegation gives an agent only the intersection of what the human user is allowed and what the agent is configured to use, which is the least-privilege enforcement this job asks for
  • The cryptographic act claim chain (RFC 8693 §4.1) plus Keycloak and gateway logging give a full audit trail: who authenticated, which agent acted on their behalf, which tool was called, and when
  • Session management and URL whitelisting at the gateway layer are the enforcement points; the "dumb MCP server" pattern means individual servers don't need their own auth logic for this to hold

Pillar: Agents (Agent Ops theme + MCP Ecosystem theme). Evidence status: not yet backed by direct customer evidence (tracked as "candidate" in the narrative inventory).
