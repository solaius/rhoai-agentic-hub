Gateway Integration

MCPLO-deployed servers register with the MCP Gateway manually today. Automating that handoff is explicitly scoped to GA, not Technology Preview, tracked as OCPMCP-347.

Today: Manual Registration

MCPLO has no install-time dependency on the MCP Gateway: it installs first (or alongside RHOAI) to manage server deployment, while the Gateway is downstream for routing traffic to already-deployed servers. That independence is deliberate, but it means the handoff between the two is currently manual:

Stage Mechanism today
Deploy MCPLO creates Deployment + Service from the MCPServer CRD
Register Manual An admin registers the server with the Gateway via a separate CRD (MCPGatewayExtension / HTTPRoute)
Configure Manual ConfigMap gen-ai-aa-mcp-servers in redhat-ods-applications, hand-edited to point Gen AI Studio at the Gateway or at individual server URLs

OCPMCP-347: The Formal Proposal (GA Scope)

aliok wrote a formal design document (May 2026, shared twice in #forum-mcp-lifecycle-operator) for how MCPLO-deployed MCPServer instances integrate with the MCP Gateway. It covers:

  • HTTPRoute-based server registration automatically registering deployed MCPServer instances with the gateway for external routing, auth, and observability
  • Server-Side Apply (SSA) for status updates, chosen specifically to avoid status field conflicts between MCPLO and the Gateway both writing to the same resource
  • Auth integration patterns between MCPLO-managed workloads and the Gateway's auth stack

An upstream proof-of-concept exists: PR #219 (WIP/XXL). This design drives OCPMCP-347.

GA scope, not TP

This was an explicit decision, made in Slack in May 2026, not an oversight: Gateway integration for MCPLO-deployed servers is scoped to GA. It is never Technology Preview scope. Do not read TP milestones on this hub as including automated Gateway registration.

Target Flow: Deploy, Register, Configure

The MCP Management Knowledge Hub's lifecycle-flow analysis lays out where this is headed for the ecosystem as a whole, once both Gateway integration and the MCP Registry exist:

Stage Target mechanism
Deploy MCPLO creates Deployment + Service and auto-registers the server in the Registry
Register Auto Registry creates a governed record, with an approval workflow
Configure Auto Registry approval flows into Gateway configuration; no manual ConfigMap edit

This target state depends on the MCP Registry existing (DP not landing in RHOAI 3.5 stable; TP targeted RHOAI 3.6 EA1) as well as OCPMCP-347 shipping. Neither has landed as of this writing. MCPLO's Gateway integration work is the deploy-side half of this chain; the Registry work is the governance-side half.

update_mode: SYSTEM_MANAGED, the Registry Handoff Mechanism

The MCP Registry's current data model already has the field that would carry this handoff, even though nothing writes to it automatically yet. Two independent enums govern a registered server: MCPServerStatus (ACTIVE, DEPRECATED, RETIRED) at the server level, and MCPPublishState (DRAFT, PUBLISHED, DEPRECATED, RETIRED) at the version level. Alongside them, update_mode (MANUAL or SYSTEM_MANAGED) controls whether a controller such as MCPLO may write to the record; it does not itself change lifecycle or publish state.

Connecting the two facts on this page: once OCPMCP-347's HTTPRoute-based registration and the Registry both exist, update_mode: SYSTEM_MANAGED is the mechanism by which a controller (MCPLO itself, or a Registry-side controller reacting to MCPLO's deploy events) could write deployment status back to the governed Registry record automatically, replacing today's manual ConfigMap step. This is an inference drawn from reading OCPMCP-347 and the Registry's data model together; it is not a confirmed integration decision, and no source describes this pairing explicitly yet.
