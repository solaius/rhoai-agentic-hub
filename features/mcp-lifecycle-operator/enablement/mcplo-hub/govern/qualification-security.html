Qualification & Security

MCPLO is qualified along two tracks that converge at GA, and carries a security posture that is strong on workload hardening but still has an in-cluster TLS gap. This page covers both, plus how the incoming stateless MCP spec shifts each.

Qualification Split: OCP Functional vs RHOAI Deployment

Team Scope Target Jira
OCP team Functional qualification: server deployment, discovery, and management via the MCPServer CRD TP: mid-July 2026 OCPSTRAT-3263 (TP), OCPSTRAT-2879 (GA)
RHOAI team Deployment and integration qualification: rollout via the RHOAI operator with all prerequisites satisfied TP: mid-August 2026 RHAISTRAT-1773 (TP), RHAISTRAT-1995 (GA)

The OCP-side TP lands about a month before the RHOAI-side TP because the two tracks qualify different things: OCP verifies MCPLO can deploy, discover, and manage target MCP servers; RHOAI verifies MCPLO rolls out smoothly through the RHOAI operator itself. Both converge at GA (joint OCP + RHOAI), targeted for RHOAI 3.6, code freeze Oct 23, 2026.

Where the docs live

Official MCPLO documentation lives in the RHOAI documentation, not OCP docs. This was discussed as OCP docs with RHOAI install links, but settled as RHOAI-owned since MCPLO ships via the RHOAI operator and OCP customers without RHOAI can fall back to the upstream project's own docs.

TLS: Edge Hardening vs In-Cluster Gap

Two distinct TLS stories apply to MCPLO, and they should not be conflated:

MCPLO endpoint TLS: now hardened (QE-verified July 2026)

The MCP Management Knowledge Hub's security model page reports that MCPLO's endpoint TLS now offers TLS 1.3 with ML-KEM (post-quantum key exchange), QE-verified in July 2026. This is the Gateway-facing, edge-terminated TLS posture.

In-cluster traffic is still unencrypted

status.address.url on the MCPServer resource still shows http://, not https://. MCP server traffic is unencrypted within the cluster today; the MCP Gateway handles TLS termination only at ingress. RHOAIENG-72309 ("Central TLS Profile consistency") tracks closing this gap: the module operator would pass TLS config to MCPLO, and MCPLO would handle TLS from environment variables. The currently-empty MCPLifecycleOperatorCommonSpec on the DSC CR is the likely vehicle for that TLS profile configuration, and upstream PR #257 adds the TLS environment propagation.

Full detail: Architecture: The TLS Gap.

Security Posture Against External Frameworks

Two external frameworks, published mid-2026, now form the enterprise compliance baseline PMs and customers measure MCP deployments against.

NSA/CISA MCP Security Guidance (June 2026)

Key recommendations: every MCP server should run in an isolated container or microVM; egress allowlists to prevent SSRF; input validation against well-defined schemas; host-level shell execution blocked at the container boundary.

MCPLO's Restricted Pod Security Standard defaults (non-root, read-only rootfs, dropped capabilities, seccomp RuntimeDefault) align with these recommendations. The NetworkPolicy gap, ingress-only with no source restrictions and no egress restrictions, is a partial miss against the egress-allowlist recommendation.

OWASP MCP Top 10

The first dedicated OWASP Top 10 for MCP identifies Credential/Secret Exposure (#1), Privilege Creep, Tool Poisoning, Command Injection, Inadequate Authentication, and Shadow MCP Servers as the key MCP-specific risks. Real-world threat data from H1 2026 backs the urgency: 30+ CVEs filed against MCP infrastructure (Jan-Feb 2026), a 78.3% attack success rate when 5 MCP servers are connected to one agent (Palo Alto Unit 42), and 36.7% of over 7,000 surveyed MCP servers potentially SSRF-vulnerable (BlueRock Security).

MCPLO's per-server ServiceAccount scoping (spec.runtime.security.serviceAccountName) and its refusal to auto-create ServiceAccounts both work against Privilege Creep specifically. Credential Exposure and Tool Poisoning are largely out of MCPLO's control surface; they are addressed at the Gateway (auth, wristband signing) and at the individual MCP server implementation, not at the deployment layer.

Stateless Spec Shift: Qualification & Security Implications

The MCP 2026-07-28 spec, a Release Candidate locked May 21, 2026, removes the initialize/initialized handshake and the session header, making every request self-contained (protocol version, client info, and capabilities travel in _meta on each call). New Mcp-Method and Mcp-Name routing headers let gateways route without body parsing.

  • Removing session stickiness removes a class of session-hijacking and session-fixation concerns that qualification would otherwise have to cover.
  • Self-contained, per-request auth context means every request must independently validate; there is no persisted session state to lean on for authorization decisions.
  • The new routing headers give gateways policy-enforcement hooks without body inspection, useful for security policy but also a new interface that qualification needs to cover.
  • New caching (ttlMs, cacheScope) and W3C Trace Context propagation hooks are new protocol-level surface, not yet assessed against the NSA/OWASP frameworks above.

Full architectural detail on the stateless shift, including its impact on scaling and rollout strategy, lives on Architecture; a cross-cutting analysis of the full breaking-change set (Gateway, Registry, Catalog/MCPLO/Studio) is published as MCP 2026-07-28 RC: RHOAI Impact Analysis.
