Architecture

The MCP Lifecycle Operator (MCPLO) reaches RHOAI through a three-repo integration pattern, sits inside the DataScienceCluster (DSC) component model, and reconciles the MCPServer CRD into a running, security-hardened MCP server.

The Three-Repo Model

RHOAI's integration of MCPLO spans three repositories, each with a distinct job:

Repository Role
kubernetes-sigs/mcp-lifecycle-operator Upstream SIG Apps operator. Reconciles MCPServer custom resources, the deployment primitive itself.
opendatahub-io/mcp-lifecycle-module-operator ODH module wrapper. Reconciles MCPLifecycleOperator CRs and deploys the upstream controller as an operand.
opendatahub-io/opendatahub-operator Parent operator. Embeds the module handler in the DataScienceCluster (DSC) CR as the mcplifecycleoperator component.

The downstream fork openshift/mcp-lifecycle-operator (264 commits) adds CI/build infrastructure (Dockerfile.ocp, .ci-operator.yaml, .tekton/ pipelines) on top of the upstream repo; core operator code tracks upstream. That midstream repo moved from the openshift/ org to the opendatahub-io/ org in June 2026.

Two-Tier Image Pattern

Two separate container images are involved, matching the module-operator/operand split:

  • Module operator image (RELATED_IMAGE_ODH_MCP_LIFECYCLE_MODULE_OPERATOR_IMAGE): runs in the RHOAI namespace, reconciles MCPLifecycleOperator CRs
  • Lifecycle operator image (RELATED_IMAGE_ODH_MCP_LIFECYCLE_OPERATOR_IMAGE): the actual MCPLO controller, deployed by the module operator as the operand that watches MCPServer CRs

Manifest sources are pinned to commit SHAs in get_all_manifests.sh (ODH branch: main, RHOAI branch: rhoai-3.5).

DSC Integration

MCPLO integrates via the DataScienceCluster (DSC) CR as the component mcplifecycleoperator. The DSC API defines DSCMCPLifecycleOperator with a standard ManagementSpec (Managed/Removed toggle per component) plus MCPLifecycleOperatorCommonSpec, currently an empty placeholder for future configuration (the likely vehicle for TLS profile settings; see below). This component registration and the reconciliation chain below are present in the opendatahub-operator source today. Productizing MCPLO as an automatically-deployed RHOAI Tech Preview platform component, not bundled inside AI Hub but installed the same way as FeastOperator and LlamaStackOperator, is proposed under RHAIRFE-1456, status Approved, not started in engineering as of the RHCL hub's 2026-07-10 verification. Today's actual install remains a standalone operator: manual, not listed in OperatorHub, not available via OLM, gated on an RHOAI restricted use entitlement.

DataScienceCluster CR
  → DSC Reconciler (opendatahub-operator)
    → MCPLifecycleOperator module handler
      → checks ManagementState (Managed/Removed)
      → deploys kustomize manifests, creates MCPLifecycleOperator CR
        → module operator reconciles, deploys the MCPLO controller
          → MCPLO watches MCPServer CRs, creates Deployment + Service + NetworkPolicy

The DSC type names and reconciliation chain above are drawn from source-code-level research against opendatahub-operator (fact-mcplo-three-repo-architecture.md, research/02-architecture.md, both 2026-07-11): code that exists, distinct from the still-pending productization described above.

CRD to Controller to Workload

The MCPServer CRD (mcp.x-k8s.io/v1alpha1) is the deployment primitive. Its reconciliation loop:

  1. Validate config: check storage mounts and env references for ConfigMap/Secret existence. Missing references produce a permanent Accepted=False validation error (no requeue); transient errors get exponential backoff.
  2. Reconcile Deployment: build the pod template with security defaults, apply a config-hash annotation that drives rolling updates on Secret/ConfigMap changes, create or update via server-side apply.
  3. Reconcile Service: ClusterIP, port mcp, session affinity governed by the stateless flag.
  4. Reconcile NetworkPolicy: ingress-only on the configured TCP port.
  5. MCP protocol handshake: once the Deployment is available, connect via StreamableClientTransport (10s timeout), send an Initialize request, extract server info and capabilities. Auth errors (401/403) count as reachable. Failure backs off from 10s to a 2-minute cap, up to 10 retries.

Auto-created resources share the MCPServer's name: a Deployment (non-root, read-only rootfs, all capabilities dropped, seccomp RuntimeDefault), a Service, a NetworkPolicy, and pods labeled mcp-server=<name>. Status surfaces Accepted (reason: Valid) and Ready (reason: Available) conditions, plus status.address.url (pattern http://<name>.<namespace>.svc.cluster.local:<port>/mcp).

Deploy button click creates an MCPServer CR; MCPLO validates, sets Accepted, creates the workload, waits for readiness, performs the handshake, then sets Ready and populates status.address.url. The Gateway then federates deployed servers behind a single endpoint (per-server MCPServerRegistration targeting an HTTPRoute, tool-level access control, JWT auth via Keycloak). Deployed servers then appear in Gen AI Studio, where users combine models and MCP tools, with Llama Stack handling tool invocation during inference.

Security Architecture

Pod Security Standards default to Restricted: AllowPrivilegeEscalation: false, ReadOnlyRootFilesystem: true, RunAsNonRoot: true, all capabilities dropped, SeccompProfile: RuntimeDefault. Users can override via spec.runtime.security.securityContext. RBAC scoping is per-server ServiceAccount via spec.runtime.security.serviceAccountName, defaulting to the namespace's default SA; the operator itself cannot auto-create ServiceAccounts, a deliberate security decision. NetworkPolicy today is ingress-only on the configured port with no source restrictions and no egress restrictions, a known gap for production environments that need source-scoped policy.

The TLS Gap

Current status: status.address.url shows http://, not https://. MCP server traffic is unencrypted within the cluster today; the MCP Gateway handles TLS termination at ingress. RHOAIENG-72309 ("Central TLS Profile consistency") addresses this in-cluster gap: the module operator passes TLS config to MCPLO and MCPLO handles TLS from environment variables. The currently-empty MCPLifecycleOperatorCommonSpec is the likely vehicle for that TLS profile configuration, and upstream PR #257 adds TLS environment propagation.

Stateless Spec Shift: Impact on Scaling

The MCP 2026-07-28 spec, a Release Candidate locked May 21, 2026, makes MCP stateless at the protocol layer: the initialize/initialized handshake is removed, the session header is removed, and every request becomes self-contained (protocol version, client info, and capabilities travel in _meta on each call). A new server/discover method lets clients fetch capabilities on demand. New Mcp-Method and Mcp-Name routing headers let gateways route without body parsing, and caching (ttlMs, cacheScope) and W3C Trace Context propagation land as protocol-level hooks.

For MCPLO, this is a tailwind:

  • No session stickiness needed. MCP servers become standard HTTP microservices, exactly what Kubernetes operators are built to manage.
  • The MCPConfig.Stateless field (added in v0.1.0 to v0.2.0 for session affinity control) may need revisiting now that the protocol itself is stateless.
  • Standard RollingUpdate with maxSurge/maxUnavailable works without sticky-session concerns; progressive delivery tooling like Flagger applies cleanly.
  • Routing headers enable gateway-level policy without body inspection; caching and tracing standards give the operator hooks for managed observability.

A cross-cutting analysis of the full breaking-change set (Gateway, Registry, Catalog/MCPLO/Studio) is published as a standalone artifact: MCP 2026-07-28 RC: RHOAI Impact Analysis.
