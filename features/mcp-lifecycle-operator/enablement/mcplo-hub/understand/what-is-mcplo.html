What Is the MCP Lifecycle Operator?

The MCP Lifecycle Operator (MCPLO) is a Kubernetes-native operator that provides a declarative API, the MCPServer custom resource, to deploy, manage, and safely roll out MCP servers with security-hardened defaults. It is the deployment primitive underneath the MCP ecosystem on Red Hat OpenShift AI (RHOAI) and OpenShift.

Repository: kubernetes-sigs/mcp-lifecycle-operator
Current version: v0.2.0 (released June 17, 2026), API mcp.x-k8s.io/v1alpha1
Status: Developer Preview (DP) shipped; Technology Preview (TP) targeted OCP-side mid-July 2026 and RHOAI-side mid-August 2026; GA targeted RHOAI 3.6 (code freeze Oct 23, 2026)

Purpose: Deployment Primitive, Not Governance Owner

MCPLO is the deployment layer: it deploys MCP servers. It does NOT own governance or lifecycle state. That is the job of the MCP Registry, which represents the governed MCP asset and its deployment and runtime associations once MCPLO reports is_deployed. Concretely:

  • MCPLO reconciles the MCPServer CRD into running Kubernetes workloads and reports deployment state back to the Registry
  • The Registry (still in Design phase, DP not landing in RHOAI 3.5 stable, TP targeted RHOAI 3.6 EA1) is the intended system of record for identity, approval, and lifecycle state
  • The MCP Catalog is the discovery layer in front of MCPLO: it disables its Deploy button when the MCPServer CRD is absent from the cluster

In short: Catalog discovers, MCPLO deploys, Registry governs, Gateway connects.

Component Flow

MCPServer CRD
Custom resource, one per server
→
Controller
Watches CRDs, validates config
→
Deployment + Service + NetworkPolicy
Running, security-hardened MCP server

When an MCPServer resource is created, the operator validates referenced ConfigMaps and Secrets, then provisions a Deployment, a Service, and a NetworkPolicy with security-hardened defaults (non-root, read-only filesystem, dropped capabilities, seccomp RuntimeDefault). It injects a default readiness probe and performs an MCP protocol-level handshake to verify the server is actually serving MCP, not just listening on a port, before marking it Ready. Full field-level detail lives on Architecture.

What It Is NOT

  • Not a governance layer. No approval workflow, no identity/trust tracking; that is Registry scope.
  • Not a gateway. MCPLO does not route external traffic, terminate TLS at the edge, or provide tool-level auth; that is MCP Gateway scope, and it sits downstream of MCPLO.
  • Not an auto-provisioner of identity. The operator cannot auto-create ServiceAccounts (a deliberate security decision from the upstream project); users bring their own ServiceAccount with appropriate RBAC bindings via spec.runtime.security.serviceAccountName.
  • Not a stdio transport. SSE and streamable-HTTP only, an explicit upstream decision, not a gap.
  • Not something that deploys servers by default. Installing MCPLO deploys zero MCP servers; it presents a catalog of available servers during setup, and the user selects which to deploy. Additional servers install as Day 2 operations via the catalog UI or CRDs.

Independence from Gateway

No installation dependency on MCP Gateway. MCPLO installs first (or alongside RHOAI) to manage server deployment; the MCP Gateway is downstream for routing traffic to already-deployed servers. MCPLO also ships with a standard OCP entitlement, not an RHOAI-only one: base OpenShift customers get MCPLO's deployment capability without RHOAI, and can upgrade to RHOAI later (which adds Registry, Authorino, and Limitador) without losing MCPLO functionality.

Gateway integration for MCPLO-deployed servers (HTTPRoute-based registration with the MCP Gateway) is explicitly scoped to GA, tracked as OCPMCP-347. It is never Technology Preview scope.

Where It Sits in the RHOAI Component Set

Today, MCPLO installs as a standalone operator: manual, outside OperatorHub and OLM, under the standard OpenShift entitlement, and not bundled inside AI Hub. RHOAI plans to productize it as an automatically-deployed DataScienceCluster (DSC) component, installed the same way as FeastOperator and LlamaStackOperator, under RHAIRFE-1456, status Approved, not started in engineering as of the RHCL hub's 2026-07-10 verification; the underlying DSC component code already exists in the opendatahub-operator source. See Architecture for the full three-repo integration model and code-level detail, and MCP Ecosystem Hub: Component Directory for how MCPLO compares to Gateway, Catalog, and Registry side by side.
