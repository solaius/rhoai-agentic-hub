MVP Scope & Prototype

What the MCP Registry's initial scope is meant to cover, a sketch of the data model as an API surface, and the interactive UI prototype that shows what a Registry list/detail/version experience could look like. Everything on this page is design work, not shipped software.

Scope, not delivery

The bullets and sketches below describe what the initial Registry release is scoped to cover on the way to Tech Preview (targeted RHOAI 3.6 EA1). No Dev Preview has shipped and none is landing in RHOAI 3.5 stable.

MVP Scope

  • Governed registration of MCP servers
  • Metadata-first MCP records
  • Lifecycle state tracking
  • Certification and approval visibility
  • RBAC-aware visibility and governance
  • Support for internal and external MCP representation
  • Conceptual integration with catalog, deployment, gateway, Studio

Read "certification and approval visibility" carefully. This scope bullet names an intent, not a data-model field. The current two-enum data model (MCPServerStatus, MCPPublishState) has no approval_status or certification_status field. Approval and certification as independently-tracked states belong to the deferred four-track proposal; see Data Model for the bounded proposal section. This scope bullet is the stated goal the design is working toward, not a claim that the field exists today.

Source: RHCL hub, govern/registry.html, "MVP Scope (Registry DP / TP 3.6 EA1)" (data-verified 2026-07-11).

Data Model as an API/Entity Sketch

The current data model, restated as a rough record shape rather than a table. This is illustrative, not a published schema; the canonical field list lives on Data Model.

MCPServer {
  name: string
  description: string
  status: MCPServerStatus            // active | deprecated | retired
  tags: string[]
  aliases: string[]
  workspace: string                  // MVP visibility boundary
  update_mode: MCPUpdateMode         // MANUAL | SYSTEM_MANAGED
  last_registered_version: string
}

MCPServerVersion {
  version: string
  server_json: object                // immutable canonical MCP payload
                                      // (upstream ServerJSON schema)
  publish_state: MCPPublishState     // draft | published | deprecated | retired
  tags: string[]
  source: string
  run_id: string
  is_deployed: boolean               // reported by the Lifecycle Operator
}

MCPObservedTool / MCPObservedToolSnapshot {
  // cached, non-authoritative tool metadata observed from a live endpoint
  // populated from tools/list or gateway sync
  // supports version pinning and change detection
}

Field list per Data Model, sourced from the MLflow MCP Registry Data Model Proposal and the MCP Registry MVP Requirements doc. No API endpoints have been designed publicly yet; this is an entity sketch, not an interface contract.

UI Prototype

An interactive, client-rendered UI prototype (MLflow-style, using the browser tab title "MLflow" paired with "MCP Servers" in the mockup itself) shows what a Registry list/detail/version experience could look like. It is a design artifact, not a working product; there is no backend behind it.

What the prototype shows
  • A server list view with search/filter, mirroring MLflow's model registry UX pattern
  • A server detail view showing an individual MCPServer record
  • A version detail view showing an individual MCPServerVersion, including its publish state
  • Create/edit modals for both servers and versions
  • A lifecycle-state transition modal, illustrating how a governance state change could be surfaced in the UI
  • Two iterations: an earlier v0.1 mockup and the more developed v0.2 mockup (the prototype's default entry point)

Live prototype: solaius.github.io/rhoai-agentic-hub-pages/mcp-registry/ui-prototype/

Source: registry-ui-prototype artifact, migrated from the prior ai-asset-registry prototype (clean batch). Published publicly via publish/manifest.yaml.

MLflow Upstream Collaboration Status

Upstream design partnership, still in progress

Red Hat is collaborating upstream with MLflow and Databricks on MCP registry design. The data model proposal (including the four-track governance model) is going through MLflow's RFC approval process; it is a brain-dump-stage document, not a finalized or merged design. This collaboration also informs the future generalized AI asset registry architecture, beyond just MCP.

Databricks' review process favors small, focused PRs, which is in tension with proposing a full four-track governance model in one step. That tension is an open scoping question (what belongs in the initial MLflow upstream proposal now versus later), and is one reason the current design scope collapsed to two enums plus an update mode rather than proposing all four tracks at once.

Still Undecided for the MVP

The data model proposal describes the four-track governance model, but these specifics remain open for the initial release cut:

  1. What exact metadata should be mandatory on a registry record?
  2. Which of the four proposed tracks' lifecycle states are actually required for the initial cut, versus deferred to a later release?
  3. Should certification programs be backed by an admin-managed allowlist?
  4. Should lifecycle history (an audit trail of state transitions) ship in the MVP or be deferred?
  5. Do approval, verification, and certification transitions need distinct permissions beyond general update permissions?
  6. Do Registry entities need EntityAssociationType support in the MVP, or can it wait?

Six open decisions per question-mcp-registry-mvp-data-model-gaps.md. Originally scoped against a "3.5 Dev Preview" cut; the underlying decisions are unchanged, but that release label is superseded by the 2026-07-11 owner ruling (Dev Preview will not land in RHOAI 3.5 stable; Tech Preview is targeted for RHOAI 3.6 EA1) and is not repeated here.

Requirements Source

The MVP scope above traces back to an MCP Registry MVP requirements document covering scope and integration expectations against the Catalog, Gateway, and Lifecycle Operator. That document's own title still reads "3.5 Dev Preview requirements"; treat that label as historical framing, not a current delivery commitment. The requirements it captures have not been re-litigated; only the release vehicle has changed.
