Integrations

The MCP Registry does not stand alone: it is designed to sit between the MCP Lifecycle Operator, the MCP Catalog, and the MCP Gateway, governing what each of them does without deploying, discovering, or routing anything itself. This page maps the three Registry-facing integration flows, each flow's Jira tracking and timing, and the open questions still blocking a full answer.

Design, not delivery

None of the flows below are running in a shipped product today. "Design resolved" means a design decision has been made, not that code exists. "Planned" timing reflects the current release-train intent (Tech Preview targeted RHOAI 3.6 EA1, GA targeted RHOAI 3.6, code freeze October 23, 2026), not a shipped Dev Preview.

The Three Flows

Flow: MCPLO → Registry (Design Resolved)

MCPLO deploys MCP server
Pod + Service created
MCPLO auto-registers server
Creates a "discovered" entry in the Registry
update_mode: SYSTEM_MANAGED
Only description and tags remain user-editable

An earlier open question asked whether the MCP Lifecycle Operator should create read-only, system-managed registry entries (Chris Hambridge and Matt Prahl feedback on the data model proposal comments). It is answered in the current design: the MCPUpdateMode enum (MANUAL vs. SYSTEM_MANAGED) governs whether a registry entry's mutable fields are user-edited or kept in sync by another system. MCPLO auto-registers deployed servers as "discovered" entries with update_mode: SYSTEM_MANAGED, distinct from MANUAL records an admin enters by hand.

Tracked under RHAISTRAT-1762 (Operational Governance), the in-flight strategy feature covering how the Registry becomes the operational source of truth for every MCP server in use; no separate Jira key is broken out for this specific mechanism.

Flow: Catalog ↔ Registry (Planned, RHOAI 3.6 EA1)

MCP servers configured in the Catalog can be registered into the Registry for governance tracking. Tracked as RHAISTRAT-2027, which supersedes a closed and obsolete predecessor (RHAISTRAT-1938), with fix version RHOAI 3.6 EA1. Current scope delivers lifecycle-status enrichment only; approval, verification, and certification tracks are proposed but not yet modeled (see Lifecycle & Governance for the bounded four-track proposal).

  • Registration: Catalog-discovered servers are submitted to the Registry for governance tracking.
  • Governance metadata: The Registry adds MCPServerStatus / MCPPublishState lifecycle state; approval, certification, and ownership tracks are proposed but not yet modeled in the current data model.
  • Lifecycle-status enrichment: The Registry's lifecycle state (active or deprecated) informs how servers are surfaced back in the Catalog UI; trust tiers and verification status are a later-phase capability, not delivered by this integration.
  • Deprecation tracking: Registry lifecycle state (deprecated or retired) can signal servers that should be removed from active use.
Key principle: Flow is always Catalog to Registry for registration, never the reverse. The Registry is the authoritative governance layer; the Catalog is the discovery layer.

Flow: Registry → Gateway (Planned, GA target)

MCP server registered
In the Registry (manual or auto)
Approval workflow
Governance review and approval
Registry updates Gateway
Auto-config MCPGatewayExtension
Gateway routes traffic
Tools visible to Studio

Today, an admin manually registers a deployed server with the Gateway via an MCPServerRegistration CRD and manually updates the gen-ai-aa-mcp-servers ConfigMap. The target design replaces that manual bridge: once a Registry record clears approval, the Registry auto-configures Gateway routing and auth policy directly, with no manual ConfigMap and no cluster-admin privileges required for that step.

Auto-config timing is a GA target, tracked under RHAISTRAT-1993 (General Availability), aligned to RHOAI 3.6 (code freeze October 23, 2026). Until then, this remains the current governance gap called out on the RHCL hub: the manual ConfigMap bridge is the only integration point between deployed MCP servers and Gen AI Studio.

Open Questions

Resolved: MCPLO write access to the Registry

"Should MCPLO create read-only 'discovered' registry entries, or get full governance write access?" is resolved via update_mode (MANUAL vs. SYSTEM_MANAGED); see the MCPLO → Registry flow above. This was the third of three open questions on registry state propagation; the other two remain open below.

State propagation to Gateway and Catalog

  • How should Registry-governed state inform Gateway behavior? For example, should an unapproved or unverified server be routable at all, or should the Gateway only ever see approved records?
  • How should Registry state influence Catalog surfacing? For example, does a not-yet-published record show up next to a published one, or stay hidden until it clears governance? (This question predates the current two-enum model, whose MCPPublishState has no separate "Candidate" step; the underlying surfacing question is unchanged even though the original wording referenced states the current model does not have.)

Source: question-mcp-registry-state-propagation.md, open since 2026-07-06.

MLflow / Databricks upstream scope split

A separate, open scoping question sits upstream of the propagation questions above: how much of the Registry design goes into the initial MLflow upstream proposal now, versus later. Databricks' review process favors small, focused PRs, which is in tension with proposing the full four-track governance model in one step. That tension is one reason the current design scope collapsed to two enums plus an update mode rather than proposing all four tracks at once, and it is not yet resolved for what comes after the initial cut.

Source: question-mlflow-upstream-registry-scope.md, open since 2026-07-06.
