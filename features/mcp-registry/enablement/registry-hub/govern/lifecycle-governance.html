Lifecycle & Governance

The MCP Registry sits in the middle of a seven-stage lifecycle that runs from discovery through retirement. This page walks that lifecycle end to end, states the current governance model exactly as it exists today (two enums plus an update mode, nothing more), and draws a bounded line around what is proposed for later versus what is real now.

Nothing on this page is shipped

The Registry has no code and no shipped Dev Preview. "Today" below means RHOAI 3.4, the last shipped state. "Target" means the current design intent the Registry is working toward on the way to Tech Preview (targeted RHOAI 3.6 EA1); it is not a claim about RHOAI 3.5, which does not carry a Registry Dev Preview.

Target (Design)

Stage Mechanism
Discover MCP Catalog UI, enhanced with lifecycle-status enrichment sourced from the Registry
Deploy MCPLO creates Deployment + Service, then auto-registers the server in the Registry
Register Auto Registry creates a governed record; approval workflow gates it
Configure Auto Registry approval flows into Gateway configuration
Consume Gen AI Studio reads from the Registry API, no ConfigMap needed
Monitor Unified observability: per-tool metrics, health status, usage analytics
Retire Registry lifecycle state moves to deprecated, then retired (auto-revokes Gateway access)
Key difference: Today, each handoff (Deploy to Register to Configure) requires manual admin intervention. The target design makes the Registry the single source of truth that drives automation across the rest of the ecosystem, without the Registry deploying or enforcing anything itself.

Current Governance Model (Two Enums, Not Four Tracks)

The MCP Registry's current data model uses two independent enums plus an update mode. It does not use the four-track governance model described in early design proposals; that model is a separate, clearly-bounded proposal covered further down this page.

Enum States Scope
MCPServerStatus active, deprecated, retired Asset-level (server)
MCPPublishState draft, published, deprecated, retired Version-level
MCPUpdateMode MANUAL, SYSTEM_MANAGED Controls whether a controller (for example, the MCP Lifecycle Operator) can system-manage the record

This is the whole current governance surface: two enums plus an update mode. No approval_status, verification_status, or certification_status field exists in the current model. See Data Model for the full entity and property list.

Governance Invariants

  • MCPServerStatus and MCPPublishState are independent enums; no approval, verification, or certification field exists in the current model to gate either transition.
  • workspace on a version resolves to the same value as the parent MCPServer.workspace.
  • update_mode (MANUAL or SYSTEM_MANAGED) controls whether a controller such as MCPLO may write to the record; it does not itself change lifecycle or publish state.

Registry-Owned vs Catalog-Surfaced State

The Registry, not the Catalog, is the authoritative owner of governance state. The Catalog only ever reflects a narrow slice of that state back to its own UI; it never holds an independent copy of it.

Owner State
Registry-owned (canonical) MCPServerStatus (active, deprecated, retired), MCPPublishState (draft, published, deprecated, retired), update_mode, and the immutable server_json payload
Catalog-surfaced (derived, read-only) Lifecycle-status enrichment only: active/deprecated flagging sourced from the Registry's MCPServerStatus. Trust tiers and verification status are not surfaced; that remains out of scope until the four-track model below ships.

The Catalog-Registry integration that carries this enrichment is planned for RHOAI 3.6 EA1 (RHAISTRAT-2027); see Integrations for the full flow. Flow is always Catalog to Registry for registration, and Registry to Catalog for status enrichment; the Registry is the authoritative governance layer, the Catalog is the discovery layer, and the direction never reverses.

Phasing: Current Model vs Post-MVP Proposal

Proposed: four-track model (deferred, not shipped)

Earlier design drafts described four independent governance tracks (lifecycle, approval, verification, and certification), each with its own state enumeration. That is a deferred proposal, not the current model. It comes from the MLflow MCP Registry Data Model Proposal, a brain-dump-stage document, not a finalized design. Nothing in this section is implemented and nothing here belongs in the current-model tables above.

Proposed Track Proposed States
Lifecycle State Draft → Candidate → Published → Deprecated → Retired
Approval Status Draft → Pending → Approved → Rejected → Revoked
Verification Status Unverified → Verified
Certification Status None → Candidate → Certified → Expired → Revoked

The proposal's design intent: a version could be verified but not yet approved, or approved but not yet certified, independent progression per track, constrained by one governance invariant: lifecycle_state=PUBLISHED implies approval_status=APPROVED; a REJECTED or REVOKED approval status cannot coexist with a PUBLISHED lifecycle state.

Not abandoned, deferred

A doc comment thread on the proposal states the phasing intent directly: "For MVP/DP we should follow the same path as they are taking for the skills registry upstream to speed up approval/implementation... we should follow immediately after with the governance implementation as proposed here." That is the source of the current scope: the lighter two-enum model ships first, following the same lighter-weight path taken for the upstream skills registry, with the fuller four-track model planned as a deliberate follow-on, not a scrapped idea.

The four-track proposal also supersedes an even older, pre-proposal linear seven-state governance flow (Draft → Candidate → Verified → Approved → Published → Deprecated → Retired) that was documented before the four-track split. That linear seven-state flow was a governance-state sequence for a single server version; it is unrelated to the seven ecosystem lifecycle stages (Discover through Retire) at the top of this page, which describe stages across components, not governance states on one record. Neither the old linear flow nor the four-track split is what the current two-enum model implements; both are historical or proposed, not current.
