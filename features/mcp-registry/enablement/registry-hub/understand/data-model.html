Data Model

The MCP Registry's current data model has two entities and three enums. A separate, clearly-bounded proposal describes a four-track governance model; that proposal is deferred, not shipped, and never appears mixed into the tables below.

Nothing on this page is shipped

The Registry has no code and no shipped Dev Preview. "Current model" below means the current design, the model the Tech Preview (targeted RHOAI 3.6 EA1) is expected to implement, not a running system.

Entity Flow (Current Model)

MCPServer
Logical governed asset
Scoped to a workspace
→
MCPServerVersion
Immutable payload
+ mutable governance metadata

Two additional entities separate cached, non-authoritative tool metadata from the canonical registry payload: MCPObservedTool, a tool as observed from a live MCP endpoint, tracked independently of the server's canonical server_json; and MCPObservedToolSnapshot, a point-in-time snapshot of observed tool metadata intended for display and search UX. Both are populated either manually or by syncing an MCP client's list_tools() output, or from a gateway sync.

Entities (Current Model)

Entity Role Key Properties
MCPServer Logical governed asset, scoped to a workspace name, description, status, tags, aliases, workspace, update_mode, last_registered_version
MCPServerVersion Immutable MCP payload plus mutable governance metadata version, server_json (immutable), publish_state, tags, source, run_id, is_deployed
MCPObservedTool / MCPObservedToolSnapshot Cached tool metadata observed from a running server, tracked independently of the canonical payload Populated from tools/list or gateway sync; supports version pinning and change detection

Source: the MLflow MCP Registry Data Model Proposal and the MCP Registry MVP Requirements doc. server_json is the canonical immutable payload following the upstream MCP registry schema.

Governance Enums (Current Model)

Enum States Scope
MCPServerStatus active, deprecated, retired Asset-level (server)
MCPPublishState draft, published, deprecated, retired Version-level
MCPUpdateMode MANUAL, SYSTEM_MANAGED Controls whether a controller (for example, the MCP Lifecycle Operator) can system-manage the record

This is the whole current governance surface: two enums plus an update mode. No approval_status, verification_status, or certification_status field exists in the current model.

Design Principles (Current Model)

  • server_json is the canonical, immutable payload on MCPServerVersion.
  • Governance metadata is first-class and kept separate from tags: it is not encoded as freeform labels.
  • Workspace is the MVP visibility boundary for RBAC-aware access.
  • Deployment visibility is minimal: an is_deployed boolean only, reported by the MCP Lifecycle Operator, not a full deployment/runtime model.
Proposed: four-track model

Earlier design drafts described four independent governance tracks (lifecycle, approval, verification, and certification), each with its own state enumeration. That is a deferred proposal, not the current model. It comes from the MLflow MCP Registry Data Model Proposal, a brain-dump-stage document, not a finalized design. Nothing in this section is implemented and nothing here belongs in the current-model tables above.

Proposed Track Proposed States
Lifecycle State Draft → Candidate → Published → Deprecated → Retired
Approval Status Draft → Pending → Approved → Rejected → Revoked
Verification Status Unverified → Verified
Certification Status None → Candidate → Certified → Expired → Revoked

The proposal's design intent: a version could be verified but not yet approved, or approved but not yet certified, independent progression per track, constrained by one governance invariant: lifecycle_state=PUBLISHED implies approval_status=APPROVED; a REJECTED or REVOKED approval status cannot coexist with a PUBLISHED lifecycle state.

This proposal supersedes an even older, pre-proposal linear 7-state lifecycle flow (Draft → Candidate → Verified → Approved → Published → Deprecated → Retired) that was documented before the four-track split. Neither the linear flow nor the four-track split is what the current two-enum model implements; both are historical or proposed, not current.

Why the Scope Collapsed

The four-track proposal is going through MLflow's RFC approval process as part of Red Hat's upstream collaboration with MLflow and Databricks on MCP registry design. Databricks' review process favors small, focused PRs, which is in tension with landing a full four-track governance model in one step. That tension is one reason the current scope collapsed to two enums plus an update mode rather than shipping all four tracks at once: the lighter two-enum model follows the same lighter-weight path taken for the upstream skills registry, with the fuller governance model planned as a follow-on rather than abandoned.

Open Items from Design Review

The specific reviewers behind these questions are named on an internal-audience source page; this hub carries the substance of the open items without attributing them to individuals by name, referring to them collectively as design reviewers.

  • Design reviewers flagged that MLflow's native model-registry states (None, Staging, Production, Archived) may need expansion to fit MCP-specific governance needs; this observation predates and partly motivated the four-track proposal above.
  • Design reviewers raised whether certification programs should be backed by an admin-managed allowlist, a question that may connect to a related open RFE.
Resolved: system-managed registry entries

An earlier open question, whether a controller like the MCP Lifecycle Operator should be able to create read-only, system-managed registry entries, is answered in the current model. The MCPUpdateMode enum (MANUAL vs SYSTEM_MANAGED) governs whether a registry entry's mutable fields are user-edited or kept in sync by another system. This is the concrete mechanism behind a "discovered" MCP server concept raised in review: system-populated entries where only description and tags remain user-editable.
