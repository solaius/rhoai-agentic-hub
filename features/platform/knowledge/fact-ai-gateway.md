Unified, multi-purpose gateway for all AI traffic — inference, egress, agentic/stateful APIs, tool calling, A2A — distinct from MCP Gateway, which handles MCP protocol traffic specifically. Core framework is IPP (Inference Payload Processor), a foundational ext_proc plugin chain within Envoy. Four traffic types: model inference, tool calling/MCP, agent-to-agent (A2A), external provider egress. Red Hat contributes to the industry-wide Open Responses Specification alongside OpenAI, NVIDIA, vLLM, Llama Stack, HuggingFace, AWS, and Databricks.

Status: ~60% detailed as of source capture; follow-up design needed on sub-request routing, Envoy re-entrance, conversation state management, and tenancy strategy. See [decision-ai-gateway-f2f-architecture.md](/features/platform/knowledge/decision-ai-gateway-f2f-architecture.md) for the binding architectural calls, and [ref-ai-gateway-payload-processing-repo.md](/features/platform/knowledge/ref-ai-gateway-payload-processing-repo.md) for the IPP implementation.

**Key CRDs**: `MaaSModelRef` (user-facing API for internal/external models), `MaaSAuthPolicy` (MaaS authentication policy), `MaaSSubscription` (subscription/entitlement management), `ExternalModel`/`ExternalProvider` (external provider connectivity — evolving toward a split with traffic splitting and `apiFormat`), `InferencePool` (backend pool for inference serving).

**Guardrails in the gateway**: MCP Gateway already ships guardrails (NeMo/TrustyAI) via an ext_proc adapter; the convergence target is extracting guardrails into a shared IPP plugin used by both inference and MCP paths (see [decision-mcp-gateway-tool-governance-scope.md](/features/mcp-gateway/knowledge/decision-mcp-gateway-tool-governance-scope.md) for the related scope boundary). Defense-in-depth has three layers: global gateway (RHCL) PII detection at the edge (future state), cluster gateway (IPP plugin) PII/prompt-injection checks before the model, and the inference layer (TrustyAI/NeMo) for output filtering. Ordering decision: guardrails run before token-limit checks in the output flow, since guardrails may mutate the response and change its token count. Streaming sync checks target a ~100-200ms latency budget; async/parallel guardrails are proposed for audio/video.

**Competitive landscape (AI gateways specifically, distinct from the MCP-catalog competitive set in [fact-competitive-landscape-mcp-catalogs.md](/features/mcp-ecosystem/knowledge/fact-competitive-landscape-mcp-catalogs.md))**: IBM API Connect AI Gateway (GA — API governance, no model-aware routing), Google GKE Inference Gateway (Preview — closest analog, GKE-only), Portkey/Bifrost (SaaS — multi-provider routing). No single competing product solves the full enterprise ask spanning cluster-level, enterprise-wide, and model-governance needs together (see the restricted F2F customer detail in [ref-ai-gateway-f2f-outcomes.md](/restricted/features/platform/knowledge/ref-ai-gateway-f2f-outcomes.md)).

**Roadmap alignment**: 3.5 (Aug 2026) — GA current Llama Stack as bridge, IPP framework foundations, capacity-based routing, agent identity Tier 1. 3.6 (late fall 2026) — capabilities building on 3.5, Responses API gateway-native implementation progress. 2027+ — full agentic platform, AI Gateway as enterprise AI control plane, Agent Mesh, proxy technology evolution. Several additional engineering-level open items (sub-request routing architecture, cache-aware token rate limiting, Files API gateway frontend requirements, streaming guardrails session, Envoy version requirements) remain undecided in the source F2F doc but are execution detail rather than PM-level open questions — see [ref-ai-gateway-f2f-outcomes.md](/restricted/features/platform/knowledge/ref-ai-gateway-f2f-outcomes.md) (restricted) for the full list if needed.
