Partner Onboarding Pipeline

A partner MCP server doesn't reach the MCP Catalog by a technical merge alone: it moves through a six-step business-plus-technical pipeline from curation to catalog integration. This is the pipeline that put 5 partner servers into the RHOAI 3.4 DP catalog by the April 10, 2026 code freeze (see What Is the MCP Catalog? for the full 10-server shipped list).

The Six Steps

1. Curation

Invitation-only selection across strategic categories (CI/CD, databases, hyperscalers, security). Partners aren't accepted on open submission; the catalog team selects who to approach.

2. BDM Outreach

Business Development Manager outreach to the selected partner, pairing the value proposition with the consent letter that starts the formal process.

3. Consent Letter

The partner signs a consent form, the formal go-ahead to build and list their MCP server in the Catalog.

4. Technical Attestation

Ecosystem Engineering validates the server against the technical requirements below (transport, deployment, container image, metadata, and security indicators).

5. Image Build

Build and push the container image to quay.io/rhoai-partner-mcp, the shared Quay org hosting shipped partner server images.

6. Catalog Integration

Integration and testing by the RH AI team before the server appears in the live Catalog.

Technical Requirements for Partner MCPs

Requirement Details
Transport Streamable HTTP only (no stdio)
Deployment On-cluster deployment, local hosting only for DP; remote-hosted MCP server support planned after the DP release
Container Image UBI-based container image in Quay
Core Metadata name, description, provider, logo, license (SPDX), licenseLink, readme
MCP-Specific Metadata tools[], transportType, deploymentMode, artifacts[], endpoints[]
Security Indicators verifiedSource, sast, secureEndpoint, readOnlyTools
Access & Revocation Metadata Each tool entry also carries an access-type (read/write) flag plus revoked (boolean) and revokedReason (string) fields

Table carried verbatim from the RHCL hub's Catalog page (govern/catalog.html).

STDIO transport: open asks, not shipped. The Transport requirement above (Streamable HTTP only) rules out stdio-based servers today. Three RFEs filed 2026-06-26 target this gap: RHAIRFE-2514 (Catalog: list STDIO MCPs with local install info), RHAIRFE-2515 (Registry: allow registration/tracking of STDIO MCPs), and RHAIRFE-2516 (spec-compliant IDE-native registry API for Cursor/Windsurf/VS Code). None are committed to a fix version yet.

Disqualification Criteria

A prospective partner server can be turned away on either technical or business grounds:

  • Technical: stdio-only transport, stale or unmaintained assets, remote-only servers (not eligible for DP's on-cluster-only requirement)
  • Business: no signed consent, no ongoing maintenance commitment, no meaningful partner investment in the integration

Source: fact-partner-mcp-catalog.md, the mcp-ecosystem knowledge partition.

Consent and Attestation, in Practice

Consent and technical attestation are two separate gates, run by two separate teams: the consent letter is a business commitment (a partner agreeing to be listed and to maintain their server), while technical attestation is Ecosystem Engineering independently verifying the server meets the requirements table above. A partner passing one gate doesn't waive the other. Post-DP, the pipeline is expected to scale with a lighter-weight maintenance and support model, distinct from Red Hat's standard operator support model, since AI assets age differently.
