Trust & Policy

This page formalizes the MCP Catalog's trust tier wording (also used in the sell-side Positioning & Trust Tiers page), then covers the governance controls layered on top of it: RBAC and approval, federation, version pinning, and health surfacing.

Trust Tiers

The Catalog surfaces four tiers of provenance:

Tier Meaning What It Certifies
Red Hat built and supported Authored and supported by Red Hat; the highest trust tier in the Catalog Standard Red Hat product support commitment
Partner built and certified Built by an ISV or technology partner; verified through the partner onboarding pipeline (consent, technical attestation, image build) Passed the Partner Onboarding Pipeline's consent and technical-attestation gates; ongoing maintenance commitment from the partner, not Red Hat support
Community curated Sourced from the open community; included based on curation criteria rather than a formal partner agreement Met the catalog team's curation bar; no formal maintenance or support commitment
Bring your own A customer-supplied MCP server deployed through the same Catalog to Lifecycle Operator to Gateway path, outside the curated tiers Nothing beyond what the customer's own packaging meets; see Bring Your Own MCP Server for the packaging bar (restricted-SCC compliance, catalog-yaml metadata)

Trust tier meanings are the same wording used across the hub (Quick Reference glossary, Positioning & Trust Tiers); trust tier display itself is sourced from the Registry starting in 3.5+.

RBAC and Approval Workflow

Roadmap, not shipped. "RBAC and approval workflow integration with Registry" is a stated Catalog key feature, but it depends on the Registry, which is still in Design phase: Dev Preview is not landing in RHOAI 3.5 stable, and Tech Preview is targeted for RHOAI 3.6 EA1. The Registry's current data model has exactly two governance enums (MCPServerStatus: active, deprecated, retired; MCPPublishState: draft, published, deprecated, retired) and no approval_status, verification_status, or certification_status field. An earlier four-track governance model (Lifecycle, Approval, Verification, Certification) from early design drafts is a deferred proposal for a post-MVP follow-on, not a shipped capability.

Until Registry-backed approval lands, trust tier assignment and access control are enforced by the same manual pipeline steps and Kubernetes RBAC (ServiceAccounts, ClusterRoleBindings) documented in Bring Your Own MCP Server and the Partner Onboarding Pipeline above, not by an in-product approval workflow.

Federation Posture

The Catalog's stated long-term scope includes multi-registry federation (Docker MCP Catalog, GitHub MCP Registry, partner catalogs): planned, not shipped. Current scope is MCP server discovery, browsing, and selection against the Catalog's own curated set.

Open question: the Catalog has no stated federation posture toward the protocol-official MCP registry (registry.modelcontextprotocol.io): whether Red Hat aggregates from it, implements its subregistry spec, or does neither. This is a decision for the TP window (RHOAI 3.6 EA1); it shapes TP scope, the metadata schema, and whether Red Hat publishes its own servers to the official registry namespace (none found there today). See Upstream & Ecosystem and question-official-registry-federation.md.

Version Pinning and Health Surfacing

  • Version pinning with deprecation notices: planned, not shipped
  • Health and status surfacing (uptime, latency, availability), with unhealthy servers deprioritized in search: planned, not shipped

Neither control exists in the RHOAI 3.4 DP catalog today. Both depend on the same Registry-as-system-of-record work as RBAC and approval above; there is no interim manual substitute documented for either.
