Upstream & Ecosystem

The MCP Catalog draws on two separate upstreams, and sits inside a wider MCP registry ecosystem that is still finding its footing: a big but immature official registry, a foundation-level governance change, and a spec that is about to move under everyone's feet.

Two Upstreams, Two Jobs

The Catalog's code and its content come from different places, and it's easy to conflate them:

  • Code upstream: kubeflow/hub, confirmed by owner ruling on 2026-07-10
  • Content upstream: opendatahub-io/model-metadata-collection, the merged pull request carrying Red Hat, Partner, and Community source metadata

Kubeflow Hub: Confirmed, and the Timing That Got Us There

Kubeflow Hub (formerly kubeflow/model-registry; its README states Red Hat drives the project's development) shipped an MCP Catalog in February 2026:

  • PR #2213, "feat(mcp): introduce mcp catalog openapi spec," merged 2026-02-16: McpServer and McpTool entities, /api/mcp_catalog/v1alpha1/* endpoints. The v1alpha1 version is itself a signal of early maturity, not a stable v1 API.
  • PR #2269, source-label filtering, merged 2026-02-26, followed by catalog landing page and tools-section UI work in March 2026
  • The pattern mirrors Kubeflow Hub's Model Catalog (YAML and HuggingFace sources, federated discovery), and the timing (a February build ahead of the May RHOAI 3.4 DP announcement) plus Red Hat's stewardship of the project made the relationship clear enough that the owner formally confirmed it on 2026-07-10: kubeflow/hub is the RHOAI MCP Catalog's upstream implementation.

A separate, unrelated repo intake once pointed to kubeflow/mcp-server (KEP-936) as "the upstream." That project wraps Kubeflow Trainer operations as MCP tools and has no releases; it is a catalog-entry candidate, not the Catalog's upstream. The correction is documented and closed.

The Official MCP Registry: Scale Without Maturity

The protocol-official registry (registry.modelcontextprotocol.io) is large and growing, but still explicitly a preview product:

  • Scale: at least 36,000 total server-version records and at least 10,000 latest-version servers, confirmed by live API pagination on 2026-07-09 (both are floors; the cursor was not exhausted, and no stats endpoint exists). Earlier snapshots (roughly 9,600 in May, roughly 2,000 in April) are stale and should not be cited going forward.
  • Still preview: the site banner reads "currently in preview. Breaking changes or data resets may occur before general availability." The API contract has been frozen at v0.1 since 2025-10-24, and no GA date appears in the 2026 roadmap.
  • Moderation is deliberately minimal: only illegal, malware, spam, or non-functioning servers are removed. Quality, vulnerabilities, and duplicates are explicitly not policed; the registry's own moderation policy describes this as pushing moderation to subregistries.

Governance: the Agentic AI Foundation

MCP itself changed hands at the governance level on 2025-12-09: Anthropic donated the protocol to the Agentic AI Foundation (AAIF), a Linux Foundation directed fund co-founded by Anthropic, Block, and OpenAI, with AWS, Google, Microsoft, Cloudflare, and Bloomberg as platinum backers. Day-to-day maintainer continuity was unchanged; a community Registry Working Group (with maintainers from PulseMCP, Stacklok, TeamSpark, and Ravenmail) still runs the registry itself. Notably, the 2026 roadmap post does not commit to a registry GA date or to any server-signing standard: both remain "on the horizon."

No protocol-level signing standard exists. Trust chains such as Docker's Cosign plus SBOM, or ToolHive's Sigstore plus attestations, live entirely at the tooling layer. The Catalog's enterprise-tier trust story has no upstream default to inherit and has to state its own chain.

Subregistry Federation: the Expected Shape

Upstream's own layering model is: the official registry holds canonical metadata ("metaregistries host metadata, not code"), subregistries add curation and trust signals on top, and package registries (npm, PyPI, Docker Hub) hold the actual code. Downstream catalogs are expected to either poll the official registry as an aggregator, or implement its subregistry OpenAPI spec, adding namespaced _meta extensions for exactly the kind of metadata Red Hat already adds (scan results, certification, tiers). Host applications are explicitly told not to consume the official registry directly. GitHub's publish-once flow shows the pattern working in production, and ToolHive realigned onto the official server.json schema in April 2026.

Open question: the RHOAI Catalog has no stated federation posture toward the official registry (aggregator, subregistry-spec implementer, or neither). This is a decision for the TP window (RHOAI 3.6 EA1); it shapes TP scope, the metadata schema, and whether Red Hat publishes its own servers to the official registry namespace (none found there today). See question-official-registry-federation.md.

Spec Version Churn in the TP to GA Window

The largest MCP specification revision since launch finalizes on 2026-07-28 (a release candidate since 2026-05-21):

  • Streamable HTTP becomes stateless: the initialize handshake and Mcp-Session-Id are removed, a new mandatory server/discover RPC is added, along with new Mcp-Method and Mcp-Name headers, and SSE resumability is removed
  • Tool schemas open up to full JSON Schema 2020-12, with ttlMs and cacheScope caching fields added
  • Auth tightens: RFC 9207 iss validation is required, and OAuth Dynamic Client Registration is deprecated in favor of Client ID Metadata Documents (a 12-month deprecation window applies)
  • No server.json or registry schema changes ride along with this spec release: the registry and spec release trains are decoupled

The Catalog's own release train is DP 3.4 shipped, DP continues through 3.5, TP targeted for RHOAI 3.6 EA1, and GA targeted at RHOAI 3.6 (code freeze Oct 23, 2026). The spec finalizes roughly three months ahead of that code freeze, which means Catalog GA has to certify servers spanning both the pre- and post-2026-07-28 spec generations.

Open question: no compatibility-matrix policy is stated yet for which spec versions catalog entries declare, what the Gateway accepts, or what the Red Hat and Partner tiers require at GA. See question-spec-version-policy-ga.md.

Adjacent Coopetition

ToolHive (Stacklok) is co-maintained by Red Hat, stated on both stacklok.com and developers.redhat.com (GitHub governance files lag the announcement). ToolHive's registry server federates Git, Kubernetes, upstream registries, and internal APIs into named catalogs, and its v1.2.0 release (around April 2026) dropped its native format in favor of the official server.json schema, extended via _meta. ToolHive is not the Catalog itself, and the full competitive read, including the coopetition reclassification, is on Competitive Landscape.
