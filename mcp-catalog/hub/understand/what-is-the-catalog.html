What Is the MCP Catalog?

The MCP Catalog is the storefront for MCP server discovery in Red Hat OpenShift AI (RHOAI): a curated catalog of MCP servers you can discover, deploy, and manage directly on Red Hat OpenShift.

Purpose

Unlike discovery-only registries, the Catalog provides a governed path from discovery to consumption, not just a list. It provides:

  • Discovery and browsing of MCP servers
  • Selection interface for deployment
  • Publisher verification and moderation
  • Trust tier display (certified, community, unverified)

The full trust tier breakdown (Red Hat, Partner, Community, Bring Your Own) lives on Positioning & Trust Tiers.

The Discover, Deploy, Connect, Consume Chain

The Catalog is the first stop in a four-stage chain: discover in the catalog UI, deploy via the MCP Lifecycle Operator, connect through the MCP Gateway, consume in Gen AI Studio. AI Hub treats MCP servers as first-class citizens alongside models.

Key Features

  • Trust tier display (certified, community, unverified), sourced from the Registry starting in 3.5+
  • RBAC and approval workflow integration with the Registry, planned: see the roadmap caveat on Positioning & Trust Tiers, since the Registry's current data model has no approval status field yet
  • Multi-registry federation support (Docker MCP Catalog, GitHub MCP Registry, partner catalogs), planned; Red Hat's own federation posture toward the official MCP registry is an open question
  • Catalog Admin UI (Settings page, CRUD on Catalog source ConfigMaps): in progress, targeted for the RHOAI 3.5 release (RHAISTRAT-1859)
  • Health and status surfacing (uptime, latency, availability), with unhealthy servers deprioritized in search: planned
  • Version pinning with deprecation notices: planned
Current limitation: the Catalog does not create prerequisites automatically. ServiceAccount, ConfigMap, and ClusterRoleBinding objects must be created manually before deploying from the catalog in 3.4. Registry integration in 3.5+ is expected to enable automatic prerequisite creation via the Lifecycle Operator.
Relationship: the Catalog builds on the Registry as the intended authoritative source (the Registry itself is still in Design phase, not shipped; see Upstream & Ecosystem), complements the Gateway's runtime enforcement, and is meant to operate in a federated, interoperable ecosystem rather than a closed list.
