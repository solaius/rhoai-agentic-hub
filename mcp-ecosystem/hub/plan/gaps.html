Gaps & Open Questions

Two parts on one page. Known Gaps are the cross-cutting limitations in current releases - documentation, integration, automation, and governance. Open Questions are the design decisions, integration points, and strategic choices still under discussion across the ecosystem.

Known Gaps

Documentation & User Experience

Gap Product Documentation Circular References

Issue: RHOAI product documentation references MCP features (Catalog, Gen AI Studio) without explaining the full prerequisite chain (Gateway API, Service Mesh, RHCL, MCPLO).

Impact: Users attempting to follow product docs hit missing dependencies and cannot complete setup without external knowledge.

Owner: RHOAI Documentation Team

Status: Known issue, documentation improvements planned for 3.5

Workaround: Use community notes (Fernando Lozano's minimal setup guide, Jaideep Rao's full setup guide) for end-to-end flow.

Source: Fernando Lozano's MCP Notes (Google Doc), Knowledge Registry #25

Gap Workbench MCP Integration Missing

Issue: No documented or supported way for Python code running in workbenches to call MCP servers. Langchain-style libraries for in-notebook MCP interaction do not exist.

Impact: AI Engineers using RHOAI workbenches cannot programmatically invoke MCP tools from their code — only from Gen AI Studio UI.

Owner: RHOAI AI Engineering Team

Status: Not scoped for 3.5; potentially 3.6+ or partner-contributed library

Workaround: Use Gen AI Studio for manual tool invocation; use direct HTTP calls to MCP endpoints (not governed path).

Source: Fernando Lozano's MCP Notes, Knowledge Registry #26

Deployment & Automation

Gap Catalog Does Not Create Prerequisites

Deploying an MCP server from the catalog requires manually creating ServiceAccount, RBAC, and ConfigMap resources; prerequisite automation is targeted for RHOAI 3.6 EA1 (RHAIRFE-2434). Full detail: MCP Catalog Knowledge Hub: Gaps & Open Questions.

Gap Namespace Label Requirement

Issue: MCP server deployments require namespace with opendatahub.io/dashboard: "true" label. This label cannot be added via oc new-project — requires manifest creation.

Impact: Users cannot use standard namespace creation workflow; must apply YAML manually.

Owner: RHOAI Dashboard Integration

Status: Known limitation; no fix planned (namespace labels are design-by-contract for RHOAI dashboard discovery)

Workaround: Use oc apply -f namespace.yaml with labels pre-set instead of oc new-project.

Source: Fernando Lozano's MCP Notes, Source Verification Report

Integration & Governance

Gap ConfigMap as Bridge (Temporary)

Issue: Gen AI Studio discovers MCP servers via gen-ai-aa-mcp-servers ConfigMap in redhat-ods-applications namespace. This is a manual, single-tenant integration point with no governance.

Impact: MCP server registration is decoupled from Catalog and Lifecycle Operator. No RBAC, no approval workflow, no lifecycle state visibility in Studio.

Owner: RHOAI Registry Team (replacement), RHOAI Gen AI Studio Team (current bridge)

Status: Will be replaced by MCP Registry (DP planned; will not land in RHOAI 3.5 stable; TP targeted for RHOAI 3.6 EA1, PM direction Jul 2026)

Workaround: Manually update ConfigMap with MCP service URLs after deployment. Template: {"url": "http://...", "description": "..."}.

Source: Source Verification Report, Knowledge Registry #27

Gap Cluster-Admin Requirement (Temporary)

Both MCP Lifecycle Operator installation and MCP Gateway configuration currently require cluster-admin privileges; no namespace-scoped path exists for either, and both are expected to lift once the MCP Registry provides tenant-scoped visibility and approval workflows. MCPLO's side of this gap, including its workaround, now lives in the MCP Lifecycle Operator Knowledge Hub: Installation; the Gateway's namespace-scoped configuration gap remains tracked here.

Gap Registry Not Yet Built

The MCP Registry does not exist in RHOAI 3.4 or 3.5; the governance backbone connecting Catalog, Lifecycle Operator, Gateway, and Gen AI Studio is missing. Dev Preview will not land in RHOAI 3.5 stable; Tech Preview is targeted for RHOAI 3.6 EA1 (owner ruling, 2026-07-11). Full detail, including the workaround and the design-vs-target comparison: MCP Registry Knowledge Hub: Gaps & Open Questions.

Gap STDIO Transport Not Supported

Issue: MCP Catalog and Registry currently support Streamable HTTP transport only. STDIO-based MCP servers cannot be listed, registered, or consumed through the governed path.

Impact: Local and IDE-native MCP servers (a large share of community and developer tooling) are invisible to Catalog search and cannot be tracked in Registry.

Owner: RHOAI Catalog Team, RHOAI Registry Team

Status: Committed direction: three RFEs filed 2026-06-26: RHAIRFE-2514 (Catalog: list STDIO MCPs with local install info), RHAIRFE-2515 (Registry: allow registration/tracking of STDIO MCPs), RHAIRFE-2516 (spec-compliant IDE-native registry API for Cursor/Windsurf/VS Code). Delivery Features RHAISTRAT-2131/2132/2133, status New, no fix version yet.

Workaround: None today; STDIO MCP servers must be installed and tracked manually outside the governed Catalog/Registry path.

Source: Jira sweep 2026-07-10 (RHAIRFE-2514/2515/2516, RHAISTRAT-2131/2132/2133). Catalog-side detail: MCP Catalog Knowledge Hub: Gaps & Open Questions. Registry-side detail (RHAISTRAT-2132): MCP Registry Knowledge Hub: Gaps & Open Questions.

Gap Registry→Gateway Auto-Config Missing

Issue: MCP Registry approval state does not automatically configure MCP Gateway. Manual registration of MCPs in Gateway required even when Registry shows "Approved" state.

Impact: Governance and runtime enforcement are disconnected. Approved MCPs do not automatically become available through gateway-secured endpoints.

Owner: MCP Gateway Team + MCP Registry Team (integration work)

Status: GA target, tracked under RHAISTRAT-1993 (code freeze Oct 23, 2026); manual registration path remains until then.

Workaround: After Registry approval, manually create an MCPServerRegistration CRD for Gateway or apply Gateway configuration YAML.

Source: MCP Gateway Roadmap, Knowledge Registry. Registry-side detail and the full target flow: MCP Registry Knowledge Hub: Gaps & Open Questions and MCP Registry Knowledge Hub: Integrations.

Platform Compatibility

Gap RHOAI 3.5 + OCP 4.19 Compatibility (Untested)

Issue: RHOAI 3.5 (Aug 2026) supports OCP 4.19, but RHCL 1.4 likely will not. MCP Gateway TP in RHOAI 3.5 may run on an unsupported OCP version.

Impact: Configuration (RHOAI 3.5 + RHCL 1.4+ + OCP 4.19) documented as untested, best-effort support only. OCP 4.19 maintenance support ends Dec 2026 (4-month overlap).

Owner: RHCL Product Team + RHOAI Product Team (alignment)

Status: Customers advised to upgrade to OCP 4.20+ for full support; OCP 4.19 gap accepted as temporary

Workaround: Deploy on OCP 4.20+ for validated configuration. If stuck on 4.19, treat as unsupported/experimental.

Source: RHOAI-RHCL Support Agreement, Source Verification Report

Gap Gateway API Prerequisite Not Enforced

Issue: MCP Gateway requires Gateway API (GA in OCP 4.19+), but installation does not validate prerequisite or fail gracefully if Gateway API unavailable.

Impact: Silent failure mode — Gateway Helm chart deploys successfully, but routing does not work. Users must diagnose missing Gateway API CRDs manually.

Owner: MCP Gateway Operator Team (future), RHCL Team (current Helm chart)

Status: Pre-flight checks planned for MCP Gateway Operator (GA target)

Workaround: Verify Gateway API CRDs exist before installing MCP Gateway: oc get crd gateways.gateway.networking.k8s.io.

Source: MCP Gateway Component Doc, Source Verification Report

Resolved Questions

Questions that have been answered or closed. For reference only.

Registry Metadata: What's Mandatory?

Question: What exact MCP metadata should be mandatory for registry records?

Answer: Mandatory fields: name, version, description, hosting model, lifecycle state. Optional (with graceful degradation for consumers): readme, tools list, parameters, runtime prerequisites.

Status: Resolved (MCP Registry MVP Requirements doc, revision dated 2026-05-07)

Lifecycle States for the Registry MVP

Question: Which lifecycle states are required for the Registry MVP? (Source doc originally scoped this as "3.5 Dev Preview"; Registry DP is now not landing in RHOAI 3.5 stable, with TP retargeted to RHOAI 3.6 EA1, PM direction Jul 2026.)

Answer: Four lifecycle states: active, deprecated, disabled, retired. This replaces the earlier "publish state" model (draft/published/deprecated), which the doc's own changelog calls a terminology correction.

Status: Resolved (MCP Registry MVP Requirements doc, revision dated 2026-05-07; doc scoped to 3.5 DP, release vehicle now retargeted to RHOAI 3.6 EA1 TP per PM direction Jul 2026)

Registry State → Catalog Surfacing

Question: How should registry state influence catalog surfacing?

Answer: Resolved via Story 5: Registry provides lifecycle-status feedback to Catalog; Catalog can query multiple lifecycle states (active + deprecated) in a single call to support sync workflows, surfacing active entries and detecting/flagging/removing deprecated ones.

Status: Resolved (MCP Registry MVP Requirements doc, Story 5)

Lifecycle History: MVP or Deferred?

Question: Whether lifecycle history should be added in MVP or deferred.

Answer: In MVP scope, Story 10 ("Preserve version history and manage lifecycle across versions"). Each version has an independent lifecycle state; older versions remain queryable.

Status: Resolved (MCP Registry MVP Requirements doc, Story 10)

MCPLO "Discovered" Registry Entries

Question: How should MCP Lifecycle Operator interact with registry entries: read-only "discovered/managed" entries vs. full governance? (Chris Hambridge, Matt Prahl)

Answer: Resolved via the update_mode field (MANUAL / SYSTEM_MANAGED) on the MCPServer entity, which controls whether a controller like MCPLO can system-manage the record. Discovered entries populate is_deployed/runtime_metadata via an MLflow client method, manual entry, the mcp client's list_tools() output, or a gateway-deployment-sync controller calling the REST API, with basic metadata (description, tags) but not the full field set.

Status: Resolved (MCP Registry Data Model Proposal + MVP Requirements doc comment threads)
