Part 1 - Customer Stories

Large Enterprise OEM Customer — "Dumb MCP Server" Validation

Customer Need: The customer wanted their MCP servers to be minimal — "I just want it to get the access token in its header and not have to worry about anything else." They use Okta as IdP, needed group-based authorization, and required token exchange to backend services (Jira, GitHub).

"I just want it to get the access token in its header and not have to worry about anything else."

— Customer security lead, OEM customer, May 2026

What We Delivered

10 Questions Answered: Comprehensive technical deep-dive covering Okta group mapping via Keycloak protocol mappers, token exchange for backend services, in-process token caching (10K entries, 30-second pre-expiry refresh), external Keycloak support (no co-location required), OBO configuration patterns, and full audit trail via Keycloak events + JWT act claim chain.

Key Validation: AuthBridge designed to transparently offload auth — MCP server receives validated token, no auth code required. Envoy sidecar (injected by Kagenti) handles JWT validation. The customer's "dumb MCP server" requirement confirmed as fully supported. Kagenti is being removed from the roadmap as of 2026-07-10, with OpenShell expected to expand to cover this capability; this delivery claim is under revalidation. The historical attribution (Kagenti delivered and injected this) stands.

Strategic Hardware Partner — Gap Analysis & Positioning

Customer Need: A strategic hardware partner's joint-solution POC — success criteria for MCP Gateway & Tool Routing. 10 requirements evaluated against current capabilities to determine POC feasibility and positioning strategy.

Assessment Results

4 Fully Met: Unified interface contract (MCP JSON-RPC), policy-driven routing (Envoy ext_proc + Gateway API), tool access allowlisting (Authorino identity-aware filtering), credential brokering (AuthBridge OBO + Vault).

4 Partially Met: Multi-transport connectivity (HTTP/SSE only, WebSocket/stdio not supported), tool-level QoS (rate limits yes, timeouts/concurrency not yet configurable per-tool), tool KPIs & observability (metrics/traces/audit yes, cost tracking gap), tool versioning & lifecycle (registry tracks lifecycle, gateway doesn't enforce version-aware routing).

2 Gaps: Central metadata registry (exists in design, not yet integrated with gateway — planned for GA Q4 2026), health checks & circuit breakers (not implemented — planned for GA).

POC Positioning Recommendations
  • Lead with strengths: Auth and policy routing are differentiated capabilities solving hard enterprise problems — centerpiece of POC demo.
  • Leverage Envoy for gaps: Health checks and timeouts available at Envoy layer even if not yet surfaced through MCP Gateway CRDs.
  • Demo registry and gateway separately: Show registry lifecycle, show gateway routing/auth, present integration path as GA milestone.
  • Frame observability as extensible: OTEL and metrics foundation is solid; cost tracking is a metric dimension to add, not a new system to build.

An Internal User Story

Context: A Red Hat Principal Software Engineer attempted to deploy MCP in RHOAI 3.4 using only product documentation. Their experience revealed every gap between "what the docs say" and "what actually works."

"RHOAI 3.4 displays 'MCP' in different parts of its web UI, but none of these features work without installing and configuring several components that are currently undocumented or scattered across multiple sources."

— Red Hat Principal Software Engineer, June 2026

What They Found

Documentation Gaps: Product docs reference MCP features but don't explain prerequisites. Circular references between docs, blogs, and GitHub repos. No end-to-end runbook for the minimal path (playground without gateway).

Missing Prerequisites: Catalog doesn't create ServiceAccount/ConfigMap/RBAC automatically (privilege escalation concern). Namespace requires special label (opendatahub.io/dashboard: "true") that can't be applied via oc new-project or console — must use manifest. gen-ai-aa-mcp-servers ConfigMap is the only integration point between deployed MCP servers and Gen AI Studio (undocumented, requires cluster-admin to patch).

Workbench Gap: "I need some Python code (Langchain?) for remote MCP calls." Playground's exported code doesn't work for MCP calls from workbenches. No documented pattern for calling MCP servers from Python in workbenches.

What We Fixed
  • Documented the undocumented: End-to-end minimal path now captured in knowledge docs (7 steps from operator install to playground usage).
  • Identified gaps, now retargeted: ConfigMap bridge will be replaced by registry integration once Registry TP lands (retargeted from the original 3.5 plan to RHOAI 3.6 EA1). Catalog prerequisite creation (RHAIRFE-2434) is likewise now targeted for 3.6 EA1, not 3.5.
  • Validated the happy path: Another engineer's comprehensive guide (full Gateway + Keycloak + Vault setup) now cross-referenced with this minimal path — users can choose their level of complexity.

Top 3 Customer Asks

1. Identity & Token Exchange

Customer Need: "How do I handle identity when my agent calls backend services? How do tokens flow through the stack?"

Status: Shipped MCP Gateway + AuthBridge + Keycloak OIDC + Vault integration addresses this. Token exchange via Keycloak, OBO support via act claim chains, wristband signing for per-tool authorization, Vault secrets injection for backends requiring stored credentials (GitHub PATs, API keys). AuthBridge's current implementation is Kagenti-delivered; Kagenti is being removed from the roadmap as of 2026-07-10, with OpenShell expected to expand to cover this capability. Claim under revalidation.

Evidence: A large enterprise OEM customer validated "dumb MCP server" pattern (May 2026) — MCP server receives validated token in Authorization header, no auth code required. 10 technical Q&As answered covering Okta federation, token caching, OBO configuration, audit trail.

2. Observability & Monitoring

Customer Need: "How do I monitor MCP server usage? Which tools are being invoked? By which agents? What's the latency? What's failing?"

Status: Shipped MCP Gateway provides OpenTelemetry traces, per-tool invocation metrics, structured audit logging, and agent-to-tool correlation. Metrics cover latency, retries, errors, and token usage.

Gap: Planned Cost tracking is not explicitly modeled (strategic hardware partner gap analysis, June 2026). Retry metrics are not differentiated from primary invocations. Cache-aware token rate limiting (cache hit/miss feedback into limiting decisions) is an open question. Planned for GA (Q4 2026).

3. Cost Optimization & Rate Limiting

Customer Need: "How do I control costs when agents fan out to external APIs? How do I prevent runaway usage?"

Status: Partial Rate limiting supported per-tool via Limitador (Red Hat Connectivity Link component). Timeouts and concurrency limits not yet configurable per-tool through MCP Gateway CRDs (strategic hardware partner gap analysis, June 2026).

Roadmap: Full tool-level QoS (timeouts, concurrency, cost budgets) planned for future releases. Envoy provides infrastructure-level timeout configuration today — can be leveraged for near-term POCs even if not yet exposed through MCP Gateway CRDs.

Agent-to-Agent (A2A): Not Yet

A2A Status: Future Work

Agent-to-Agent (A2A) communication is one of the four traffic types AI Gateway must serve (inference, tool calling/MCP, A2A, external provider egress). A2A covers agent discovery (AgentCard), invocation, and orchestration.

Current State: Discussed at AI Gateway F2F (April 2026) but not yet substantively addressed. A2A is planned for future releases but not part of RHOAI 3.5 or MCP ecosystem initial deliverables.

Related Context: The healthcare system's enterprise AI control plane mapping includes "Agent registry and lifecycle management" — overlaps with A2A discovery requirements. The bank's Team B use case (multi-cluster/region/cloud) may require A2A for cross-cluster agent orchestration.

Key Takeaways

What Customers Care About Most

  • Identity is table stakes: Every enterprise customer asks about identity and token flow before anything else. MCP Gateway's auth story (Keycloak + AuthBridge + Vault) is a differentiator.
  • Observability for trust: Customers won't deploy AI agents at scale without visibility into what's happening. OpenTelemetry traces, per-tool metrics, and audit logging are critical.
  • Cost control for safety: Runaway agent usage is a real fear. Rate limiting is a start; cost tracking and budgets are the next ask.
  • Governance before runtime: The telecommunications-provider customer asked about registry early access because they need approval workflows before deploying 20+ MCP servers. Discovery alone isn't enough.
  • Developer experience wins deals: The OEM customer's "dumb MCP server" requirement proves that complexity kills adoption. Gateway-layer auth means developers write business logic, not auth plumbing.
