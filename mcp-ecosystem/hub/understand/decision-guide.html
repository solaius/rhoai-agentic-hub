Which Component Do I Need?

Start from what you are trying to do, and this table points you at the component that does it and the hub that documents it. The four components compose - most real deployments use more than one - but each need below has a primary owner.

Decision Table

I need to... Component What it gives you Hub
Deploy an MCP server MCP Lifecycle Operator (MCPLO) The MCPServer CRD as the deployment primitive: a Kubernetes-native operator that creates Deployment, Service, and NetworkPolicy from a declarative spec. Ships under the standard OpenShift entitlement. MCPLO Hub →
Govern tool access at runtime MCP Gateway An API gateway for AI tool calls: virtual MCP servers that aggregate tools from multiple backends, identity-aware tool filtering, per-tool authentication, metrics, and audit logging. Built on Red Hat Connectivity Link. Gateway Hub →
Discover approved servers MCP Catalog A discovery layer to browse partner, community, and Red Hat MCP servers with metadata - tool lists, security indicators, license (SPDX), provider logo, and README - and a Deploy hand-off to the Lifecycle Operator. Catalog Hub →
Prove provenance and audit MCP Registry The authoritative governance layer and system of record: tracks lifecycle state, approval, certification, and ownership across a server's life from discovery through consumption. In design; TP targeted for RHOAI 3.6 EA1. Registry Hub →
They compose into one governed path

Discover in the Catalog, register into the Registry for governance, deploy with the Lifecycle Operator, and enforce at runtime through the Gateway - with Gen AI Studio as the consumption surface. If you only need a playground demo, the Lifecycle Operator plus the Catalog gets you there without the Gateway. For the full picture of how these connect, see Umbrella Architecture; to set one up, see Integration Scenarios.
