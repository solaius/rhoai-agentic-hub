Auth Setup Guide

Step-by-step guide for configuring authentication and authorization for MCP Gateway using Keycloak and Authorino. This guide covers token exchange, on-behalf-of (OBO) delegation, and IdP federation.

Prerequisites

  • MCP Gateway deployed (see Deployment Guide)
  • Red Hat Connectivity Link (RHCL) installed with Authorino and Limitador
  • Keycloak instance (can be in-cluster or external — no co-location requirement)
  • Basic understanding of OAuth 2.0 and OpenID Connect
External Keycloak is Supported

No co-location requirement — Keycloak can be hosted anywhere (different cluster, different cloud, on-premises) as long as it is network-accessible from the OpenShift cluster.

Confirmed by the Kagenti engineering team (May 2026): "On the MCP Gateway side, Keycloak can be hosted anywhere. On the Kagenti side, tested with in-cluster Keycloak, but it should work with any external Keycloak as long as it is exposed."

No shared SPIFFE trust domain is needed — SPIFFE is used as a client ID, not a trust anchor.

Step 1: Keycloak Setup

1.1 Create a Realm

Create a dedicated realm for your MCP deployment (e.g., mcp-realm):

  1. Log in to Keycloak Admin Console
  2. Hover over the realm dropdown (top left) and click Create Realm
  3. Enter mcp-realm as the realm name
  4. Click Create

1.2 Create Clients

You need two clients: one for the MCP Gateway and one for the AuthProxy sidecar.

Gateway Client:

# In Keycloak Admin Console:
# Clients → Create client

Client ID: mcp-gateway
Client type: OpenID Connect
Client authentication: ON
Authentication flow: Standard flow, Direct access grants

# After creation, go to Credentials tab:
# Copy the Client Secret — you'll need this for the gateway configuration

AuthProxy Client (for token exchange):

# Clients → Create client

Client ID: mcp-authproxy
Client type: OpenID Connect
Client authentication: ON
Authentication flow: Standard flow, Direct access grants

# After creation, go to Credentials tab:
# Copy the Client Secret

# Enable token exchange in Permissions tab:
# Permissions → Token Exchange → Enable
# Add policy: Allow token exchange from any authenticated client
Explanation: Why Two Clients?

mcp-gateway: Authenticates end-user agents. Agents present tokens from this client when calling /mcp.

mcp-authproxy: Performs token exchange to get backend-scoped tokens. When the gateway needs to call a backend MCP server on behalf of a user, the AuthProxy exchanges the user's mcp-gateway token for a new token scoped to the backend.

This separation enables the delegation chain: user → agent → gateway → backend, with each hop having appropriately scoped credentials.

1.3 Configure Token Exchange Policies

Allow the mcp-authproxy client to exchange tokens from mcp-gateway:

# In Keycloak Admin Console:
# Clients → mcp-authproxy → Permissions → Token Exchange

# Add a permission:
# Resource: mcp-gateway
# Policy: Allow exchange from mcp-authproxy
# Effect: Permit

This tells Keycloak: "The mcp-authproxy client is allowed to exchange tokens issued to the mcp-gateway client for new tokens scoped to backend services."

Step 2: Authorino AuthConfig

Create an Authorino AuthConfig to enforce authentication at the MCP Gateway endpoint:

apiVersion: authorino.kuadrant.io/v1beta2
kind: AuthConfig
metadata:
  name: mcp-gateway-auth
  namespace: mcp-system
spec:
  # Apply to all requests to the MCP Gateway HTTPRoute
  hosts:
  - "gateway.example.com"

  # Authentication — validate JWT from Keycloak
  authentication:
    "keycloak-jwt":
      jwt:
        issuerUrl: https://keycloak.example.com/realms/mcp-realm
        ttl: 300  # Cache validated tokens for 5 minutes

  # Authorization — check user has required roles
  authorization:
    "user-role-check":
      opa:
        rego: |
          default allow = false

          # Allow if user has "mcp-user" role
          allow {
            input.auth.identity.realm_access.roles[_] == "mcp-user"
          }

  # Inject user identity into headers for downstream processing
  response:
    success:
      headers:
        "x-auth-user":
          plain:
            value: "{auth.identity.preferred_username}"
        "x-auth-email":
          plain:
            value: "{auth.identity.email}"
Explanation: What This Does

Authentication: Validates that the incoming request has a valid JWT issued by Keycloak for the mcp-realm realm. Caches validation results for 5 minutes to reduce Keycloak load.

Authorization: Uses Open Policy Agent (OPA) to check that the authenticated user has the mcp-user role. Requests from users without this role are rejected with 403 Forbidden.

Response: Injects x-auth-user and x-auth-email headers into the request before forwarding to the MCP Gateway. Backend MCP servers can read these headers to know which user authorized the action.

Step 3: Token Exchange Flow

Configure the AuthProxy sidecar to perform token exchange when calling backend MCP servers:

3.1 Deploy AuthProxy Sidecar

Inject the Kagenti AuthProxy sidecar into your backend MCP server pods:

apiVersion: v1
kind: Pod
metadata:
  name: jira-mcp-server
  namespace: mcp-backends
  annotations:
    # Inject AuthProxy sidecar
    kagenti.io/auth-inject: "true"
spec:
  containers:
  - name: jira-mcp
    image: example.com/jira-mcp-server:latest
    ports:
    - containerPort: 8080
  # AuthProxy sidecar is automatically injected by Kagenti webhook

3.2 Configure Token Exchange

Create a Kubernetes Secret with the AuthProxy configuration:

apiVersion: v1
kind: Secret
metadata:
  name: authproxy-config
  namespace: mcp-backends
type: Opaque
stringData:
  config.yaml: |
    # Keycloak endpoint for token exchange
    tokenEndpoint: https://keycloak.example.com/realms/mcp-realm/protocol/openid-connect/token

    # Client credentials for token exchange
    clientID: mcp-authproxy
    clientSecret: 

    # Audience for exchanged tokens (backend MCP server)
    audience: jira-mcp-backend

    # Cache exchanged tokens for 5 minutes
    cacheTTL: 300s

Reference this secret in the pod annotation:

metadata:
  annotations:
    kagenti.io/auth-inject: "true"
    kagenti.io/auth-config: authproxy-config
Explanation: Token Exchange in Action

When an agent calls a tool that routes to the Jira backend:

  1. Agent → Gateway: Agent sends a request with a JWT from the mcp-gateway client.
  2. Gateway → Authorino: Authorino validates the JWT and injects user identity headers.
  3. Gateway → Router → Backend: Request is routed to the Jira MCP server pod.
  4. AuthProxy Intercepts: The AuthProxy sidecar intercepts the request before it reaches the Jira container.
  5. Token Exchange: AuthProxy exchanges the incoming JWT for a new token scoped to audience: jira-mcp-backend.
  6. Forward with New Token: AuthProxy forwards the request to the Jira container with the new token in the Authorization header.
  7. Jira MCP Server: Receives a validated, backend-scoped token. No auth logic needed in the application code.

Step 4: On-Behalf-Of (OBO) Configuration

On-Behalf-Of (OBO) allows an agent to act on behalf of a user with reduced permissions. This is configured via token exchange with a dedicated client scoped for a specific domain.

4.1 Create an OBO Client

Create a Keycloak client for the agent with limited scopes:

# In Keycloak Admin Console:
# Clients → Create client

Client ID: agent-obo-jira
Client type: OpenID Connect
Client authentication: ON

# Scopes tab:
# Remove all default scopes
# Add only: jira.read, jira.write

# Service account roles tab:
# Assign minimal roles needed for agent operations

4.2 Configure OBO Token Exchange

Update the AuthProxy config to exchange for the OBO client when calling Jira tools:

tokenEndpoint: https://keycloak.example.com/realms/mcp-realm/protocol/openid-connect/token
clientID: mcp-authproxy
clientSecret: 

# Request a token for the OBO client instead of the full user token
requestedTokenType: urn:ietf:params:oauth:token-type:access_token
audience: agent-obo-jira

# Include the original user's token as subject_token
# This creates an "act" claim chain showing delegation
subjectTokenType: urn:ietf:params:oauth:token-type:access_token
Explanation: OBO vs Direct Token Exchange

Direct Exchange: User token → backend token with same permissions. Backend sees user identity and full user permissions.

OBO Exchange: User token → agent token (limited scopes) + "act" claim showing user delegation. Backend sees agent identity acting on behalf of user, with reduced permissions.

Use OBO when you want to enforce "least privilege" for agents — the agent can only perform actions explicitly allowed in the OBO client's scopes, even if the user has broader permissions.

4.3 Audit Trail with OBO

OBO creates a verifiable audit trail via the act claim in JWTs:

{
  "sub": "agent-obo-jira",         # Agent acting
  "act": {
    "sub": "user@example.com"      # On behalf of this user
  },
  "aud": "jira-mcp-backend",
  "scope": "jira.read jira.write"
}

Keycloak emits events for every token exchange, including OBO flows. Combined with the act claim, you can trace the full delegation chain:

  • User authenticated: User logged in via Keycloak
  • Token issued: User received mcp-gateway token
  • Token exchanged (OBO): AuthProxy exchanged user token for agent-obo-jira token
  • Tool called: Agent called jira/create_issue using OBO token

Step 5: IdP Federation (Okta Example)

For enterprises using an external IdP like Okta, configure Keycloak to federate identity via OIDC.

5.1 Add Okta as an Identity Provider

# In Keycloak Admin Console:
# Identity Providers → Add provider → OpenID Connect v1.0

Alias: okta
Display name: Okta SSO
Authorization URL: https://your-org.okta.com/oauth2/default/v1/authorize
Token URL: https://your-org.okta.com/oauth2/default/v1/token
Client ID: 
Client Secret: 

# Sync Settings:
Sync mode: force  # or "legacy" — determines when user data is refreshed
First login flow: first broker login  # Auto-create users on first login

5.2 Map Okta Groups to Keycloak Roles

Use protocol mappers to import group claims from Okta tokens:

# Identity Providers → okta → Mappers → Create

Mapper type: Attribute Importer
Name: import-okta-groups
Claim: groups  # Okta sends groups in the "groups" claim
User attribute: groups
Sync mode override: force  # Update groups on every login

Then map Okta groups to Keycloak roles:

# Client Scopes → roles → Mappers → Create

Mapper type: User Attribute
Name: map-groups-to-roles
User attribute: groups
Token claim name: realm_access.roles
Claim JSON type: String
Add to ID token: ON
Add to access token: ON
Explanation: Sync Timing and Nested Groups

Sync timing is configurable per identity provider:

  • Sync at first login only: User attributes (including groups) are imported when the user first logs in via Okta. Subsequent logins do not update Keycloak's copy.
  • Sync every time: User attributes are refreshed on every authentication. Use this to keep Keycloak in sync with Okta's group memberships.

Nested groups are traditionally flattened when presented (e.g., Engineering/Infra/Platform becomes ["Engineering", "Infra", "Platform"]). Reconstructing a hierarchy is possible but requires custom work — treat as an edge case, not a default capability.
