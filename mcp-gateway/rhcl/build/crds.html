CRDs & Configuration

MCP Gateway introduces three Kubernetes Custom Resource Definitions (CRDs) for declarative configuration: MCPServerRegistration, MCPVirtualServer, and MCPGatewayExtension.

CRD Group, Version, and Kind Update (2026-07-10)

Upstream promoted the MCP Gateway CRD APIs to mcp.kuadrant.io/v1 (v1alpha1 remains served but is deprecated). The former MCPServer Kind has been split: backend registration is now MCPServerRegistration (shortname mcpsr), and curated virtual servers are a dedicated MCPVirtualServer Kind (shortname mcpvs). Examples below use the current mcp.kuadrant.io/v1 group and version.

MCPServerRegistration CRD

The MCPServerRegistration CRD (formerly MCPServer) registers an upstream MCP server with the gateway for tool federation. It tells the gateway: "This backend service exposes MCP tools under this prefix."

MCPServerRegistration Fields

Field Type Description
metadata.name string Name of the MCPServerRegistration resource (used in logs and references).
metadata.namespace string Namespace where the MCPServerRegistration is defined (typically same as backend service).
spec.prefix string Prefix added to all federated tools from this server to avoid naming conflicts (e.g., jira_, github_). Lowercase alphanumeric and underscores only; immutable once set. Replaces the former spec.toolPrefix field.
spec.path string URL path where the MCP server endpoint is exposed. Defaults to /mcp.
spec.targetRef.group string API group of the target resource (gateway.networking.k8s.io).
spec.targetRef.kind string Kind of the target resource (HTTPRoute).
spec.targetRef.name string Name of the HTTPRoute that routes traffic to this backend MCP server. Required.
spec.targetRef.namespace string Namespace of the HTTPRoute (optional, defaults to the same namespace).

Other optional fields: spec.credentialRef (Secret with auth credentials for the upstream server, used by the Broker for tool discovery), spec.caCertSecretRef (per-server CA certificate bundle for TLS verification), spec.category, spec.tags, and spec.hint (discovery metadata for agents), spec.state (Enabled/Disabled), and spec.userSpecificList (per-user tool lists).

MCPServerRegistration Example

apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: jira-mcp
  namespace: mcp-backends
spec:
  # All tools from this server are prefixed with "jira_"
  # Example: "jira_create_issue", "jira_search_issues"
  prefix: jira_

  # Reference to the HTTPRoute that routes to this backend
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
    namespace: mcp-backends

What this does: When the MCP Gateway Controller sees this MCPServerRegistration, it:

  1. Reads the jira-mcp-route HTTPRoute to find the backend service
  2. Adds this backend to the Broker's configuration with the jira_ prefix
  3. Updates the Router so it knows to route jira_* tool calls to this backend

MCPVirtualServer CRD

The MCPVirtualServer CRD (new alongside the v1 promotion) defines a virtual server that exposes a curated set of tools. It enables tool-level access control and federation by listing exactly which tools should be accessible through the virtual endpoint.

MCPVirtualServer Fields

Field Type Description
spec.tools array List of tool names to expose through this virtual server. Required (at least one). The tools must be available from the underlying registered MCP servers.
spec.prompts array List of prompt names to expose. Optional; when omitted, all prompts are exposed.
spec.description string Human-readable description of this virtual server's purpose. Optional.

See Scenario 2 below for a worked example.

MCPGatewayExtension CRD

The MCPGatewayExtension CRD configures gateway-level behavior. This is where you specify which Gateway API Gateway resource (and listener) the MCP Gateway should attach to, plus broker-router settings such as log level, session storage, and CA trust.

MCPGatewayExtension Fields

Field Type Description
metadata.name string Name of the MCPGatewayExtension resource (typically one per gateway deployment).
metadata.namespace string Namespace where the MCPGatewayExtension is defined.
spec.targetRef.group string API group of the target Gateway (gateway.networking.k8s.io).
spec.targetRef.kind string Kind of the target resource (Gateway).
spec.targetRef.name string Name of the Gateway resource to attach to. Required.
spec.targetRef.namespace string Namespace of the Gateway (optional, defaults to the same namespace).
spec.targetRef.sectionName string Name of a listener on the target Gateway. Required. The controller reads the listener's port and hostname to configure the MCP Gateway instance. Only one MCPGatewayExtension is allowed per namespace.
spec.logLevel string Log verbosity for the broker-router: debug, info, warn, or error. Added 2026-07-03.
spec.caCertBundleRef object Reference to a Secret containing a PEM-encoded CA certificate bundle, used as the base trust pool for all upstream MCP server connections (per-server caCertSecretRef on MCPServerRegistration appends to it). The Secret must have the label mcp.kuadrant.io/secret=true. Added 2026-07-03.
spec.sessionStore object References a Secret (secretName) containing a CACHE_CONNECTION_STRING redis connection string for redis-based session storage. When not set, in-memory session storage is used.
spec.oauthProtectedResource object Configures the OAuth protected resource metadata served at /.well-known/oauth-protected-resource. Requires an authorizationServers list; optional resource, resourceName, scopesSupported, and bearerMethodsSupported.

Other optional fields: spec.backendPingIntervalSeconds (how often the Broker pings upstream MCP servers, default 60), spec.httpRouteManagement (whether the operator manages the gateway HTTPRoute, default Enabled), spec.privateHost and spec.publicHost (host overrides), spec.trustedHeadersKey (key pair for JWT-based tool filtering), and spec.urlElicitation (URL-based token elicitation, default Disabled).

MCPGatewayExtension Example

apiVersion: mcp.kuadrant.io/v1
kind: MCPGatewayExtension
metadata:
  name: mcp-gateway-config
  namespace: mcp-system
spec:
  # Reference to the Gateway API Gateway resource
  # sectionName (required) names the listener to use on that Gateway
  targetRef:
    group: gateway.networking.k8s.io
    kind: Gateway
    name: istio-gateway
    namespace: gateway-system
    sectionName: http

  # Optional: broker-router log verbosity (debug, info, warn, error)
  logLevel: debug

  # Optional: redis-based session storage
  # (in-memory session storage is used when not set)
  sessionStore:
    secretName: mcp-session-redis

What this does: Tells the MCP Gateway Controller to deploy the broker-router Deployment and Service in the mcp-system namespace and create an EnvoyFilter targeting the istio-gateway's http listener in the gateway-system namespace, routing MCP traffic to the broker.

Common Configuration Scenarios

Scenario 1: Register a Single MCP Server

Use case: You have a Jira MCP server running as a Kubernetes service and want to make it available through the gateway.

Step 1: Create an HTTPRoute for your backend service:

apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: jira-mcp-route
  namespace: mcp-backends
spec:
  parentRefs:
  - name: istio-gateway
    namespace: gateway-system
  hostnames:
  - "gateway.example.com"
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /jira-backend
    backendRefs:
    - name: jira-mcp-service
      port: 8080

Step 2: Create an MCPServerRegistration that references this HTTPRoute:

apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: jira-mcp
  namespace: mcp-backends
spec:
  prefix: jira_
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
    namespace: mcp-backends

Result: Agents calling /mcp on the gateway will see tools like jira_create_issue, jira_search_issues. Calls to these tools are routed to the jira-mcp-service.

Scenario 2: Create a Virtual MCP Server Aggregating 3 Backends

Use case: You want to present a single "internal-tools" virtual MCP server that aggregates tools from Jira, GitHub, and Confluence backends.

Step 1: Register each backend as an MCPServerRegistration (as shown in Scenario 1):

apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: jira-mcp
  namespace: mcp-backends
spec:
  prefix: jira_
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: jira-mcp-route
---
apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: github-mcp
  namespace: mcp-backends
spec:
  prefix: github_
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: github-mcp-route
---
apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: confluence-mcp
  namespace: mcp-backends
spec:
  prefix: confluence_
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: confluence-mcp-route

Step 2: Create an MCPVirtualServer that curates the tools to expose from those backends:

apiVersion: mcp.kuadrant.io/v1
kind: MCPVirtualServer
metadata:
  name: internal-tools
  namespace: mcp-system
spec:
  description: "Curated internal tool set from Jira, GitHub, and Confluence"
  # Tool names to expose through this virtual server
  # (must be available from the registered backends)
  tools:
  - jira_create_issue
  - jira_search_issues
  - github_create_pr
  - confluence_create_page

Result: Agents connected through the virtual server see a single curated tool list: jira_create_issue, github_create_pr, confluence_create_page, etc. The Broker federates tools from every registered backend, and the MCPVirtualServer scopes which of those tools this endpoint exposes.

Scenario 3: Add Tool Prefix to Avoid Naming Collisions

Use case: You have two different Slack MCP servers (one for internal Slack, one for customer Slack) and need to distinguish their tools.

Step 1: Register each with a unique prefix:

apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: slack-internal
  namespace: mcp-backends
spec:
  # Prefix all tools with "slack_internal_"
  prefix: slack_internal_
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: slack-internal-route
---
apiVersion: mcp.kuadrant.io/v1
kind: MCPServerRegistration
metadata:
  name: slack-customer
  namespace: mcp-backends
spec:
  # Prefix all tools with "slack_customer_"
  prefix: slack_customer_
  targetRef:
    group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: slack-customer-route

Result: Agents see tools like slack_internal_post_message and slack_customer_post_message. The Router knows which backend to send each request to based on the prefix.

Why this matters: Without prefixes, both backends would expose a tool called post_message, and the Router wouldn't know which one to call. The prefix field makes every tool name globally unique within the gateway.

Best Practices

  • Use descriptive tool prefixes: Choose prefixes that make it obvious which backend a tool belongs to (jira_, github_, not j_, g_). Prefixes are immutable once set, so pick them deliberately.
  • One MCPServerRegistration per backend service: Don't try to register multiple tool prefixes in a single MCPServerRegistration. Create one per backend.
  • Use ReferenceGrants for cross-namespace references: If your MCPServerRegistration is in a different namespace than its HTTPRoute, you need a ReferenceGrant to allow the reference.
  • Keep MCPGatewayExtension minimal: Most deployments only need the targetRef field (with its required sectionName). The other spec fields are optional and rarely needed.
  • Version your CRDs: MCP Gateway CRDs were promoted to v1 on 2026-07-10 (schema identical to v1alpha1, which remains served but is deprecated). Use kubectl apply --server-side to safely update existing resources.
