MCP Registry

The authoritative governance layer for MCP servers in RHOAI, from the Gateway's vantage point: the system of record the Gateway will eventually poll for approved MCP servers, replacing today's manual registration.

Target: Design phase, no code shipped. Dev Preview planned but will not land in RHOAI 3.5 stable. Tech Preview targeted for RHOAI 3.6 EA1 (owner ruling, 2026-07-11), alongside MCP Catalog TP. GA targeted for RHOAI 3.6 (code freeze Oct 23, 2026).

Purpose

System of record for governed MCP servers (internal or external): ingestion and lifecycle management, security and certification context, versioning and immutability, and policy-aware visibility. The current data model uses two enums (MCPServerStatus: active/deprecated/retired; MCPPublishState: draft/published/deprecated/retired) plus an update_mode field, not the four-track (lifecycle/approval/verification/certification) model described in earlier design drafts, which remains a deferred, unshipped proposal.

Resolved (managed registry entries): Chris Hambridge's question ("should Lifecycle Operator create read-only 'managed' registry entries?") is answered. An update_mode field (MANUAL vs SYSTEM_MANAGED) governs whether a registry entry's mutable fields are user-edited or kept in sync by another system. This is the concrete mechanism behind Matt Prahl's "discovered" MCP server concept: system-populated entries where only description and tags remain user-editable.
Integration Gap: Not yet integrated with Gateway — automated registry→gateway flow is GA target, tracked under RHAISTRAT-1993 (code freeze Oct 23, 2026). Today's manual gateway registration (an MCPServerRegistration CRD) will be replaced by registry-driven configuration once a record clears approval.

Full depth: MCP Registry Knowledge Hub: data model, governance enums and invariants, the four-track proposal, MVP scope, the interactive UI prototype, integrations with Catalog/MCPLO/Gateway, roadmap, and open questions.
