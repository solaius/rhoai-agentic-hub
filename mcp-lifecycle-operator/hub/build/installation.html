Installation

The MCP Lifecycle Operator (MCPLO) installs today as a standalone operator: manual, cluster-admin gated, and outside OperatorHub and OLM. This page covers the two-tier ecosystem it fits into, the exact install commands, validated versions, and the DataScienceCluster (DSC) path RHOAI has proposed but not yet built.

Where MCPLO Sits in the Two-Tier Model

The broader MCP ecosystem on RHOAI installs in two tiers. Tier 1 is platform infrastructure (OpenShift Service Mesh, cert-manager, Red Hat Connectivity Link, RHOAI itself), installed via OperatorHub. Tier 2 is the MCP-specific layer (MCP Gateway Operator, MCPLO, RHBK, HashiCorp Vault), installed via a mix of OLM custom catalog, manual YAML, and Helm. Full cross-component install steps live in the MCP Management Knowledge Hub: Integration Scenarios (Operator Tiers & Validated Versions); this page focuses on the MCPLO-specific path.

Tier Scope MCPLO's role
Tier 1
Platform Prerequisites		 OpenShift Service Mesh, cert-manager, Red Hat Connectivity Link, RHOAI Not required for MCPLO's own install. Required before the MCP Gateway can route traffic to MCPLO-deployed servers.
Tier 2
MCP Ecosystem		 MCP Gateway Operator, MCP Lifecycle Operator, RHBK (Keycloak), HashiCorp Vault MCPLO installs here. It has no install-time dependency on the MCP Gateway: MCPLO installs first (or alongside RHOAI) to manage server deployment, while the Gateway is downstream for routing traffic to already-deployed servers.
Installation Order Still Matters

Even though MCPLO has no install-time dependency on the Gateway, Tier 1 operators must be healthy before Tier 2 components are installed. Deploying MCPServer CRs before MCPLO itself is installed, or deploying Gateway-side CRDs before their prerequisites exist, is the leading cause of operator pods crashlooping with CRD-related errors. See Supported Servers & Troubleshooting for the full symptom/cause/fix.

Installing MCPLO Today (Manual)

MCPLO is not available via OLM or Helm. Install manually from the GitHub release asset (the repo is kubernetes-sigs/mcp-lifecycle-operator; openshift/mcp-lifecycle-operator is a fork with no releases of its own):

oc apply -f https://github.com/kubernetes-sigs/mcp-lifecycle-operator/releases/download/v0.2.0/install.yaml

# Wait for operator to be ready
oc wait --for=condition=Ready pod -l control-plane=controller-manager \
    -n mcp-lifecycle-operator-system --timeout=300s
Since v0.2.0 (upstream)

The operator now reconciles a NetworkPolicy per MCPServer operand automatically, so no manual NetworkPolicy prerequisite is needed. Its metrics/webhook server also supports TLS security-profile tuning via the TLS_MIN_VERSION and TLS_CIPHER_SUITES environment variables.

Cluster-Admin Requirement (Temporary)

MCPLO installation currently requires cluster-admin privileges; no namespace-scoped installation path exists yet. This blocks AI Engineers from self-service deploying MCP infrastructure on multi-tenant clusters with restricted permissions. Workaround: Platform Engineers with cluster-admin deploy the infrastructure, and AI Engineers consume it via RBAC-scoped catalog access. The requirement is expected to lift once the MCP Registry provides tenant-scoped visibility and approval workflows.

The DSC Path (Proposed, Not Yet Available)

RHOAI has proposed productizing MCPLO as an automatically-deployed DataScienceCluster (DSC) component, installed the same way as FeastOperator and LlamaStackOperator, under RHAIRFE-1456. That proposal is Approved but not started in engineering as of the RHCL hub's 2026-07-10 verification. The underlying DSC component registration and reconciliation chain (DSCMCPLifecycleOperator type, mcplifecycleoperator component name, module handler wiring) already exists in the opendatahub-operator source today; what is missing is the automated, out-of-the-box install flow. Until that ships, the manual oc apply path above is the only way to get MCPLO running, whether or not RHOAI is installed. Full architectural detail: Architecture.

Validated Component Versions

These versions were tested and validated together on ROSA/OpenShift 4.21.6 with RHOAI 3.4.0, as of the RHOAI 3.4 validation run (2026-04-21). This matrix reflects that run, not current-release validation; re-validate before relying on it against a newer RHOAI train.

Component Version Notes
Platform ROSA, OpenShift 4.21.6, RHOAI 3.4.0 Baseline platform
OpenShift Service Mesh 3.3.2 Envoy gateway, mTLS
Red Hat Connectivity Link 1.3.2 (Authorino 1.3.0, Limitador 1.3.0) Includes MCP Gateway 0.6.0 TP
MCP Gateway v0.6.0 RHCL 1.3.3 bundle
MCP Lifecycle Operator release-0.1 Manual install only
Keycloak 26.4.11-opr.1 (RHBK) Identity management
Vault 1.21.2 (Helm 0.32.0) Optional, credential injection

The MCPLO row above (release-0.1) predates the current upstream release, v0.2.0 (shipped June 17, 2026): the validation run itself ran April 21, 2026. Treat this matrix as the last cross-component validation snapshot, not as coverage of the current MCPLO version.

Verification

Verify the operator is healthy after install:

oc get pods -n mcp-lifecycle-operator-system
oc get crd mcpservers.mcp.x-k8s.io
