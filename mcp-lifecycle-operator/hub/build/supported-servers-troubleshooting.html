Operator Behavior: Dependencies & Defaults

The following is drawn from an internal, anonymized field Q&A (the "FAQ on RHOAI Limited," an internal collaborative FAQ fielded July 2026), not a customer-facing document. It answers questions PMs and field teams asked directly about MCPLO's install-time behavior.

Q: Does the MCP Lifecycle Operator depend on the MCP Gateway?

No. MCPLO has no installation dependency on the MCP Gateway. It is installed first (or alongside RHOAI) to manage server deployment, while the MCP Gateway is downstream for routing traffic to deployed servers.

Q: Does it deploy any MCP servers by default?

No. Installing MCPLO does not automatically deploy any MCP servers.

Q: Can servers be installed after MCPLO is set up?

Yes. The operator provides configuration options (catalog UI or CRDs) for deploying and managing additional MCP servers as Day 2 operations. During installation, the full MCP catalog is displayed and users select which servers and tools to install.

Troubleshooting: Operator Pods Crashlooping

Operator Pods Crashlooping

Symptom

MCP Lifecycle Operator or MCP Gateway Operator pods crashloop with CRD-related errors.

Cause

Missing CRD dependency. For example, deploying MCPServer CRs before MCPLO is installed, or deploying MCPGatewayExtension before Gateway API CRDs are present.

Fix

Verify installation order. Ensure all Tier 1 operators are healthy before installing Tier 2:

# Check Gateway API CRDs
oc get crd gateways.gateway.networking.k8s.io

# Check MCPLO installation
oc get pods -n mcp-lifecycle-operator-system

# Check Kuadrant
oc get kuadrant -n gateway-system

# If CRDs are missing, install in the correct order:
# 1. Service Mesh
# 2. cert-manager
# 3. RHCL (Kuadrant)
# 4. RHOAI
# 5. MCPLO
# 6. MCP Gateway Operator
