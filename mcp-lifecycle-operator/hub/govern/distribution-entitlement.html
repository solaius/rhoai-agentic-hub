Distribution & Entitlement

MCP Lifecycle Operator (MCPLO) ships under the standard OpenShift entitlement, not an RHOAI-only one. This page corrects an older, mistaken framing that circulated in early hub content, and covers how MCPLO is distributed, updated, and where entitlement questions remain open.

Corrected Entitlement Story

MCPLO ships with OCP entitlement

MCPLO ships as a standalone operator under the standard OpenShift entitlement, not an RHOAI-restricted one. This was an explicit architecture decision (mrunalp, Slack #forum-mcp-lifecycle-operator, Feb 2026): base OCP customers get MCPLO's deployment capability without RHOAI, and can upgrade to RHOAI later without losing MCPLO functionality. RHOAI adds governance and enforcement on top: MCP Registry, Authorino (auth), and Limitador (rate limiting).

Correcting an Earlier Claim

Two other pages in this hub family (the RHCL hub's Lifecycle Operator page and the Management hub's Entitlement page) state that MCPLO requires an RHOAI-restricted subscription tier to install or use. That claim is stale. It has not been corrected on those pages as of this writing; treat the framing on this page, and on this hub's own Quick Reference, as current.

Update Delivery

Today's actual mechanism and the FAQ's described target model differ, because productization (RHAIRFE-1456) has not started in engineering yet.

  • Today (actual): MCPLO is a standalone, manually-installed operator. "Updating" means re-applying a newer install.yaml release from the upstream GitHub repo. There is no OLM update channel.
  • Target model (per the RHOAI Limited FAQ, not yet implemented): once productized as a DSC component, MCPLO would ship bundled within RHOAI operator releases. Connected clusters would receive updates automatically via RHOAI operator updates; disconnected clusters would update via the official RHOAI offline/disconnected installation bundle. The RHOAI operator engineering team would receive new MCPLO versions from the OCP team and bundle them into RHOAI operator releases.
  • Upstream alternative: the upstream kubernetes-sigs project is public and usable independently by customers who want to bring their own MCP servers without a downstream RHOAI distribution.

Disconnected Catalog Delivery (Open Question)

Open: no oc-mirror equivalent defined yet

Raised in the RHOAI Limited FAQ: "How will the MCP server catalog be delivered to disconnected customers? Is there an equivalent to oc-mirror that mirrors the MCP catalog in disconnected mode?" No answer has been provided yet; this affects both MCPLO (which deploys from the catalog) and the MCP Catalog surface itself.

The requirements research for this hub recommends, as a P0 item, packaging the MCP server catalog as an OCI artifact (following the OLM File-Based Catalog pattern) with oc-mirror support via ImageSetConfiguration additional images, plus an offline catalog UI reading from a local or mirrored source. None of this exists today.

OCP 5.0: RHOAI Limited Install Flow (Future)

Per this hub's roadmap profile, OCP 5.0 is planned to include MCP Servers, the MCP Gateway, and MCPLO via a RHOAI Limited entitlement, with an install flow that presents RHOAI with capability toggles (Inference, Gateway, Guardrails, MCPLO on/off). This is a forward-looking roadmap statement, not a shipped mechanism; treat it as directional.

OLS Transition: A Related Entitlement Question

OpenShift Lightspeed (OLS) currently uses its own internal OCP MCP server. When MCPLO is productized, OLS plans to remove that internal server and ask users to install the OCP MCP server via MCPLO instead. This raises entitlement questions directly relevant to this page, still open as of 2026-07-11:

  • What is the transition timeline?
  • Does this mean all OLS users need RHOAI installed, even if just a limited-use entitlement?
  • What happens to "continued same experience" for existing OLS users?

In the Slack discussion, Calum Murray confirmed OLS's current internal-server behavior, Gaurav Singh confirmed the transition plan exists, Steve Gordon clarified that OCP customers should not have to think about an RHOAI Limited SKU for platform-internal applied-AI use cases, and Joshua Wilson raised a concern about OLS potentially carrying a cost as a result. None of this is resolved.
