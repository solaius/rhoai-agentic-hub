Gaps & Open Questions

MCPLO's five known gaps, four of them P0 for GA readiness per this feature's own requirements research, plus three open questions carried from the management hub's cross-cutting log, numbering preserved for traceability.

Known Gaps

Gap
Operator Maturity Gap (Level 1-2, GA Needs 3-4)

Issue: MCPLO sits at Operator Capability Level 1-2 today (Basic Install, plus partial Seamless Upgrades). GA (RHAISTRAT-1995, OCPSTRAT-2879) requires Level 3-4 (Full Lifecycle, Deep Insights).

Impact: No ServiceMonitor or alerting rules, unclear errors when the Gateway is absent, no production deployment patterns or upgrade runbooks, and only TP/DP-level support commitment today.

Owner: OCP + RHOAI teams (joint GA)

Status: P0 for GA. Recommendations: a Prometheus ServiceMonitor auto-created per MCPServer, a PrometheusRule for alerting (server down more than 5 minutes, handshake failures), a GatewayAvailable condition with actionable messages, an operator-level /metrics endpoint, and an admission webhook for CRD validation.

Workaround: Treat TP-era deployments as non-production; monitor via generic pod-health checks until ServiceMonitor support lands.

Source: research/04-requirements.md §1, citing the Operator SDK Capability Levels model.

Gap
Disconnected Catalog Delivery

Issue: No oc-mirror equivalent exists for delivering the MCP server catalog to disconnected or air-gapped customers.

Impact: Affects both MCPLO, which deploys from the catalog, and the MCP Catalog surface itself: no offline browse-and-deploy experience exists today.

Owner: RHOAI Catalog Team, MCPLO Team

Status: Open question, raised in the RHOAI Limited FAQ with no answer provided yet. P0 recommendation: package the MCP server catalog as an OCI artifact (following the OLM File-Based Catalog pattern) with oc-mirror support via ImageSetConfiguration additional images, plus an offline catalog UI reading from a local or mirrored source. None of this exists today.

Workaround: None today.

Full detail: Distribution & Entitlement: Disconnected Catalog Delivery. Source: question-disconnected-catalog-delivery.md.

Gap
OLS Transition (Subscription Cliff Risk)

Issue: OpenShift Lightspeed (OLS) uses its own internal OCP MCP server today. Once MCPLO is productized, OLS plans to remove that internal server and ask users to install the OCP MCP server via MCPLO instead.

Impact: A subscription cliff (today's OCP-only users would need at least an RHOAI Limited entitlement), install complexity rising from zero steps to three, version coupling between OLS releases and the RHOAI operator, and no rollback path if the MCPLO-deployed server has issues.

Owner: OLS Team, MCPLO Team

Status: Open question, transition timeline and subscription implications unresolved. P0 recommendations: a transparent fallback (OLS detects an MCPLO-managed server and uses it if available, continuing with its internal implementation if not) and a zero-config path for OCP users (a single-click "Enable MCP servers" in console, not a multi-step RHOAI operator install).

Workaround: None today; OLS continues on its internal MCP server until a transition decision is made.

Full detail: Distribution & Entitlement: OLS Transition. Source: question-ols-mcplo-transition.md.

Gap
Helm Deprecation Path

Issue: The standalone Helm chart is slated for deprecation once MCPLO reaches TP, but no migration guide, migration tooling, deprecation notice, or resource-adoption mechanism exists yet.

Impact: Customers who deployed MCP servers via the Helm chart have no supported path to the CRD-based MCPServer model.

Owner: MCPLO Team

Status: P0 for TP: a configuration-mapping document (Helm values to MCPServer spec) and a deprecation notice in Chart.yaml. P1 for GA: a migration script that reads a Helm release and generates MCPServer YAML, resource adoption (MCPLO adopting existing Helm-created resources), and coexistence documentation.

Workaround: None today.

Source: research/04-requirements.md §5.

Gap
Cluster-Admin Requirement (Temporary)

MCPLO installation currently requires cluster-admin privileges; no namespace-scoped installation path exists yet. Full issue detail, the Platform-Engineer-deploys/AI-Engineer-consumes workaround, and the Registry-dependent path to lifting it now live on Build: Installation.
