Manage my agent fleet at scale
Related, mapped at the ecosystem level

Persona: agentops-admin   Status: candidate

When I operate many agents across my organization,
I want to centrally register, discover, and manage my agent fleet with lifecycle governance and emergency controls,
so I can discover shadow agents, enforce policies fleet-wide, and revoke agent access in seconds during incidents.

Why it is only related, not mapped: this job's features list names agent-ops and agent-registry, not mcp-lifecycle-operator. MCPLO deploys the MCP servers an agent fleet would call, but the job as written is scoped to fleet-level agent registration and governance, a layer above the individual server deployment primitive MCPLO provides.

Pillar: Agents (Agent Ops theme). Evidence status: candidate, not yet backed by direct customer evidence.

Govern agent tool and model access
Related, mapped at the ecosystem level

Persona: agentops-admin   Status: candidate

When I manage a fleet of agents,
I want to govern which tools and models each agent can access,
so I can enforce least-privilege, rate-limit token usage per team, and audit every agent action.

Why it is only related, not mapped: this job's features list names mcp-gateway and agent-ops, not mcp-lifecycle-operator. The job's three governance layers (model access, tool access, identity) map onto the MCP Gateway and AI Gateway at runtime; MCPLO sits one step upstream, deploying the servers that expose the tools those layers then govern. MCPLO's own security posture (Restricted Pod Security Standards, no auto-created ServiceAccounts, per-server RBAC scoping) hardens the deployment, but the job as written describes runtime tool-access governance, not deployment.

Pillar: Agents (Agent Ops theme + MCP Ecosystem theme). Evidence status: candidate, not yet backed by direct customer evidence.

Source: jtbd-govern-agent-tool-access.md
