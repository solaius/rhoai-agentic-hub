Gaps & Open Questions

What is missing because the MCP Registry does not exist yet, what is still undecided in its data model, and what has already been resolved. Scope: the Registry-owned and Registry-co-owned gaps and questions from the Management Knowledge Hub's cross-cutting log, cross-checked against this feature's own fresh Jira sweep (swept 2026-07-12) and knowledge entries.

Known Gaps

Gap MCP Registry Not Yet Built

Issue: The MCP Registry does not exist in RHOAI 3.4 or 3.5. The governance backbone connecting the Catalog, the Lifecycle Operator, the Gateway, and Gen AI Studio is missing.

Impact: No system of record for MCP lifecycle, approval, certification, or trust. Each component operates in isolation with manual handoffs; see Lifecycle & Governance for the full Today vs. Target comparison.

Owner: RHOAI Registry Team

Status: Dev Preview planned but will not land in RHOAI 3.5 stable (RHAIRFE-294, RHAIRFE-2421); Tech Preview targeted for RHOAI 3.6 EA1 (owner ruling, 2026-07-11). This gap's status flips once Registry TP lands.

Workaround: Manual workflows for governance: document MCP servers in spreadsheets or wikis, enforce approval via team process.

Source: Management Knowledge Hub, plan/gaps.html ("Registry Not Yet Built," data-verified 2026-07-12).

Gap Registry→Gateway Auto-Config Missing

Issue: Registry approval state does not automatically configure the MCP Gateway. Manual registration in the Gateway is required even once a Registry design exists and a record clears approval.

Impact: Governance and runtime enforcement stay disconnected. Approved MCPs do not automatically become available through gateway-secured endpoints. See the Registry → Gateway flow on Integrations for the full target design.

Owner: MCP Gateway Team + MCP Registry Team (joint; the Gateway-side half of this gap, including its manual-CRD workaround, is tracked in full on the Management Knowledge Hub: Known Gaps).

Status: GA target, tracked under RHAISTRAT-1993, aligned to RHOAI 3.6 (code freeze October 23, 2026). Manual registration remains the only path until then.

Workaround: After a manual governance decision, an admin creates an MCPServerRegistration CRD for the Gateway directly.

Source: Management Knowledge Hub, plan/gaps.html ("Registry→Gateway Auto-Config Missing," data-verified 2026-07-12); this hub's own Integrations page.

Gap STDIO Transport Not Supported

The Registry currently supports Streamable HTTP transport only; STDIO-based MCP servers cannot be registered or tracked through the governed path. This is co-owned with the Catalog team (STDIO listing is a Catalog-side concern too); the full gap, including the three filed RFEs and the Catalog-side detail, is tracked on the MCP Catalog Knowledge Hub: Gaps & Open Questions. The Registry's own half of the fix is RHAISTRAT-2132 (STDIO MCP Server Registration), status New, no fix version yet, confirmed in this feature's own fresh sweep.

Resolved Questions

Questions that have been answered or closed. Kept here for reference, with the retarget-in-progress annotation verified against this feature's fresh Jira sweep.

Registry Metadata: What's Mandatory?

Question: What exact MCP metadata should be mandatory for registry records?

Answer: Mandatory fields: name, version, description, hosting model, lifecycle state. Optional (with graceful degradation for consumers): readme, tools list, parameters, runtime prerequisites.

Status: Resolved (MCP Registry MVP Requirements doc, revision dated 2026-05-07).

Lifecycle States for the Registry MVP

Question: Which lifecycle states are required for the Registry MVP? (Source doc originally scoped this as "3.5 Dev Preview"; Registry DP is now not landing in RHOAI 3.5 stable, with TP retargeted to RHOAI 3.6 EA1, owner ruling 2026-07-11.)

Answer: Four lifecycle states: active, deprecated, disabled, retired. This replaces the earlier "publish state" model (draft/published/deprecated), which the doc's own changelog calls a terminology correction.

Status: Resolved (MCP Registry MVP Requirements doc, revision dated 2026-05-07; doc scoped to 3.5 DP, release vehicle retargeted to RHOAI 3.6 EA1 TP).

Verified against the fresh sweep: retarget is confirmed in Jira

Every other page in this hub network describes the RHAISTRAT-1762 fix-version retarget as "in progress" or "pending." The 2026-07-12 sweep shows it is no longer pending: RHAISTRAT-1762's fix version now reads "3.6 EA1 RHOAI RELEASE" (the 2026-07-10 snapshot had "3.5 GA RHOAI RELEASE"), and its updated timestamp, 2026-07-11, matches the day of the owner ruling. See Jira Tracker for the row.

Registry State → Catalog Surfacing

Question: How should registry state influence catalog surfacing?

Answer: Resolved via Story 5: the Registry provides lifecycle-status feedback to the Catalog; the Catalog can query multiple lifecycle states (active + deprecated) in a single call to support sync workflows, surfacing active entries and detecting, flagging, or removing deprecated ones. This is one of the three original state-propagation questions; see Integrations for the full Catalog ↔ Registry flow.

Status: Resolved (MCP Registry MVP Requirements doc, Story 5).

Lifecycle History: MVP or Deferred?

Question: Whether lifecycle history should be added in MVP or deferred.

Answer: In MVP scope, Story 10 ("Preserve version history and manage lifecycle across versions"). Each version has an independent lifecycle state; older versions remain queryable.

Status: Resolved (MCP Registry MVP Requirements doc, Story 10).

MCPLO "Discovered" Registry Entries

Question: How should the MCP Lifecycle Operator interact with registry entries: read-only "discovered/managed" entries vs. full governance?

Answer: Resolved via the update_mode field (MANUAL / SYSTEM_MANAGED) on the MCPServer entity, which controls whether a controller like MCPLO can system-manage the record. This is the third of the three original state-propagation questions; see Integrations for the full MCPLO → Registry flow.

Status: Resolved (MCP Registry Data Model Proposal + MVP Requirements doc comment threads).
