Quick Reference

Essential facts for the MCP Registry: the current data model, the two governance enums, the release train, key Jiras, and the upstream MLflow collaboration. The Registry is in design phase; nothing below is shipped.

Data Model (Current)

Entity Role Key Properties
MCPServer Logical governed asset, scoped to a workspace name, description, status, tags, aliases, workspace, update_mode, last_registered_version
MCPServerVersion Immutable MCP payload plus mutable governance metadata version, server_json (immutable), publish_state, tags, source, run_id, is_deployed
MCPObservedTool / MCPObservedToolSnapshot Cached tool metadata observed from a running server, tracked independently of the canonical payload Populated from tools/list or gateway sync; supports version pinning and change detection

Source: the MLflow MCP Registry Data Model Proposal and the MCP Registry MVP Requirements doc. server_json is the canonical immutable payload following the upstream MCP registry schema.

Not the four-track model

Earlier design drafts described four independent governance tracks (lifecycle, approval, verification, and certification, each with its own state enum). That is a proposal, not the current model. Today's data model has two enums plus an update mode; no approval_status, verification_status, or certification_status field exists yet. Those tracks are deferred post-MVP, not abandoned.

Governance Enums (Current)

Enum States Scope
MCPServerStatus active, deprecated, retired Asset-level (server)
MCPPublishState draft, published, deprecated, retired Version-level
MCPUpdateMode MANUAL, SYSTEM_MANAGED Controls whether a controller (for example, the MCP Lifecycle Operator) can system-manage the record

Governance metadata is first-class and kept separate from tags; workspace is the MVP visibility boundary; deployment visibility is minimal (is_deployed boolean only).

Key Jiras

  • RHAISTRAT-138: Centralized Governance. Closed/Obsolete; the original strategy feature that seeded the governance core (gated import, certification, scoped visibility, audit trail).
  • RHAISTRAT-1762: Operational Governance. In Progress; the main in-flight feature, extending governance so the Registry becomes the operational source of truth for every MCP server in use.
  • RHAISTRAT-1993: General Availability. New; the graduation feature, stable APIs and production support.
  • RHAISTRAT-2027: Catalog and Registry Integration for Tech Preview (3.6 EA1). In Progress; supersedes the closed/obsolete RHAISTRAT-1938.
  • RHAISTRAT-1677: OGX MCP Registry Integration. New; registry awareness for OGX (formerly Llama Stack) agent workloads.
  • RHAISTRAT-2132: STDIO MCP Server Registration. New; transport-aware registration for servers with no endpoint URL.
  • RHAISTRAT-2133: IDE-Native MCP Registry API. New; an upstream-spec-compliant registry API so IDE tools can discover and install governed servers.

MLflow / Databricks Collaboration

Upstream design partnership

Red Hat is collaborating upstream with MLflow and Databricks on MCP registry design. The data model proposal is going through MLflow's RFC approval process, and this collaboration informs the future generalized AI asset registry architecture.

Databricks' review process favors small, focused PRs, which is in tension with a full four-track governance model. That tension is one reason the current scope collapsed to two enums plus an update mode rather than shipping all four tracks at once.
