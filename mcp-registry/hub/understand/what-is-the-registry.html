What Is the MCP Registry?

The MCP Registry is the system of record for governed MCP servers on Red Hat OpenShift AI (RHOAI): identity, version, lifecycle state, certification context, and policy-aware visibility, kept separate from discovery, deployment, and runtime enforcement. It is currently in design phase, not a shipped product.

This component is not built

There is no shipped Dev Preview and no code today. The Registry's Dev Preview will not land in RHOAI 3.5 stable; the current push (owner ruling, 2026-07-11) is Tech Preview at RHOAI 3.6 EA1 alongside the MCP Catalog's Tech Preview, with GA targeted for RHOAI 3.6 (code freeze October 23, 2026). Everything on this page describes intent and design, not delivered capability.

Purpose

The Registry is the system of record for governed MCP servers, internal or external. It provides:

  • Ingestion and lifecycle management
  • Security and certification context
  • Versioning and immutability
  • Policy-aware visibility

Source: RHCL hub, govern/registry.html (data-verified 2026-07-11).

What It Solves

Today's MCP ecosystem on RHOAI is fragmented and manual. The Registry fills the governance gap in the asset flow: Catalog (discovery) → deploy or register (operationalize) → Registry (governance) → Gateway (enforcement). Draft and approval states ahead of that point are Catalog-side concerns, not Registry concerns; the Registry only governs assets that have already been deployed or registered, and it has no pre-registration phase. Governance begins once an asset is active, not while it is still a draft.

It fills the gap between the MCP Catalog (discovery), the MCP Lifecycle Operator (deployment), and the MCP Gateway (runtime enforcement) by governing what is approved rather than deploying or enforcing anything itself. It is explicitly not a replacement for any of those components; it is the governance layer underneath them. The Registry's own governance model keeps approval workflows separate from runtime enforcement by design: the Registry governs what's approved, the Gateway enforces at request time. Those are two distinct concerns, not one blended step.

Current Flow vs. Target Flow

The clearest way to see what the Registry changes is to compare how an MCP server moves through the platform today against the registry-governed target.

Current (fragmented, manual)

Discover
In the Catalog
Deploy
Through the Lifecycle Operator
Register
Manually, in the MCP Gateway
Update
ConfigMap, by hand
Consume
Through Gen AI Studio

Target (registry-governed)

Discover & Evaluate
MCPs found and assessed
Govern
In the Registry (system of record)
Surface
Through the Catalog, once approved
Associate
With deployment/runtime state
Consume
Through registry-governed platform paths

The target flow replaces the manual middle steps of today's flow (hand-registering in the Gateway, hand-editing a ConfigMap) with registry governance as the source of truth. No release number is attached to this target flow here on purpose: the design intent is settled, the shipping vehicle (Tech Preview at RHOAI 3.6 EA1) is covered on the hub home release timeline, not repeated on every page.

What It Is NOT

  • Not a discovery surface. That is the MCP Catalog's job; the Registry does not present a browsable storefront.
  • Not a deployment mechanism. That is the MCP Lifecycle Operator's job; the Registry tracks an is_deployed flag, it does not create workloads.
  • Not a runtime enforcement point. That is the MCP Gateway's job; the Registry governs what is approved, the Gateway enforces at request time.
  • Not the four-track governance model some earlier drafts described. See Data Model for the current two-enum model and the bounded, deferred four-track proposal.
